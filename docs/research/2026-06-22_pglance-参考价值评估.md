# pglance 参考价值评估（针对 vex-x / yiTrace）

- 日期：2026-06-22
- 评估对象：[lance-format/pglance](https://github.com/lance-format/pglance) —— 用 pgrx 写的 PostgreSQL 扩展，通过 FDW 把 Lance 数据集暴露成 PG 外部表
- 方法：下载全部真实源码（约 4000 行 verbatim）→ 8 个模块逐行精读 + vex-x 本地代码现状勘察 + 三方对抗评审（挺/怼/替代方案）

## 一句话结论

**基本不值得精读，可搬运价值极窄。** 之前"可以参考"的判断偏乐观，撤回。

- 8 个模块：7 个 `not-worth-it`，1 个（测试）`marginal`
- 技术点归类合计：PG 专属管道 **27**、Arrow/DataFusion 已免费提供的常识 **22**、反模式 **13**、真正可搬运 **仅 2**
- 三方评审：挺方 `partial-narrow`、怼方 `not-worth-it`、替代方案 `not-worth-it`

## 关键勘察纠偏：vex-x 当前还不是 DataFusion-native

本地 `yitrace-engine`（5 crate，~4900 行）是**刻意 std-only、零外部依赖的正确性骨架**：单写者协调、manifest reader-pin、三水位回收、WAL fsync、确定性 event_id 折叠、快照隔离都是真代码 + ~60 测试兜底。但 **DataFusion / Arrow / Vortex / tokio / async 一行都没真接**，全是注释愿景与 `todo!()`。

→ 拿"已接 Lance+tokio+arrow 的成品 PG 扩展"对照"std-only 内核骨架"，本就不在同一抽象层。现阶段该做的是读 **DataFusion 的 trait 契约**把未来 FFI 接入点设计对，而非抄管道代码。

## 为什么搬不动

1. **它解决的是 Arrow→PostgreSQL 阻抗失配——vex-x 不存在这个问题。** 一半代码是 Arrow 类型→PG 类型字符串翻译、63 字节 NAMEDATALEN 截断、PG-2000 纪元偏移（magic number `946684800`）、`heap_form_tuple`/`construct_md_array`/`lookup_type_cache`/SPI 拼 DDL。vex-x schema 本身就是 arrow Schema，这整层翻译是要**省掉**的负担。convert.rs 厚达 1163 行恰恰反向印证"不寄生 PG"路线对。
2. **另一半是任何 Arrow 引擎免费拥有的常识**：DataType 大 match、嵌套递归遍历、Dictionary 解包、BTreeSet 求差集对账。DataFusion 的 type coercion / CastExpr / SchemaAdapter / TableProvider 做得更彻底。
3. **被当亮点的细节经核验是坑/反模式**（见下）。

## 反面教材（引以为戒）

- `scan.rs` **完全没做投影/谓词下推**：`dataset.scan()` 全表全列读进内存，`get_foreign_plan` 把 qual 原样退回给 PG 自己过滤，`rows` 硬编码 1000.0、cost 是假常数，EXPLAIN 里的 Projection 是装饰性的。→ vex-x 必须走 `TableProvider::scan(projection, filters, limit)` + `supports_filters_pushdown` 真下推。
- `scan.rs:135` 每次 `begin_foreign_scan` 都 `Runtime::new()`，`scan.rs:313` 每取一个 batch 都 `block_on(stream.next())` —— sync-over-async 桥接，在 async-native 引擎里会 panic/死锁。
- `convert.rs` 手写两套平行 per-DataType match（Datum 路径 + JSON 路径）→ **u64 在标量列报 value_out_of_range（line 365），在 jsonb 列裸 `as i64` 静默回绕成负数（line 943-948）**。→ JSON 输出直接用 `arrow-json`，别手写第二套 match。
- 复合类型去重 key 是被截断到 60 字节的类型名 → 不同结构截断后撞名，静默复用错误类型定义。→ 名字截断要补内容 hash 后缀。
- `sync_namespace.rs:509-544` 把结构化 schema 降级成 `canonical_type_string` 字符串归一化比对（手写别名表）→ 应直接比 `arrow::datatypes::Schema`/`Field`。

## 真正该看的参考源（回报高一个数量级）

要解决"DataFusion 自研引擎里怎么做 schema 映射 / 扫描下推 / catalog"：

1. **DataFusion 官方 examples**：`custom_datasource.rs`、`advanced_parquet_index.rs`、`parquet_index.rs` —— 自定义 TableProvider + 真下推 + **索引下推**，对 vex-x"两块索引焊在内核"最贴。
2. **Vortex `vortex-datafusion` 集成层** —— Arrow-native 列式格式如何把谓词/zone-map 剪枝推进 reader，对标 vex-x 现在只有段级 zone-map（`lib.rs:208`）一处真剪枝、其余占位的现状。
3. **InfluxDB 3 (IOx)** —— 生产级 DataFusion-native 可观测引擎，catalog / 分区裁剪 / merge-on-read / 对象存储扫描全是 async 原生，领域与栈与 vex-x 几乎同构，对口一个数量级。
