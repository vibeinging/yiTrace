# yiTrace 路线:验证级骨架 → 生产就绪

> 目标:把"技术前提已验证的骨架"推进到"能扛真实负载、敢上生产"。本文按维度拆、按优先级排,每项给出"做到什么算过关"的验收标准。

## 核心原则

**生产就绪不是一两个开关,是"逐维度补 + 用真实负载逼出问题"。** 顺序:
1. **先补会数据丢失/正确的** —— 正确性(没有这个,快没用)
2. **再补能扛负载的** —— 容量/性能(真实客户量级)
3. **然后补能扛故障的** —— 可观测/运维(出了事能不能救)
4. **最后补合规(场景需要时)** —— 安全/等保(金融政企准入)

每一步都配"真实负载验证",不是单测绿就算完。

---

## 第一阶段:正确性硬化(最高优先级)

骨架的正确性有测试钉,但有几处"骨架近似、上量必塌"的点。**生产前必须补,否则会静默丢数据。**

### 1.1 持久化 GC 日志(防崩溃丢段)
- **现状**:`reclaim` 用内存版本近似"manifest 不再引用",safe_version 与 dead_set 间无联合原子性。骨架在"段 id 不复用 + compaction 不复活旧段"下安全。
- **风险**:删一半崩溃、manifest 没更新 → 段文件留盘但没人管(或反之)。
- **做法**:持久 GC 日志(写"将删 seg X" → fsync → 删 → 标记完成 → fsync)。崩溃恢复时扫描日志,未完成的删完。
- **验收**:对每条 dead 段,构造"在 unlink 前 kill -9"场景,重启后状态确定(要么删完、要么回滚成未删)。

### 1.2 safe_version 精确下限
- **现状**:对 Tentative 读者返回 0(保守、完全不回收)。
- **风险**:高并发读时任何 pending pin 都完全停回收,dead_set 堆积。
- **做法**:按 observed_epoch 设精确下限(标准 EBR 做法)。
- **验收**:有 N 个活跃读者时,回收水位 = min(observed_epoch),不是 0。

### 1.3 真 kill 崩溃测试(不只 simulate)
- **现状**:测试用 `simulate_crash_lose_memtable` 模拟。
- **风险**:真实进程被 SIGKILL/断电时的状态没验过。
- **做法**:集成测试脚本:起 server → 灌数据 → `kill -9` → 重启 → assert 数据完整 + 索引可用。
- **验收**:连续跑 100 次 kill-then-restart,零数据丢失/零索引坏。

### 1.4 段内删除与 upgrade 的正确性(merge-on-read 边界)
- **现状**:有"段已 flush + upgrade 已 commit + 崩溃"的重叠测试。
- **风险**:其它边界(并发删除 + compaction、跨段 upgrade 覆盖)没全覆盖。
- **做法**:模糊测试(fuzz)——随机生成"插入/删除/upgrade/compaction/崩溃"序列,断言每步折叠结果 == 串行执行。
- **验收**:1000 轮随机序列,无折叠结果差异。

## 第二阶段:容量与性能(用真实负载逼)

骨架的性能数是单机 release,但没压过"真实 Agent 的持续写入 + 持续查询"。

### 2.1 长时写入稳定性(7 天不重启)
- **现状**:单次 bench,没验"连续写 7 天内存/磁盘/性能曲线"。
- **风险**:内存泄漏、段数膨胀导致性能退化、WAL 文件无界增长、向量缓冲池命中率塌。
- **做法**:7 天持续写(模拟 Agent 流量),监控:内存、段数、QPS、缓冲命中率、compaction 吞吐。
- **验收**:7 天后内存稳定(不持续涨)、QPS 不退化 >20%、compaction 跟得上写入。

### 2.2 大规模检索(几亿 span + 多租户)
- **现状**:测过 5 万,几亿没测过(段数会到几万,检索 QPS 会塌)。
- **风险**:检索每查迭代所有段(O(段数))→ 几亿时几百段 → QPS 掉到几十。
- **做法**:阶段 2/3(分析文档已规划):全局 `key→段` 定位索引、BM25 段内化 + 段内 WAND、compaction 调大段压住段数。
- **验收**:1 亿 span、100 段以内,检索 QPS 仍 >100。

### 2.3 向量量化(几十亿向量省内存)
- **现状**:全精度 f32,1 亿向量 ≈ 50GB 内存。
- **做法**:先 SQ(float32→int8,4× 压缩、几乎无损),不够再上 PQ。
- **验收**:1 亿向量内存 < 15GB;召回衰减 <2%。

### 2.4 真实 embedder 接入(端到端性能)
- **现状**:bench 用合成向量,没接真实 embedder(它本身可能比建图慢)。
- **做法**:接一个真实 embedding 模型(如 BGE/Qwen-embedding),测端到端"打点→embed→建图→检索"吞吐。
- **验收**:embedder 成为瓶颈时,建图吞吐至少不拖累(append 缓冲)。

## 第三阶段:可观测与运维(出事能救)

生产系统必须有"出了事能发现、能定位、能恢复"的能力。

### 3.1 /metrics 端点(Prometheus 兼容)
- **现状**:零自监控。
- **做法**:`GET /metrics` 暴露:写入速率、段数、内存占用、缓冲命中率、WAL 大小、compaction 队列、QPS、延迟分位。
- **验收**:Prometheus 能抓,Grafana 出基础看板。

### 3.2 结构化日志(tracing/log)
- **现状**:审计打 stderr,无统一日志层。
- **做法**:关键路径结构化日志(写入/compaction/reclaim/崩溃恢复),带 trace_id。
- **验收**:一次故障能用日志重建"那段时间发生了什么"。

### 3.3 备份/恢复(数据安全底线)
- **现状**:无。
- **做法**:在线快照(走 pin 协议拿一致快照)+ 增量;恢复流程文档化。
- **验收":从快照恢复一台新实例,数据一致。

### 3.4 升级/迁移
- **现状**:无版本管理。
- **做法**:版本号 + 数据格式版本 + 升级路径(老格式新引擎能读、迁移工具)。
- **验收":老版本数据被新版本读取/升级不丢。

## 第四阶段:合规与安全(场景需要时)

**止损条件(产品说明 §6 已写)**:**任何真实 PoC 一立项,TLS + RBAC + 持久审计必须先落地。**

### 4.1 TLS(传输加密)
- 现状:明文 HTTP。
- 金融政企内网也要加密。

### 4.2 RBAC + 多租户物理隔离(可选)
- 现状:逻辑隔离(共享索引 + 过滤),tenant 从鉴权头取。
- 强场景:租户独立索引、行级权限、列级脱敏。

### 4.3 持久防篡改审计 + 接 SIEM
- 现状:打 stderr。
- 等保三级要求落盘 + 防篡改 + 接 SIEM。

### 4.4 PII 脱敏 + 落盘加密
- 现状:无。
- Agent trace 常含用户隐私(prompt/PII),需脱敏 + 静态加密。

### 4.5 限流 + 慢连接超时
- 现状:body 上限有,连接级无限流。
- 防 slow-loris / 恶意客户端。

## 怎么推进(建议节奏)

| 阶段 | 大概工时 | 何时做 |
|---|---|---|
| 1 正确性硬化 | 4-8 周 | **现在就该做**(任何上量前) |
| 2 容量性能 | 8-16 周 | 有真实客户量级时按需 |
| 3 可观测运维 | 4-8 周 | 有真实部署时 |
| 4 合规安全 | 4-12 周 | 有金融政企 PoC 立项时(止损点) |

**关键**:不要试图一次全补。**先用真实客户/真实负载跑起来,让它逼出"先补哪个"。** 骨架的价值正是可以快速迭代——每个维度补完都有测试兜,不会推倒重来。
