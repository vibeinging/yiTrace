# yiTrace 产品说明

> 日期：2026-06-22
> 定位：单机、可私有化、数据不出境的 **AI Agent 可观测性数据库**，自研引擎，面向中国金融政企。
> 状态：**能编译、能跑、84 个测试全绿的验证级骨架**，不是成品。核心技术前提已用代码验证通过；真正上量需换团队自有索引 FFI + 列式存储。诚实边界见文末「当前状态」。

---

## 一句话

别人把 Agent trace 存进 ClickHouse / SaaS，把楼盖得很高但数据出境、没有原生中文；我们**自己造存储引擎**，把"中文检索 + 带过滤语义召回 + 审计级不重不漏 + 单机私有化"焊进内核，做一台**装在客户机房里、数据一个字节都不出去**的 Agent 可观测性数据库。

---

## 1. 要解决什么问题

AI Agent（多轮对话、调工具、多 agent 协作）跑起来是个黑盒：哪一步慢、哪个 agent 出错、token 花在哪、改了 prompt 之后是变好还是变差——看不见。这就是"Agent 可观测性"要解决的。

国外已有成熟产品（Langfuse、LangSmith、Arize Phoenix 等），但对中国金融政企有三个硬伤：

1. **数据出境 / 多组件**：要么是 SaaS（数据出境，合规过不了），要么自托管也是 ClickHouse+Postgres+Redis+S3 一大堆组件，气隙机房难部署。
2. **没有原生中文检索**：trace 里的内容是中文（研判结论、风控日志、用户问题），它们的检索是英文分词，中文召回差。
3. **审计能力弱**：金融场景要"账本级"——同一条事件不能被算两遍（token/成本翻倍）、崩溃后不能丢数据、历史不能被篡改。

**我们就是冲这三条硬伤做的。** 而且我们自己的 Agent 产品（AgenticData / SuperAgent）就是第一个客户——自己用、自己打磨。

---

## 2. 三条护城河（差异化）

### ① 原生中文检索 + 带过滤语义召回

- **中文检索**：自研倒排索引 + BM25 打分，中文用无词典的 CJK bigram 分词。实测查"盗刷风控"这种非连续多概念词，我们能按相关性召回排序，而朴素子串匹配**一条都召不回**。
- **带过滤语义召回**：向量近邻（ANN）+ 把过滤条件**下推进图搜索**。实测在稀疏过滤条件下，"事后过滤"召回只有 0.50，"进图过滤"召回 **1.00**——这是竞品（出厂只做事后过滤）做不到的。
- **混合检索**：关键词 + 语义两路用 RRF 融合，同时命中的排更前。
- **按业务维度过滤**："找 agent『风控研判』报错的相似 span"——按 agent / 状态 / 时间过滤，不只按 id。

> 这是别人**结构上**给不了的：他们把存储外包给 ClickHouse，挂不进自定义的中文倒排和图式向量索引作一等公民。我们自研引擎，这两块原生长在里面。

### ② 审计级存储（账本级不重不漏）

- **确定性 event_id**：每条事件的 id = 它内容的哈希（跨 Python/TS/引擎逐字节一致）。重传几次、崩溃重放几次，都是同一个 id，**只算一次**——token/成本绝不翻倍。
- **崩溃重放幂等 + 快照隔离**：崩溃恢复后逐字段一致；并发读写删互不串味。
- **重启不丢**：原始数据（段 + manifest + 写前日志）+ 检索索引全部落盘，进程崩了重开数据照样在。

### ③ 自研引擎 + 数据不出境 + 自家 dogfood

- **单机、单目录、零外部依赖**：一个目录装下全部状态，气隙机房可部署，数据不出境。
- **自有 IP**：核心算法（中文 BM25、图式向量索引）是团队自己的，不是调外部库——这既是技术标杆，也是"自主可控"的真材实料。
- **自家先用**：我们自己的 SuperAgent 就是第一个客户，trace 灌进来自己看、自己评。

## 3. 功能清单（都已实现 + 有会失败的测试兜住）

### 数据进来（摄入）

| 能力 | 说明 |
|---|---|
| **自有 SDK** | Python + TypeScript，嵌套父子 span、token 计数、会话 id、agent/工具/模型标注、输入输出文本。确定性 event_id 三方一致。 |
| **OTLP / OpenInference 入口** | 任何已用 OpenTelemetry 埋点的 Agent 应用**不改一行**就能灌进来；认 OTel GenAI（`gen_ai.*`）和 OpenInference（`llm.*`/`input.value`）两套业界约定。 |
| **HTTP 网关** | 零依赖手搓 HTTP/1.1，8 线程池，Bearer token 鉴权，请求体上限堵 OOM，可选 gzip 解压，审计留痕。 |

### 数据看（存储 + 读）

| 能力 | 说明 |
|---|---|
| **一条完整 trace** | 事件流"读时折叠"成完整 span 树（谁调用谁），瀑布视图深度优先序。 |
| **trace 列表 / 摘要** | span 数、耗时、报错数、token 汇总。 |
| **会话视图** | 多轮对话按 session 串起来。 |
| **per-agent / 模型成本** | token 按 agent、按模型归因下钻。 |
| **agent 执行图（DAG）** | 把 span 树收拢成"哪个 agent 调了哪个工具、把任务交给了哪个 agent"——dogfood SuperAgent 最想看的视图。 |

### 数据搜（检索）

中文 BM25 / 向量找相似 / 混合，都能按 agent·状态·时间·trace 过滤。HTTP 端点 `POST /v1/search` 一个口子三种检索自动选路。

### 数据评（eval 闭环）—— 把产品从"看 trace"推到"评 trace"

| 能力 | 说明 |
|---|---|
| **打分** | 规则 scorer（不依赖 LLM 先跑通；换 LLM-judge 闭环骨架不变），分数走"晚到补写"通道写回 trace——评测分本质就是 trace 事后才有的字段，复用现成机制、不另起存储。 |
| **评测看板** | 通过率 / 均分，整体 + per-agent——回归视图"哪个 agent 退步了"。 |
| **评测数据集** | 把生产里的失败样本（或用语义召回捞的相似失败 trace）收集成命名数据集；同一数据集反复跑同一 scorer，通过率掉了就是退步。这是回归基准。 |

### 数据不丢（持久化）

写前日志 fsync、段落盘（原子写 + crc 守门）、manifest 持久化、向量独立落盘；重启后检索索引从持久段 + 向量文件**自动重建**。一句话：**flush 后丢掉整个进程，重开数据和索引全都回得来。**

## 4. 技术架构（自研引擎）

```
SDK(Py/TS) ─┐
OTLP/HTTP  ─┼─► 摄入网关 ─► 写前日志(WAL,fsync) ─► 内存表 ──flush──► 不可变段(落盘)
            │                                          │                    │
            │                          确定性 event_id 去重          时间分层 compaction
            ▼                                          ▼
        检索索引(中文BM25 / 图式向量ANN / 属性边车)   读时四源折叠(内存+段+删除+晚到补写)
            │                                          │
            └──────────────► 检索 / 列表 / 树 / eval / 成本 ◄──────┘
                        全程快照隔离 + 三水位安全回收
```

- **语言**：Rust。骨架刻意**只用标准库、零外部依赖**，`cargo test` 离线即过（便于审计、便于气隙部署）。
- **存储模型**：append-only 事件 + 读时折叠（merge-on-read）、不可变段、写时复制的 manifest、快照隔离（先登记后解引用的 pin 协议 + 三条水位安全回收）、时间分层 compaction（两步重读合并，并发删除/补写不丢）。
- **检索**：自研中文 BM25（CJK bigram）+ 图式向量 ANN（NSW，带过滤进图导航）。
- **真上量要换的件**（接口已留好，注释都标了）：列式段存储 → **Vortex**；中文 BM25 / 图式向量 → **团队自有索引的 C ABI FFI**（同套算法）；查询执行 → **DataFusion**。

## 5. 和竞品的关系

| 维度 | Langfuse（ClickHouse 收购） | LangSmith（LangChain） | **yiTrace** |
|---|---|---|---|
| 部署 | 多组件自托管 / SaaS | 闭源 SaaS / 自托管 | **单机单目录、气隙可部署** |
| 数据出境 | 自托管不出境但组件多 | SaaS 出境 | **数据一个字节不出去** |
| 原生中文检索 | 无 | 无 | **有（自研）** |
| 带过滤语义召回 | 弱 | 有 | **有，且实测召回不塌** |
| eval / 数据集 | 全套（标杆） | 全套 + 自动改进 | 探路闭环已通（规则 scorer + 回归集），LLM-judge 待接 |
| 自有 IP / 自主可控 | 用 ClickHouse | 闭源 | **核心索引自研** |

**老实说**：产品功能的"楼层数"我们还远不如 Langfuse/LangSmith（它们 prompt 全套、dashboard 全家桶、自动改进闭环都有）。我们的赌注不是功能更多，是**那三条护城河别人结构上给不了**——中文 + 数据不出境 + 自有 IP。

## 6. 当前状态（诚实边界）

**这是验证级骨架，不是成品。** 把话说清楚，避免"PPT 误导"：

**已经是真的（有测试）**：
- 存储正确性骨架：确定性 event_id、四源折叠、快照隔离、崩溃重放幂等、compaction 重读合并、**重启不丢**。
- 检索差异化**已用代码验证**：中文 BM25 多概念召回完胜子串；带过滤 ANN 召回 0.50 → 1.00。
- 端到端贯通：SDK / OTLP → HTTP → 折叠 → 检索 / eval / 成本，全程实测。

**还是"验证级"或占位**：
- 中文分词是 bigram（够用、是正路），**未上 jieba 词级**；图式 ANN 是单层 NSW、无量化/SIMD——真上量都换团队自有索引 FFI。
- 段是行式自研编码，**未接 Vortex 列式**（要不要加这个外部依赖是单独决策点）。
- eval 是**规则 scorer**，LLM-judge 还没接（需要出站 HTTP client）。

**用户已明确暂缓（先做功能）**：等保三级、TLS、RBAC/多租户物理隔离、落盘加密、PII 脱敏、持久防篡改审计日志。这些是金融政企**签单硬门槛**，功能打磨到一定程度后必须回来补。

---

## 7. 该决策的点 / 下一步

**需要你拍板的**：
1. **要不要为 Vortex 列式格式加外部依赖**（破"零依赖 + 离线"线，但列式是存储承重墙的下一步）。
2. **什么时候从"做功能"切回"补合规"**（等保/TLS/PII 是签单门槛，越往后越堵）。

**不需决策、可继续推进的**：LLM-judge 接入、BM25 升 jieba、检索段内行级行指针（免全段扫描）、控制台前端、prompt 管理等产品楼层。
