# 检索的跨段 / TB 扩展性分析

> 2026-06-24 ｜ 起因：5 万 span 压测时检索 QPS 从 ~1500 掉到几百。系统性看检索随数据增长的成本，
> 给出按 ROI 的优化路线。这比榨建图更影响"几 T"目标。

## 1. 一条检索查询的成本结构

`search_text_attr` / `search_similar_attr` 拆成四段：

| 段 | 做什么 | 现状结构 |
|---|---|---|
| **① 索引查找** | BM25 倒排打分 / HNSW 图遍历 → K 个候选 `(trace,span)` | **全局单一结构**（不迭代段） |
| **② 定位+折叠** | 对 K 个候选，**找到它们在哪些段、折叠其事件** | **逐段问**（时间剪枝后迭代所有段） |
| ③ memtable 扫描 | 最近未落段的尾巴 | 小、快 |
| ④ upgrade 归并 | 晚到补写（eval 分等）叠加 | 逐段 upgrade 表 |

**关键事实**：候选只是 `(trace,span)` key，**不带段位置**；段条目只有**时间 zone-map**（min_ts/max_ts），
**没有 key 级索引**。所以 ② 只能"时间剪枝 → 逐段查 by_key"。这就是 O(段数) 的来源。

## 2. 随数据增长的三条线（根因拆解）

### 轴1 — 折叠定位 O(段数)
②对每个候选 key 迭代所有时间重叠的段。段折叠缓存免了"重解码"，但仍是 `O(S × K)` 次哈希探查 +
每段固定开销（zone-map 判断、upgrade 表迭代、缓存锁）。段数 S 随数据增长（compaction 对数兜底但仍涨）。
**实测 13 段就让 QPS 掉 3×**（其中也混了轴2 的 BM25 打分变慢）。S 到几万/百万时是杀手。

### 轴2 — BM25 无 top-k 剪枝
`Bm25TextIndex.search` 对命中词的**所有** posting 打分（没有 block-max-WAND）。语料越大、postings 越长、
打分越慢。50k 比 20k 慢有一部分是这。（已在 TODO）

### 轴3 — 索引常驻内存 O(行数)
BM25 倒排、属性边车 `filter_attrs`、会话边车、HNSW 上层图 —— 都是**内存 HashMap，尺寸 ∝ 总数据**。
**这是"几 T"的硬墙**：十亿级 span 的全局倒排放不进内存。（HNSW 底层+向量已落盘、按需读，是三者里最扛 TB 的。）

## 3. 根本张力：全局索引 vs 段内索引

| | 全局索引（现状） | 段内索引（Lucene/Tantivy 式） |
|---|---|---|
| 查询 | 一次查找，快 | 每段查 + 合并 → **O(S)** |
| 内存 | ∝ 数据 → **卡在 RAM 上限** | 段内、可换出 → **可控** |

检索系统的根本矛盾就在这。真实系统的解法是**段内索引 + 狠剪段（zone-map/bloom）+ 分层合并（compaction
压住段数）+ 热段缓存**——用"剪段"把段内索引的 O(S) 压回近似 O(命中段数)。

## 4. 解决方案菜单（按轴）

### 轴1 折叠定位 —— 让"找候选在哪段"变快
- **A. 段级 key Bloom 过滤器**（manifest 每段存一个小 bloom）：对候选 key 先测每段 bloom，"肯定没有"的段直接跳，
  不碰段折叠缓存。**便宜**（位测试）、内存小（几 bit/key）、把常数砍掉一大截；但仍是 O(S) 次位测试。
  **适合段数到几千。**
- **B. 全局 `(trace,span)→段` 定位索引**：O(1) 定位每个候选 → 总 O(K)，**真正对 S 次线性**。但索引 O(去重 span 数)，
  TB 级要**落盘**（LSM/B-树）。投入大。
- **C. 索引直接携带位置**：BM25 posting / HNSW 节点存上 `(key, 段号/全局行号)`，候选**自带位置**、根本不用"定位段"
  （经典倒排 posting = docId→位置）。最优雅；但索引耦合物理布局，compaction 搬迁要同步更新。

### 轴2 BM25 —— block-max-WAND（已 TODO）
top-k 剪枝，只算可能进 top-k 的 posting。固定 k 下查询**几乎不随语料涨**。与轴3 的段内倒排配合。

### 轴3 内存墙 —— 段内索引 + 分层
- BM25 倒排 + 属性边车做成**段内**（flush 时建、随段存、mmap/分页），查询**每段出 top-k（段内 WAND）再全局归并**。
  内存受控（冷段索引在盘），代价是 O(S) → 靠轴1 的 bloom/zone-map 剪段压回来。这是 Lucene segment 模型。
- HNSW：底层+向量已落盘、缓冲池按需读，**已经是最扛 TB 的**；上层图小、可控。
- **最难的是 BM25**（现在是全内存全局倒排）。把它段内化是 TB 的主要工程。

### 配套：Vortex「索引驱动随机取行」
一旦定位到候选的"段+行号"，Vortex 列式段**只取那一行**（随机访问，正是当初选 Vortex 的理由）。
定位索引 + 列式随机取行 = "直奔 K 个候选行"，不碰其余。

## 5. 推荐路线（按 ROI / 规模里程碑）

| 阶段 | 做什么 | 解决 | 规模 |
|---|---|---|---|
| **1（✅ 已做）** | 段级 key **Bloom**（轴1A）+ BM25 **WAND**（轴2，term 级，block-max 待续） | 5万规模实测 BM25 检索 513→827 QPS、向量 805→1066 | 万~千万 span |
| **2（中）** | 全局 `key→段` 定位索引（轴1B）或索引携带位置（轴1C）+ Vortex 随机取行 | 折叠对 S 真正次线性 | 亿级 |
| **3（大，真 TB）** | BM25 倒排/边车**段内化** + 段内 WAND + 分层归并 + compaction 调大段/压段数 | 内存墙 | 几 T |

**主线判断**：
- 短期最划算 = **段级 Bloom**（轴1A，小改、直接砍掉"逐段碰缓存"）+ **WAND**（轴2，BM25 不随语料涨）。
- "几 T" 的真答案 = **段内索引 + 剪段 + 分层**（轴3 + 轴1），是 Lucene/Tantivy 走通的路；工程量大，等真有 TB PoC 再上。
- compaction 是隐形杠杆：调大段、压住段数，直接改善所有查询的 O(S)。

## 6. 开源参考（成熟方案，与本分析吻合）

四套主流开源系统，正好把上面三条轴各自走通了，值得直接参考：

### Quickwit（Rust + Tantivy，PB 级日志/trace 搜索，跑在对象存储）—— **最对口**
- **split = 我们的段**：不可变、自包含的 Tantivy 索引（**段内倒排** + 列式 fast fields + doc store）。
  → 正是轴3"段内索引"的答案。
- **split 级元数据剪枝**：每个 split 存 **时间范围 + tag 集合**，查询先按这俩跳掉不可能命中的 split。
  → 正是轴1A"段级 zone-map + bloom/tag 剪段"。
- **hotcache**：每个 split 一个极小的"内部布局"静态缓存，先取它，再只取命中的字节范围 → 在 S3 上开 split 也便宜。
  → 我们的"段折叠缓存"是同一思想的雏形。
- 存算分离、无状态搜索节点、SIMD/向量化。**结论：段内索引 + 段级剪枝 + hotcache + 不可变段 + compaction，就是这套。**

### Tantivy（Rust 版 Lucene）—— 倒排这侧的"可买"选项
- segment 模型、**block-max-WAND**（轴2 的现成解）、列式 fast fields。
- ⚠️ build-vs-buy：直接用 Tantivy 能省掉自研倒排，但**中文分词要自己接**，且**"自有中文 BM25 当一等 IP"的产品叙事会塌**
  （同当初否 DuckDB/chDB 的逻辑）。可作底层倒排引擎参考、或托底，但一等索引仍宜自研。

### ClickHouse MergeTree —— 列式数据跳过的标准答案
- **稀疏主键索引**：每 granule（默认 8192 行）一个条目，按主键范围跳 granule。
- **data skipping 索引**：`minmax` / `set(N)` / **`bloom_filter`**，每个块存轻量摘要，命中不了就跳。
  → 我们的"段级 key bloom + minmax"就是它，只是把粒度从 granule 提到段（或也下沉到段内块）。
- `GRANULARITY` 调粒度。**坐实：bloom + minmax 跳过块/段是列式的通用解。**

### 向量侧：Milvus / Qdrant / DiskANN
- **Milvus growing vs sealed 段**：growing 段暴力搜（还没索引）、sealed 段建索引 → 对应我们 memtable(暴力) →
  落段(HNSW)。坐实"段式向量 + 封段建索引"。
- **DiskANN / Vamana**：磁盘常驻图（NVMe）、十亿级 95% 召回、亚 10ms、RAM 只要 HNSW 的一小部分
  （PQ 压缩向量在内存导航 + 全精度向量在盘重排）。→ 这是磁盘向量在 TB 级的**金标准**，比我们现在的
  "HNSW 落盘"更省内存。我们的向量量化(PQ/SQ)TODO 正是往这靠。Qdrant 的 OnDisk/mmap 模式同向。

**一句话**：我们的推荐路线（段内索引 + 段级 bloom/zone-map 剪段 + block-max-WAND + 分层 compaction，
向量走磁盘图 + PQ）**就是 Quickwit/Lucene/ClickHouse/DiskANN 各自走通的路**，不是另辟蹊径。可以放心照着做、
甚至局部托底（如 Tantivy 作倒排引擎参考），但"中文 BM25 + 图式向量当一等 IP"仍按自研保。

## 7. 与现有件的衔接
- 段折叠缓存（已做）：解决了"重解码"，没解决"逐段定位" → 轴1 接着做。
- 时间 zone-map（已做）：段级剪枝的先例 → 加 key bloom 是同一套机制扩到 key 维。
- DiskGraphIndex（已做）：向量侧已落盘扛 TB；BM25 段内化是另一侧的对称工作。
- WAND 是 TODO #2，本分析把它定位成轴2 的解、与段内倒排同阶段。
