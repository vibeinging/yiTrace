# yiTrace 竞品分析

> 日期：2026-06-22
> 用途：给产品定位、销售话术、路线优先级提供一份「对手到底强在哪、我们赢在哪、威胁在哪」的依据。
> 底料：[`docs/research/2026-06-18_agent-observability-competitors.md`]（8-agent 并行调研 + 两路红队复核）。本文是在那份调研之上做的**竞争定位分析**，不是重复罗列功能。
> 一句话结论：**论功能楼层我们远不如海外标杆；论「中文原生 + 数据不出境 + 自有索引 IP + 单机私有化」这四条，海外 SaaS 结构上抄不动——这是我们唯一、但足够硬的立足点。**

---

## 1. 范围与方法

- **赛道**：AI Agent / LLM 应用可观测性（trace + eval + 成本 + 数据集），不是通用 APM。
- **取样**：海外公开资料可查的主流玩家 11 家，按商业模式 + 技术栈分四群。
- **维度**：部署形态、数据主权、中文检索、语义召回、eval 闭环、Agent 专属能力、企业合规、自有 IP。
- **诚实声明**：竞品能力按「过半竞品标配」归纳，非逐家逐版本核对；个别维度（多模态/流式/采样）只有部分家有，签单前需再精确核。海外厂商的**中国本地化**情况以公开信息为准，可能滞后。

---

## 2. 市场地图：四群玩家

```
                 数据出境(SaaS主导)          数据可不出境(自托管)
                ┌─────────────────────┬──────────────────────────┐
  功能"楼层"高  │ LangSmith            │ Langfuse(多组件)           │
  (eval/prompt/ │ Braintrust           │ Arize Phoenix              │
   dashboard全) │ Datadog LLM Obs      │ W&B Weave(自托管版)        │
                │ New Relic            │                            │
                ├─────────────────────┼──────────────────────────┤
  轻量/入口型    │ Helicone(Proxy)      │ Traceloop / OpenLLMetry    │
                │ AgentOps             │ Pydantic Logfire           │
                └─────────────────────┴──────────────────────────┘
                                        ▲
                          yiTrace 想占的格子：
                   「数据不出境 + 单机单二进制 + 中文原生 + 自有索引」
                    —— 自托管阵营里没有一家同时满足这四条
```

**关键观察**：自托管阵营（右列）里，没有一家是「单二进制 + 中文原生 + 自有语义召回」。Langfuse 自托管但要 ClickHouse+Postgres+Redis+S3 四件套，气隙机房难伺候；Phoenix/Logfire 轻但无中文、无审计级账本。**我们要占的格子是空的**——但也意味着没人验证过这个格子有多大市场。

## 3. 直接对手深扫

### 3.1 Langfuse —— 最贴近我们的对标（开源自托管，已被 ClickHouse 收购）

| 项 | 情况 |
|---|---|
| 模式 | 开源 MIT 自托管 + SaaS；**2026-01 被 ClickHouse 收购**，后端 ClickHouse+Postgres+Redis+S3 |
| 强在哪 | 全生命周期：层级 trace/span + Sessions + Environments；eval 三件套（LLM-judge / code scorer / 人工标注队列）+ 在线/离线双模；Datasets（生产 trace→数据集、CI/CD 回归门禁）；Prompt 管理全套（版本/label/回滚/Playground/A-B）；自定义 Dashboard + 告警；OTel 原生 + 20+ 框架自动埋点 |
| 对中国的硬伤 | **多组件非单二进制**（气隙部署运维重）、**无原生中文检索**、SaaS 版数据出境 |

**判断**：这是我们最该盯的对手——它证明了「开源自托管 LLM 可观测性」有真实需求和社区。但被 ClickHouse 收购后，它的存储更深地绑死在 ClickHouse 上，**中文倒排和图式向量这种自定义一等索引它结构上挂不进去**——这正是我们的结构性缺口。我们打它的方式不是功能更全，是「单二进制 + 中文 + 不出境」三连击。

### 3.2 LangSmith —— 等保方向的直接对手（LangChain 官方，闭源）

| 项 | 情况 |
|---|---|
| 模式 | 闭源 SaaS + 自托管，agent-native |
| 强在哪 | 企业合规全套（SSO/SAML/SCIM/RBAC/ABAC/审计/加密）；Agent Studio 可视化调试断点；**LangSmith Engine（2026-05 公测）自主改进闭环**：生产失败聚类成 issue→自动起草修复 PR→拉失败 trace 进回归集；评估器深（LLM-judge/pairwise/multi-turn/judge 校准）；逐节点 state diff |
| 对中国的硬伤 | **无开源核心、无中文、数据出境** |

**判断**：LangSmith 的企业合规能力（RBAC/审计/加密）正是我方**明确暂缓**的等保方向——它在这条线上遥遥领先。它的「自主改进闭环」是赛道最前沿。我们短期不可能正面拼这些。但它闭源 + SaaS + 数据出境，**在中国金融政企气隙场景里直接出局**——这不是功能差距能弥补的，是合规红线。我们和它几乎不在同一个采购清单上竞争。

## 4. 其余三群（归纳）

### 4.1 OTel / 开源轻量系：Arize Phoenix、Helicone、Traceloop(OpenLLMetry)、Pydantic Logfire

- **共同入口标准 = OTel / OpenInference 语义约定**（LLM/RETRIEVER/EMBEDDING/RERANKER/TOOL/AGENT/CHAIN/GUARDRAIL span kind + `gen_ai.*`）。**不兼容它 = 生态绝缘**——这条我们已经接了（`otlp.rs`）。
- 自动埋点覆盖 20+ 框架；Helicone 是 Gateway/Proxy 范式（代理即埋点 + 缓存 + 限流 + Key Vault）。
- 质量护栏：toxicity/PII/幻觉检测、**默认 PII 脱敏**、RAG retrieval relevancy。
- Phoenix 的 **RAG 检索可观测 + retrieval relevancy / embedding 漂移** 跟我方 BM25/graph_index 强相关，是值得对标的一块。
- Logfire 支持 **SQL 查询可观测数据**——我方有自有引擎，这块（自有查询面）可差异化。

**判断**：这群是「入口标准制定者」，轻、开源、生态广，但都**无中文原生、无审计级账本、无单机私有化定位**。我们不与它们抢入口（反而要兼容），抢的是「数据落地后的存储与检索内核」。

### 4.2 eval / 企业系：Braintrust、W&B Weave、Datadog LLM Obs、New Relic

- **在线打分**（生产 trace 落库即异步 judge，可配 scorer + 采样率 + 过滤）；**回归门禁**（eval 当 CI 单元测试，PR 跑 + 回贴评论，Braintrust 是标杆）；Experiments 不可变可对比。
- Scorer 三件套（内置 factuality/relevance/safety/regex/embedding + code + LLM-judge）；人工评审与自动打分共享、标注回流 golden dataset。
- 漂移/异常检测（自适应阈值）、语义聚类 Patterns、AI 告警降噪、成本归因到 trace/span/model/provider、尾采样、SLO 监控。

**判断**：这群代表赛道的**收钱层 = eval / prompt / dataset / dashboard / 告警**。这正是我方几乎全空的一层（我们只有规则 scorer + 回归集骨架），**其中 eval 是命门**。这是我们最需要补、也最该投入的产品楼层。

### 4.3 Agent 专属能力（最贴 AgenticData 自家 dogfood）

逐节点 **state diff**、**agent 执行图/决策路径有向图**、**trajectory 轨迹回放 + time-travel**、tool/function call 一等 span、**死循环检测**（Datadog GA）、多轮 session 串联、相似失败聚类、agent 评测（任务成功率/步数/工具正确性/轨迹质量）。

**判断**：我方已有 **agent 执行图（DAG）** 和 **session 串联**，且自家 SuperAgent 就是第一个客户——dogfood 是别人没有的迭代速度优势。但 state diff、轨迹回放/time-travel、死循环检测这些 agent 独有的高价值视图我们还没有，是中期要补的差异化点（且与我方语义召回能形成组合拳：用相似失败聚类喂回归集）。

## 6. 我们赢在哪 / 输在哪 / 威胁在哪

### ✅ 赢在哪（结构性、对手抄不动）

1. **数据不出境 + 单机单二进制**：金融政企气隙机房的**硬门槛**，SaaS 玩家直接出局，自托管玩家组件太重。这不是功能，是合规资格。
2. **中文原生检索 + 自有索引 IP**：海外没有一家把中文倒排和图式向量做成存储一等公民——他们把存储外包给 ClickHouse，结构上挂不进去。这同时满足金融政企的「自主可控」诉求。
3. **审计级账本（确定性 event_id 不重不漏）**：金融对「token/成本不能算两遍、崩溃不丢、历史不可篡改」有刚需，海外产品多是「尽力而为」而非账本级。
4. **自家 dogfood**：SuperAgent 自用，迭代反馈环比任何外部客户都快。

### ❌ 输在哪（差距真实，短期补不齐）

1. **eval / prompt / dashboard / 告警全套**——赛道的收钱层，我们几乎全空。**eval 是命门，最该优先补。**
2. **企业合规**（RBAC/审计/加密/SSO）——LangSmith 全套，我方已明确暂缓；但这恰是金融政企**签单硬门槛**，迟早要回来补。
3. **Agent 高级视图**（轨迹回放/time-travel/state diff/死循环检测）——LangSmith 标杆，我方未做。
4. **成熟度**——对手是数年打磨的成品 + 社区 + 20+ 框架自动埋点；我方是验证级骨架。

### ⚠️ 威胁在哪

1. **Langfuse 被 ClickHouse 收购后做中国本地化 / 国产化适配**——若它出单二进制 + 中文版，我方差异化被正面冲击。概率中等，但要盯。
2. **国产玩家才是真正威胁**——已专项调研，见 [`docs/research/2026-06-22_国产竞品调研.md`]。结论：分三类，云厂商（阿里百炼/火山/腾讯，强但云端绑定，私有化非主场）、**APM 老兵（博睿/听云，私有化金融客户根基在手、正接 LLM，最贴身威胁）**、开源 Agent 平台（Dify/Coze，可观测外包给 Langfuse，是潜在渠道而非敌人）。**主战场是打博睿这类 APM 老兵**——它们的强项（私有化+合规+金融客户）正是我方软肋，我方强项（中文原生+带过滤语义召回+审计账本+Agent trace 原生）正是它们「通用 APM 加 LLM 皮」补不动的深度。
3. **OTel `gen_ai.*` 标准继续收敛**——入口标准化降低迁移成本，对所有人是双刃剑：利于我们接生态，也利于客户随时换走。

## 7. 对产品路线的结论

1. **守住四条护城河，不要在功能楼层数上跟海外标杆硬拼**——拼不过，也不是我们的赢点。话术统一到「中文 + 不出境 + 自有 IP + 单机」。
2. **优先补 eval 闭环**：这是赛道收钱层、是命门、且我方已有骨架（规则 scorer + 回归集），接 LLM-judge 投入产出比最高。
3. **合规（等保/RBAC/加密）是签单门槛，不是加分项**——明确暂缓可以，但要排进路线，越往后越堵单。
4. **专项补「国产竞品 + 国产云厂商」情报**：本分析的最大盲区。海外对手结构上进不了气隙金融机房，真正的威胁来自国产玩家。
5. **把 dogfood 优势用满**：SuperAgent 的相似失败聚类 → 喂回归数据集 → 评测，这条闭环是我方独有的迭代飞轮，应作为 Agent 专属能力的突破口。
