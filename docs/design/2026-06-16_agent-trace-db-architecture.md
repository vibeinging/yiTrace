## 0. 执行摘要（先读这一页）

**产品**：以你们**已信创的主产品（yiTrace / openGauss 内核家族）为底座**，做一个**国产、可私有化、单机、完整的 Agent 可观测平台 + 专用存储**（代号 yiTrace）。既给自家 AgenticData 当可观测底座，也卖给金融政企（客户在要）。

**已锁定的战略（决策者）**：① 首客=**金融政企**（信创/等保三级/国密/HA 是采购准入前置）；② 噱头首要价值=**信创自主可控叙事**；③ 单客户中小（单盘装下）；④ **信创等得起，赶不上批次可等明年**（runway 不是瓶颈）。

**两个直接技术问题的答案（见 §1）**：
- **"存储格式支持吗"**：✅ **支持**（基于对你们真实代码的勘察）。yiTrace/openGauss 已有 UStore/CStore 列存、4 种向量索引(HNSW/IVF/IVFPQ/DiskANN)+ 专用向量存储管理器、TOAST、RANGE+INTERVAL 时间分区、JSONB+GIN、**BM25+Jieba 中文分词**——trace 所需的数据类型与索引能力**基本都已具备**。唯一缺口是 **没有原生 LSM/append-only 不可变段**，trace 的"run=事件序列/merge-on-read"要用 **应用层 append 事件表 + 查询期折叠**（最省、可先上）或内核级自研段（重、后置）来补。可行性 **中高**。
- **"DuckDB 可以吗"**：✅ **可以，而且不必为信创放弃它**（见 §2 的关键校正）。你们的 DuckDB 向量插件是真资产；单写进程在"单机单进程"形态下不是问题。

**v3 最重要的认知校正（第三轮红队，可能帮你省 18 个月）**：**"信创自主可控" ≠ "不能用外国开源"。** 官方/行业口径：开源软件（含国外 Apache/MIT）被认定为**可自主可控**（源码开放、可自编译、不受出口管制）；**openGauss 自己就含约 25% PostgreSQL 代码，照样在信创目录**。被排斥的是"不开源的境外黑盒/SaaS/限制再分发许可证"。→ **真正的护城河是"国产主体 + 可私有化 + 可气隙 + 数据不出境 + 信创在册"，不是"我们没用开源"**（后者是双标，你自己的栈里有 PG 代码、Rust/LLVM、Qwen3、Vue/ECharts、OTel，喊"完全自主可控"会被当场打脸）。

**→ 连锁结论**：引擎选型从"信创意识形态"降级为**纯工程/资产复用选择**——openGauss-based、DuckDB+你们的插件、Lance **三者都信创可行**（只要源码内置 vendoring + 跑在国产 CPU/OS + 数据不出境）。

**放行前必做的两个零成本核实（1-2 周，在任何重自研之前）**：① 拿一份目标金融客户的**采购需求书** + 向 CESI/测评机构**书面确认**：应用软件类信创认证是否真对"关键路径用外国开源"判负、有没有硬性自研占比阈值（大概率没有）；② **PoC openGauss 自定义存储引擎/Table AM 接口是否可用**（你们已为向量索引改内核，这点能快速验证）——它决定"自研段长在 openGauss 里"是插件级还是内核 fork 级。**这两件事任一翻盘都会改写引擎决策，成本几乎为零，却能避免方向性返工。**

## 1. 你的两个问题（基于真实代码勘察）

### 1.1 "基于主产品做、存储格式支持吗" —— 支持，可行性中高
对 `openGauss-vector-main` / `yitrace-vector` 的勘察结论（证据见 `appendix` 与下表）：

| trace 需求 | yiTrace/openGauss 现状 | 评级 |
|---|---|---|
| 行为日志/事件行存 | Heap + **UStore**（原地更新+undo，适合 span 状态更新） | ⭐⭐⭐⭐⭐ |
| 列式聚合（cost/latency/token） | **CStore 列存** + 向量化执行 | ⭐⭐⭐⭐（cstore 仅追加写，见下） |
| 大 payload | **TOAST**（压缩+外置+分块读） | ⭐⭐⭐⭐ |
| 时间分区/归档 | **RANGE + INTERVAL 自动分区** | ⭐⭐⭐⭐⭐ |
| JSON 元数据过滤 | **JSONB + GIN** | ⭐⭐⭐⭐⭐ |
| 全文 + 中文 | **BM25 + Jieba 中文分词**（已有 `vexjieba_add_userdict`！） | ⭐⭐⭐⭐ |
| 向量语义召回 | **HNSW/IVF/IVFPQ/DiskANN + 专用向量存储管理器** | ⭐⭐⭐⭐⭐ |
| **trace 事件序列/merge-on-read** | ❌ **无原生 LSM/append-only/不可变段** | ⭐⭐（需补） |

**唯一真缺口 = LSM/事件折叠**。SmithDB 用 LSM+deletion/upgrade vector 处理"span 早上生下午死/乱序晚到"。openGauss 是堆+MVCC，**别硬套 in-place UPDATE**（高频晚到更新→膨胀/vacuum）。两条补法：
- **(简,先上)** 应用层 **append-only 事件表 + 查询期折叠 + 时间分区**（热区 UStore、冷区 CStore）。最省、能过认证、能 ship。
- **(重,后置)** 内核级自研不可变段（需先验证 Table AM，见 §0 核实②）。

### 1.2 "DuckDB 可以吗" —— 可以，且信创不构成障碍
- 技术：单机单进程形态下"单写进程"不是问题；你们的向量插件直接提供语义召回底座。
- 信创：DuckDB（MIT，源码开放）vendoring + 跑国产 CPU/OS + 数据不出境，按 §2 的校正**算自主可控可接受**。
- 但：DuckDB 路线**不继承 yiTrace 的信创认证/不复用 openGauss 内核家族故事**，对"国产数据库家族"叙事弱于 build-on-yiTrace（见 §3）。→ 适合做**快速原型/兜底**，不一定是金融政企主线。

## 2. 关键认知校正：信创自主可控 ≠ 不用外国开源

第三轮红队联网查实，推翻了 v3 引擎本想"为信创去掉所有外国开源"的前提：

- **开源（含国外 Apache/MIT）= 可自主可控**：源码开放、可自编译、不受出口管制。官方排斥的是"不开源的境外黑盒"。
- **铁证**：openGauss 本身约 25% 源自 PostgreSQL（PG 9.2.4），**照样进信创目录**——"用开源底座照样过认证"。
- **SCA / 自研占比**是"提升自研比例"的改进手段，**不是带固定阈值的一票否决**；公开资料未见应用软件类的硬比例线。
- **不要喊"完全自主可控"**：你的栈里有 PG 代码、Rust/LLVM、Qwen3 开源模型、Vue/ECharts、OTel——喊到最满会被对手用同一把尺子双标打脸。

**真正结构性、对手补不上的护城河（改用可证伪的精确表述）**：

| 对手 | 为什么结构上过不了中国信创 |
|---|---|
| Langfuse | 境外公司主体 + 核心 ClickHouse + S3 + SaaS 演进方向；不会为中国监管重写 |
| Phoenix(Arize) | **Elastic License 2.0（非 OSI、限再分发）** → 国产厂商无法合法二次发行做信创产品 |
| SmithDB/LangSmith | 美国闭源 SaaS、对象存储、数据出境（金融红线） |

> **标书话术**："可观测能力是可追赶的工程问题；**信创自主可控是主体资格问题**。对手的公司主体、许可证、架构哲学三样里没有一样允许它们走完中国信创——这不是时间问题，是它们不会、也无法为中国监管重写自己。而我们关键数据路径国产化、源码内置可审计、数据不出境、信创在册。"

## 3. 引擎决策 = 工程选择（三选项均信创可行）

| 选项 | 资产复用 | 信创继承 | 叙事(家族) | 工程量 | 风险 |
|---|---|---|---|---|---|
| **(A) build on yiTrace/openGauss** | **最高**（内核+向量+BM25中文+分区+TOAST+JSONB 全复用） | **最强**（继承 yiTrace 信创路径） | **最强**（国产数据库家族） | 中（LSM 缺口用 app 层折叠先补；Table AM 待验证） | Table AM 是否可用（§0②） |
| (B) DuckDB + 你们的向量插件 | 高（复用 DuckDB 插件） | 弱（不继承 yiTrace 认证） | 中 | 低（最快原型） | 单写(单进程 OK)；trace 层要 bolt-on |
| (C) Lance + DataFusion | 低 | 弱 | 弱 | 最低 | 外国主导(信创 OK 但叙事弱)；与团队栈远 |

**推荐 = (A) build on yiTrace/openGauss**，因为它**同时最大化你两个锁定决策**：金融政企→继承 yiTrace 信创认证路径；信创自主可控→openGauss 国产内核家族叙事 + 资产复用最深。**(B) DuckDB+插件留作快速原型/兜底。(C) 不选**（资产复用最低、家族叙事最弱）。

**但听第三轮红队的"减法"**：先用**最省力能过认证的引擎形态**——openGauss 热表(UStore) append 事件 + 查询期折叠 + CStore 冷区聚合 + 复用 yiTrace 向量索引 + BM25 中文 + RANGE 分区——**抢认证、抢首单**；**内核级自研不可变段(LSM)推迟到认证后 v2 再做**。认证要的是"功能/安全/兼容合格"，不是"性能业界第一"。**别让重自研拖累认证长杆。**

> 自研深度的天平：你选了"信创自主可控为首要"，按直觉应做更深自研；但 §2 校正后，**信创并不要求你去掉外国开源**，所以"为信创而重自研"的理由消失了。自研深度应回到**工程/性能/差异化**判断——而差异化钩子(语义召回)你已有 yiTrace 向量资产，不需要重写存储格式来证明自主可控。**结论：自研聚焦在 trace 专用层(树编码/事件折叠/语义召回融合/飞轮)，存储底座尽量复用 yiTrace/openGauss。**

## 4. 信创 / 等保 / 国密 / 金融 认证路线图

**核心修正**：信创真正的长杆**不是测评本身**（~1 个月），而是**名录批次窗口（一年仅约 2 次）+ 知识产权固化 + 多栈适配 + 客户立项**。**"赶不上批次等明年"——你已确认 runway 撑得住，所以这根日历长杆可接受**；但 Phase 0 必启动项不能拖（错过窗口=丢半年）。

| 认证项 | 周期(净/含排队) | 关键路径 | 必须起跑 |
|---|---|---|---|
| 软著 + 自主知识产权报告 | 软著1-3月；IP报告1-2月 | 是(前置门票) | **Phase 0** |
| 适配互认证(鲲鹏/飞腾/海光 × 麒麟/统信 × openGauss) | 1-2周/栈(有环境) | 是 | **Phase 0** |
| CESI 信创产品认证(应用软件) | 测1-2月 + **批次等待最长~6月** | 是 | **Phase 0(抢批次)** |
| 等保2.0 三级 | 1-2月，**每年复测** | 是 | Phase 1 |
| 商用密码评估(密评) | **2025新规：金融KII每年强制** | 是 | Phase 1 |
| 国密 SM2/SM3/SM4 三链路集成 | 工程2-4月 | 是 | Phase 0-1 |
| 金融 HA(RTO≤4h/RPO≤0.5h、两地三中心) | 随等保+客户验收 | 是 | Phase 1 |
| 昇腾/CANN embedding 适配 | 1-3月 | 条件触发 | Phase 1-2 |

**Phase 0 不可压缩四件事**：知识产权固化、三类CPU×两类OS×openGauss适配互认证、CESI 信创产品认证抢最近批次、与麒麟/鲲鹏/统信建合作关系。
**注**：yiTrace 作为基于 openGauss 的**新产品**须**独立走信创适配认证**，不能直接套用 openGauss 的在册资格（待核实，见 §0）。

## 5. 噱头叙事 + 杀手级 demo（修正后护城河话术）

**一句话定位**：*yiTrace —— 国产、可信创、可气隙的 Agent 专用数据库：在一台国产服务器上，用一句中文问出你的 Agent 到底干了什么。*

**8 分钟杀手 demo**（每幕同时击中"国产可信创"+"语义召回"两个钩子）：① 亮国产整机(鲲鹏/麒麟/昇腾)→**当场断网进气隙**；② 中文问"哪些理财 Agent 回答'保本'时绕过了合规检查"→带过滤语义召回(保本/刚兑/零风险近义全中)；③ 下钻 trace 树(区间编码 O(1) 子树扫，百万 span 不卡)；④ 切 SQL 面板:`semantic_match('保本话术')` 作为 SQL 算子 + 列在盘上 SM4 加密 + 全程审计(等保);⑤ 双轨摄入(自家 AgenticData 深轨富 trace + LangChain 应用 OTel 框架无关轨);⑥ 对比卡:信创/等保/国密/数据不出境/中文语义召回 逐项对手打叉——**"它们不是暂时没做信创,是架构/主体/许可证上做不了"**。

**带动自家产品**：yiTrace 与 yiTrace 同 openGauss 内核家族（认证资产复用、交叉销售）；与 AgenticData 深集成回路（运行→观测→评估→优化的**全国产可气隙自闭环**，对标 LangChain+LangSmith）。打包"国产 Agent 全栈"= AgenticData(建)+yiTrace(观测)+yiTrace(RAG)，一次招标一套认证。

## 6. 双轨现金流：别浪费认证等待期

红队 HIGH 风险：全押金融政企 + 17-29 月零回款 = 现金流单点脆弱。**你说"可以等明年"解决了 runway 焦虑，但"等"不等于"干等"**：

> **认证长杆期(必然要等)正好用来跑互联网/AI公司/出海客户**——这批客户**不要信创认证、付费快、反馈快**，是认证期的**现金流引擎 + PMF 验证 + 语义召回钩子打磨**，还反哺金融政企版成熟度。把它们从"放弃"改为"等待期的双轨"。金融政企版在后台并行推认证。**这是纯增量,不冲突。**

## 7. Phase 0：在重自研前必做的零成本核实（1-2 周）

红队 kill-shot：v3 别把"可证伪的关键假设推迟到工程做完才暴露"。动重工前先花 1-2 周做三件几乎零成本的事，任一翻盘都改写引擎决策：

1. **认证门槛核实**：拿目标金融客户**采购需求书** + 向 CESI/测评机构书面确认——应用软件信创认证是否对"外国开源在关键路径"判负、有无自研占比硬阈值。（大概率"开源可接受"→ 则 §2 成立，DuckDB/Lance 都可用，无需为信创重写存储。）
2. **Table AM PoC**：验证你们 openGauss 内核分支是否有可用的自定义存储引擎/Table AM 注册接口；若无，"自研段长在 openGauss 里"实为内核 fork（重）→ 据此重估工程量，或采用 §1.1 的"应用层事件折叠"轻方案。
3. **语义召回端到端 PoC**：带过滤 ANN（向量 + 标量/JSONB 过滤下推）在你们向量索引上的真实延迟——这是招牌钩子，跑不出来差异化塌方。同时定标：落盘字节/span、每 trace embed 条数、国产 CPU 上 Qwen3-0.6B 吞吐。

## 8. 路线图（信创长线 + dogfood + 双轨）

- **Phase 0（启动即并行，~1-2 月）**：三项零成本核实(§7) + 信创 Phase-0 四件事(§4) + 引擎/平台接口冻结。
- **Phase 1 Dogfood + 双轨现金流（~4-6 月）**：在 AgenticData 上跑通 build-on-yiTrace 引擎(app 层事件折叠) + 平台 MVP(trace 浏览器/中文/语义召回/仪表盘/RBAC) + 轨 A 深集成；**同期用互联网/中小客户跑收入与 PMF**。
- **Phase 2 金融政企版 + 认证（与认证批次对齐）**：等保三级内建(三权分立/审计/双因素) + 国密三链路 + HA 主备 + 脱敏 + 轨 B(LangSmith-compat/OTel) + 私有化打包；信创认证随批次推进。
- **Phase 3 认证后 GA + 深化（认证通过后）**：内核级自研段(若 PoC 证明值得) + 飞轮在线召回回路 + 昇腾路径 + 全栈国产 SKU。
- **节奏**：自家先用 ~5 月；互联网客户回款/PMF 贯穿等待期；金融政企首单由认证批次决定（你已确认可等明年）。
