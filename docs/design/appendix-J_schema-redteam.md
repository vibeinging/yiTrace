# 附录 J：第四轮 schema 红队（risky —— 抓到 ON CONFLICT 致命语法 + 折叠/冷区/晚到正确性洞）

**最致命一击:** Every fold pipeline in all three designs is written as PostgreSQL `WITH ... INSERT ... ON CONFLICT (span_id) DO UPDATE ... WHERE` — and openGauss has no `ON CONFLICT` at all. Its upsert is MySQL-style `INSERT ... ON DUPLICATE KEY UPDATE`, which takes NO conflict-target list, forbids a leading WITH/WITH RECURSIVE on the INSERT, references the proposed row only via EXCLUDED (target referenced by qualified name, not an `sc` alias), and gates the optional WHERE behind B/Dolphin compatibility mode. So the single most important and most-repeated statement in the entire deliverable — the incremental fold upsert — does not parse on the stated platform, in any of the three schemas. Paired with the post-freeze late-feedback hole (the frozen trace's span_current row is DELETEd and CStore cannot be updated, so a legitimately-late feedback event has nowhere to land), the design is not yet落地: the write path is unbuildable and the correctness path can silently drop late data."

**缺口:**
- No design measures or bounds the live-fold cost with a concrete number. The prompt asks 'is query-time fold fast enough at mid scale' and all three assert 'yes for active traces' with no EXPLAIN, row-count model, or benchmark. At <1亿 span/day on a single disk the hot partition is ~10^8 rows/day; an unparameterized GROUP-BY view over it is not 'small'. The active-trace working-set size is never quantified.
- Concurrency/visibility between the 5-15s micro-batch fold and the live view is undefined. A span can be both partially folded into span_current and still live in span_current_live; the UNION ALL in span_current_all can return the SAME span twice (end arrived, batch not yet run) or zero times. No dedup/precedence rule between the two branches.
- event_id global monotonicity is assumed but not guaranteed: a SEQUENCE with CACHE 1000 across multiple writer sessions interleaves cached blocks (not globally monotonic), and snowflake is offered as an alternative without reconciliation. The fold_version=max(event_id) guard and last-non-null ordering can regress if event_id is non-monotonic.
- tenant isolation is not enforced physically: span_current PK is span_id only, all hot queries filter (tenant_id, trace_id) via secondary indexes; no partition-by-tenant or tenant-prefixed PK, so freeze/GC and the HybridANN scalar set assume an isolation the layout does not provide.
- CStore PARTIAL CLUSTER KEY scan locality requires pre-sorted inserts; the freeze INSERT...SELECT is not guaranteed ordered by the PCK, so cold-scan locality may not materialize.
- Vector/BM25 index lifecycle vs continuous ingestion is unspecified: HybridANN cannot build on an empty table and ANN indexes generally dislike being created before data, but WHEN to (re)build on a growing span_vectors with rolling per-tenant onboarding is undefined.
- Every vector/BM25 DDL is effectively unverified against the actual 信创 openGauss-yiTrace build: bm25_score()/@~@/vex_jieba TEMPLATE/vexjieba_add_userdict and the index keywords (hnsw vs yitrace_graph, vector vs floatvector, *_cosine_ops class names) are taken from yiTrace-Lite, a separate PG-plugin product; the prompt explicitly warns these differ on the openGauss build. None confirmed on target.

**风险:**
- **[CRITICAL] FATAL SYNTAX: All three designs' incremental fold uses PostgreSQL-style `INSERT ... ON CONFLICT (span_id) DO UPDATE SET ... WHERE EXCLUDED.fold_version > sc.fold_version`. openGauss does NOT support PG `ON CONFLICT`; its upsert is MySQL-style `INSERT ... ON DUPLICATE KEY UPDATE col = expr [WHERE condition]`. Concrete breakages: (a) NO conflict-target column list — you cannot write `ON CONFLICT (span_id)`; conflict is auto-detected against ALL unique constraints/PK in creation order; (b) the `DO UPDATE` keyword and `sc.`/`EXCLUDED` alias-on-target pattern differ — only `EXCLUDED.col` is valid to reference the proposed row, the existing row is referenced by qualified target name; (c) Design 1 wraps the upsert SELECT in a `WITH ... INSERT` and Design 2 uses `WITH folded AS (...) INSERT ... ON CONFLICT` — openGauss `INSERT ... ON DUPLICATE KEY UPDATE` does NOT support WITH/WITH RECURSIVE CTE prefixes; (d) no RETURNING with the upsert. As written, every fold pipeline statement fails to parse.**
  - 缓解: Rewrite all upserts as `INSERT INTO span_current(...) SELECT ... FROM (subquery, NOT a leading CTE) ON DUPLICATE KEY UPDATE col = COALESCE(EXCLUDED.col, span_current.col), status = GREATEST(EXCLUDED.status, span_current.status) WHERE EXCLUDED.fold_version > span_current.fold_version`. Move the folding CTE into a derived table/view. Confirm WHERE-on-update needs B/Dolphin compatibility mode (dbcompatibility='B'); in default (A/PG) mode push the fold_version monotonicity guard into the SELECT or use MERGE INTO instead.
- **[HIGH] openGauss `ON DUPLICATE KEY UPDATE` forbids updating PK/unique columns and, with multiple unique constraints, 'only the FIRST detected conflict row is updated' (order = constraint creation order). Designs do not pin this down; Design 3 adds a span_id PK plus many indexes and contemplates secondary uniqueness, making fold correctness depend on undocumented constraint order — risk of silent partial upserts.**
  - 缓解: Keep exactly ONE unique constraint (span_id PK) on span_current so conflict detection is deterministic; never SET span_id in the update list; make all other indexes non-unique. Document this invariant.
- **[HIGH] Design 1 §3c and Design 2 §5.1 compute pre/post via SQL; Design 2's `post = max(rn) WHERE dotted_order LIKE prefix||'.%'` self-join is O(n^2) (its own TODO admits this) and runs at freeze time per trace — can stall the daily batch on multi-thousand-span traces. The nested-set post via LIKE-prefix is also fragile if dotted_order segments are not fixed-width or collation reorders '.' vs digits.**
  - 缓解: Compute pre/post in application memory (single O(n) DFS) and COPY back, as Design 2 itself recommends; avoid the LIKE-prefix self-join for large traces. Zero-pad fixed-width segments and use C collation so lexical order == numeric order.
- **[HIGH] Designs 1 & 2 expose live fold as an UNPARAMETERIZED view (`span_current_live`/`span_live`) that GROUP BYs over all of span_events with no tenant/trace predicate baked in, then UNION ALL into `span_current_all`. With a custom non-inlinable aggregate (jsonb_merge_agg) and array_agg(... ORDER BY ...) in the target list, openGauss is unlikely to push a `WHERE trace_id=?` through the GROUP BY + UNION ALL. Expected behavior: full scan + full-trace aggregation of the entire hot span_events table on every live query — not the 'small, only running traces' cost claimed.**
  - 缓解: Replace the view with a set-returning function span_live(tenant_id, trace_id) that pushes predicates into the inner scan, or mandate callers always filter by (tenant_id, trace_id) and verify with EXPLAIN that the predicate reaches the span_events index scan. Benchmark on a hot table at realistic event volume before claiming 'fast enough'.
- **[HIGH] Fold ordering relies on jsonb_merge_agg(... ORDER BY seq,ts,event_id) and array_agg(... ORDER BY ...) for last-non-null. CREATE AGGREGATE is supported, but ordered delivery under parallel/partition-wise aggregation is not guaranteed deterministic. Critically, a span that starts before midnight and ends after (the prompt's 'born morning / dies afternoon' case) splits across RANGE+INTERVAL ts-partitions; cross-partition partial-array merge order is undefined, corrupting last-non-null and deep-merge results.**
  - 缓解: Fold each span within a subquery that first collects ALL its events (forcing a single sort node) before aggregation; disable parallel aggregate for the fold (query_dop=1 / force_custom_plan) and verify via EXPLAIN. Assert event_id is globally monotonic for a total tiebreaker. Add a regression test for a midnight-spanning span.
- **[HIGH] Cold-tier (CStore) loses GIN(jsonb)/HNSW/BM25 (CStore GIN = tsvector-only; confirmed). Consequence: once a trace freezes to span_current_cold, it loses semantic ANN, JSONB @> filtering, and Chinese BM25 — i.e., the product's headline 'semantic recall + Chinese full-text' silently stops working on historical data (the majority, given months of retention). Designs 1 & 3 wave this off as 'cold = aggregation only,' which is a product-capability regression for an Agent-trace search product, not a minor TODO.**
  - 缓解: Keep a row-store retrieval mirror for cold data (retain span_vectors HNSW + a BM25/GIN-bearing row-store table after the wide analytic columns move to CStore), OR keep span_current row-store and only freeze low-cardinality analytic columns. Decide retrievability retention explicitly per column; never freeze the vector/BM25/JSONB-bearing columns into CStore.
- **[MEDIUM] Filtered-ANN selectivity degradation (the prompt's explicit concern). Design 3 embeds (tenant_id, span_kind, start_time) into HybridANN. When the scalar filter is highly selective (one tenant + status=error + last 1h), the in-graph candidate set can collapse below ef_search, returning far fewer than k or poor recall (filtered-ANN under-fill). Only IVF knob (hybrid_query_ivf_probes_factor) is mentioned, not HNSW. is_sampled sampling (only root/llm/error) further shrinks the population, compounding under-fill for narrow filters.**
  - 缓解: Define a fallback: below a selectivity threshold, brute-force exact scan over the (small) filtered set instead of ANN. Benchmark recall@k vs selectivity for HybridANN+HNSW; raise ef_search adaptively. Reconsider aggressive sampling. Confirm HybridANN cannot build on an empty table — build-order (bulk-load then index) conflicts with continuous ingestion of new tenants/partitions.
- **[MEDIUM] Event-table write amplification & TOAST churn. attrs_patch JSONB plus many nullable hot columns sit in span_events (append-only). High-frequency event_type=update streams multiply rows; when attrs_patch crosses the TOAST threshold (~2KB) each event TOASTs independently with no dedup (CAS only covers payload_ref), bloating partitions with pointer churn. Also USTORE is chosen 'for undo-friendliness' on an INSERT-only table — undo machinery is pure overhead vs ASTORE here; Design 2 itself flags USTORE+INTERVAL as TODO (verified: USTORE supports L1 interval, not subpartitioning, so it parses but is the wrong engine).**
  - 缓解: Use ASTORE heap for append-only span_events (no benefit from undo). Cap attrs_patch size and route large attrs through CAS too. Consider not storing per-event JSONB for chatty 'update' events; coalesce client-side. Measure dead-tuple/TOAST growth per partition before committing.
- **[HIGH] Post-freeze late-event data loss. Freeze gate = root ended AND no events 7d. But a feedback event or async eval/tool_result can legitimately arrive >7d later. After freeze the trace lives in CStore (immutable/append-only) and is DELETEd from hot span_events (DROP PARTITION). A late event then has no landing zone: its fold target span_current row is gone, span_current_cold cannot be updated (CStore append-only + PK-update disallowed). Result: silently dropped or orphaned late event = data loss / inconsistent trace.**
  - 缓解: Maintain a frozen-trace registry; detect on ingest that a trace is frozen and route late events to an exception/append table, then either re-thaw (re-fold + rebuild cold partition, not in-place) or flag 'late events pending'. Keep DROP PARTITION grace window strictly greater than max feedback latency. Make freeze reversible via cold-partition rebuild since CStore has no in-place update.
- **[MEDIUM] dotted_order encoding non-compliant and self-inconsistent. Verified LangSmith format is {rfc3339-microsecond-timestamp}Z{full-uuid} per segment. Design 2 uses lpad(epoch::bigint,20)||to_hex(span_id): (a) epoch-seconds loses sub-second order, so same-second siblings get identical sort keys → nondeterministic DFS → unstable pre/post, contradicting the stability claim; (b) a late node with an earlier true start_time inserts mid-order in dotted_order but desyncs already-materialized pre/post (admittedly goes stale), so 'no renumber needed' holds only for dotted_order while the fast path (pre/post) still needs full recompute; the subtree query mixes the two encodings.**
  - 缓解: Use microsecond/nanosecond timestamp + full unique id per segment (LangSmith-compatible), fixed width, C collation. Accept pre/post must be recomputed per trace on any structural late arrival; treat dotted_order as the only late-stable path and route subtree queries to it (LIKE prefix) until pre/post is rebuilt, gated by encoding_state.
- **[HIGH] Partitioned-table secondary indexes default to GLOBAL. Designs 1 & 2 create indexes on span_events (RANGE+INTERVAL partitioned) WITHOUT LOCAL (e.g., Design 1 idx_se_span/idx_se_trace/idx_se_ingest/idx_se_parent). Verified: omitting LOCAL/GLOBAL defaults to GLOBAL; GLOBAL is btree-only, ≤31 cols, and must be maintained/rebuilt on DROP PARTITION — defeating the 'DROP PARTITION = zero-cost reclamation' that the freeze flow depends on. Design 1's partial index `WHERE parent_span_id IS NOT NULL` as GLOBAL is also disallowed (GLOBAL forbids partial/expression).**
  - 缓解: Add explicit LOCAL to every secondary index on span_events / span_events_cold so DROP PARTITION reclaims them for free. Verify partial-index LOCAL syntax/placement on the target version (Design 3 already flags LOCAL+WHERE as a TODO).
