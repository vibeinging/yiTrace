# 附录 N：第六轮调度红队（两路均 risky —— 时间水位/MERGE/冻结顺序 静默丢数据）

## 复审 1 —— 裁决: risky

**最致命一击:** The single most fatal, design-collapsing flaw is the freeze pipeline ordering combined with MERGE-not-being-upsert-safe (§3.2->§3.4 + §2.3). The pipeline encodes (DFS pre/post), writes cold, THEN registers in frozen_registry. A late event arriving during that window is routed by the ingestor to span_events (not the inbox, because no registry hit yet), but it is never folded (fold loop later excludes the now-frozen trace) and never re-melted (it predates registration, so it is not in the inbox). That event is silently and permanently lost from every served view — it lives only as a dead row in span_events until GC drops it. This is not a corner case: the whole premise is 'spans are born in the morning and die in the afternoon, out of order, arriving late,' so late-arrival-during-freeze is the EXPECTED steady state, not an edge. The fix inverts the pipeline (register-to-inbox FIRST, then snapshot-encode-cold, treat any concurrent arrival as remelt) — but that fix is incompatible with the design's other premise that MERGE-match-once gives idempotent folding, because the only race-safe upsert primitive in openGauss is INSERT ON CONFLICT, not MERGE. So the two foundational primitives the design is built on (MERGE for folding + encode-before-register for freeze) are BOTH individually wrong for the stated concurrent/late-arrival workload, and they are wrong in a way that loses data silently rather than erroring. Verdict-deciding because 'no silent data loss' was the explicit success criterion."}

**缺口:**
- No fencing token / generation number anywhere. Advisory locks prevent two LIVE sessions from overlapping but do not fence a worker that lost its lease, paused, and resumed mid-operation (the resumed worker may issue a stale DROP/EXCHANGE/INSERT after another worker already advanced state). Every irreversible op (DROP PARTITION, EXCHANGE, payload sweep) needs a monotonic generation check committed atomically with the effect, not just an advisory lock.
- The fold loop's correctness depends on 'span_events is append-only, no deletes' — but GC (§2.3) and freeze (§3.5) DELETE/DROP from span_events. Once events are dropped, fold_version=max(event_id) is no longer a safe monotonic dirty-detector for any span whose events were partially dropped, and the reconciliation 'max event_id in events > fold_version' check breaks. The interaction between GC-deleting-events and fold-reading-events on the SAME partition is never analyzed for ordering (must fold-complete before GC, with a durable barrier).
- No analysis of what a live-trace READ sees during folding. span_current is updated by MERGE/upsert under READ COMMITTED; a reader mid-fold sees a partially-folded trace (some spans at fold_version N, others at N-1). For 'active trace queryable while running' this may be acceptable but is never stated as a consistency contract; cross-span invariants (parent exists before child) are not guaranteed during fold.
- Cross-shard traces. Folding is sharded by hash(tenant) (§2.1) but a trace_id belongs to one tenant, so a trace stays in one shard — OK. BUT freeze/remelt are keyed by hash(tenant,trace) on a DIFFERENT hash than fold's hash(tenant). A trace can be in fold-shard X and freeze-object Y with no guaranteed mutual exclusion between the fold worker for shard X and the freeze worker for object Y operating on the SAME span_current rows. The §4.5 exclusion is scan-time-only (TOCTOU), not a held lock.
- late_event_inbox dedup. Late events flow to inbox; remelt re-folds from inbox. If the same late event is delivered twice by ingest (at-least-once delivery is normal), there is no dedup key shown on late_event_inbox (it is LIKE span_events, append-only), so attrs_merge_ordered over duplicated events double-applies non-idempotent patches. event_id uniqueness is assumed but never enforced on the inbox.
- No backpressure on the CONTROL tables themselves. tv_ctrl_remelt_jobs / freeze_jobs / payload_ref_decrement are unbounded queues; a storm of late events (e.g., a misbehaving client replaying a day of traces) floods remelt jobs, each of which triggers an EXCHANGE PARTITION (AccessExclusiveLock) on cold — serializing the entire cold table under a flood. No rate limit / coalescing window on remelt is specified beyond ON CONFLICT coalescing of duplicate (tenant,trace).
- The §4.5 claim 'capture event_id set at TX3-1 start, delete only those at TX3-3' is described but not implemented in the SQL (TX3-3 §4.4 does an unqualified DELETE FROM late_event_inbox WHERE tenant/trace), which WOULD delete late-of-late events that arrived during the rebuild = silent loss. The prose and the SQL contradict each other.

**风险:**
- **[CRITICAL] MERGE INTO is NOT race-safe against concurrent INSERT on the same PK and will raise a unique-constraint violation instead of folding (confirmed: openGauss/PG MERGE does not restart on concurrent insert; only INSERT ON CONFLICT is speculative-insert safe). The design routes ALL of folding (§2.3), remelt re-fold (§4.2 INSERT-on-NOT-MATCHED), and cold->hot rehydration through MERGE on span_current(PK=span_id). Concurrency that triggers this: (a) two fold workers on overlapping ingest_ts bands for the same span if shard assignment ever changes / rebalances / a watchdog steals a lease and the old worker is not actually dead (advisory lock is held by a DEAD session, but a zombie/paused worker that resumes mid-statement is not fenced); (b) a remelt worker doing NOT-MATCHED INSERT of a brand-new late span while the normal fold loop ALSO sees that span_id (the §4.5 exclusion filter is checked at SCAN time, not enforced as a lock, so a fold that already passed the NOT EXISTS check races a remelt INSERT); (c) rehydrate-from-cold re-INSERTs span rows that a late normal-fold also tries to INSERT. Result: statement aborts with 23505, the whole TX rolls back, watermark does not advance (good), but the job is stuck retrying forever if the duplicate is persistent (e.g., rehydrate keeps colliding) = silent stall, not silent data loss.**
  - 缓解: Replace every folding/remelt/rehydrate MERGE with INSERT ... ON CONFLICT (span_id) DO UPDATE SET ... WHERE excluded-guard, which IS speculative-insert race-safe. The fold_version / max(event_id) monotonic guard moves into the ON CONFLICT DO UPDATE ... WHERE clause. This is a direct, mechanical fix but it invalidates the explicit reliance on MERGE-match-once stated in the prompt premise and in §2.3/§4.2. Keep MERGE only where the target PK can never be concurrently inserted (it cannot here, since ingest, fold, and remelt all create span_current rows).
- **[CRITICAL] DROP PARTITION / TRUNCATE PARTITION / EXCHANGE PARTITION take AccessExclusiveLock on the WHOLE table (confirmed), conflicting with every concurrent SELECT/INSERT. This directly violates the stated hard requirement 'active traces are queryable while running' and 'ingest is append-only INSERT'. Freeze step d (§3.5 DROP PARTITION span_events), GC (§2.3 DROP PARTITION span_events / span_current_cold), and remelt cold rebuild (§4.3 EXCHANGE PARTITION span_current_cold) will each FREEZE the entire table for the duration: ingest INSERTs into span_events stall (back-pressure spike or ingest errors / lock-wait timeout), and live-trace read queries block. Worse, the lock is acquired by lock-UPGRADE during the partition operation, so it queues behind in-flight readers AND new readers queue behind it = a classic lock convoy / write-starvation. On a busy ingest table this can cascade into lockwait_timeout aborts of the GC/freeze TX itself (the irreversible step), repeatedly.**
  - 缓解: Never run partition DDL on span_events (the hot ingest table) inline with ingest. Options: (1) schedule all partition DDL in a maintenance window with lock_timeout set low + retry, accepting a brief ingest stall; (2) for span_events use DROP PARTITION only on partitions strictly older than the active write range so the lock contends only with stragglers, and set a short lock_timeout so a failed acquire aborts cleanly instead of convoying; (3) for span_current_cold EXCHANGE, ensure the cold table is NEVER on the read path of live-trace queries (route live reads to span_current only) so the AccessExclusiveLock only blocks cold analytics, not hot serving. Document that partition DDL is a stop-the-world op for that table and budget for it; do not pretend it is online.
- **[HIGH] EXCHANGE PARTITION ... WITHOUT VALIDATION UPDATE GLOBAL INDEX correctness + atomicity is assumed but the design admits it is unverified for column-store ordinary-table source. Confirmed: omitting UPDATE GLOBAL INDEX invalidates the global index; the doc does NOT confirm a column-store ordinary table can be exchanged into a column-store partition, nor that the CREATE TABLE + INSERT...SELECT + EXCHANGE + DROP block in TX3-2 is fully atomic with rollback on crash. If EXCHANGE partially applies or the global index is left invalid, cold reads return wrong/incomplete results SILENTLY (no error), which for a remelt means the corrected late-data version is lost or the trace appears with stale cold rows. This is a silent-wrong-answer path, the most dangerous failure class.**
  - 缓解: Smoke-test the exact EXCHANGE-with-column-store path on the target openGauss version before relying on it; verify global index validity after exchange with an explicit query. Prefer the stated fallback (DROP+recreate partition + INSERT...SELECT in one TX) ONLY after confirming DDL transactionality for interval partitions (note: interval sys_pN partitions cannot be manually ADDed per the design's own finding, so the DROP+ADD fallback may be impossible for interval partitions — this is a contradiction the design has not resolved). If EXCHANGE is unavailable, the cold-rebuild for remelt has no working primitive and the whole remelt round is blocked.
- **[HIGH] Advisory locks do NOT fence a zombie/paused worker and the lease/watchdog can double-execute. The design uses pg_try_advisory_lock for fold (session-level, §0/§2.5) and watchdog-steals-after-2x-TTL (§0). These two mechanisms are inconsistent: if worker A holds the advisory lock but stalls (GC pause, network), its DB session is still alive so the lock is NOT released; the watchdog 'lease_until' expiry lets worker B believe it can take over, but B's pg_try_advisory_lock will FAIL (A still holds it) — so either B skips (safe but the shard is stuck until A's session dies) or, if B uses the lease table instead of the advisory lock as the real gate, B and A both run = double folding / double payload decrement. The two coordination mechanisms (advisory lock vs lease_until watchdog) must agree on which is authoritative; the design uses both and never reconciles them. For freeze/remelt the design uses xact-level advisory in some places and session-level in others, inconsistently.**
  - 缓解: Pick ONE authority. Recommended: use pg_try_advisory_xact_lock (auto-released on TX end / crash) as the sole gate for each unit of irreversible work, and drop the lease_until watchdog entirely (it adds a competing notion of ownership). If long-running work spans multiple TXs (freeze pipeline), re-acquire the xact lock at the start of each step's TX and re-check the durable step column; never assume the lock is held across TX boundaries. Make the lock authoritative and the lease table diagnostic-only.
- **[HIGH] Fold watermark hi_ts = max(ingest_ts) - delta can SILENTLY DROP events when an ingest transaction commits out of ingest_ts order beyond delta. ingest_ts is assigned at row-build time but rows become VISIBLE at COMMIT; a long ingest TX (or a stalled connection) can commit a row whose ingest_ts is far below the current max AND below (last_fold_ts - delta) re-scan band. The design's only defense is the fixed/dynamic delta backstep, but the dynamic delta from pg_stat_activity oldest xact_start does not cover a connection that opened its TX, inserted, then sat idle-in-transaction past delta — that row's ingest_ts is below the watermark band when it finally commits and will NEVER be re-scanned = SILENT DATA LOSS (the span event is in span_events but never folded into span_current, so queries on the folded view miss it permanently).**
  - 缓解: Do not derive the fold cursor from ingest_ts wall-clock at all; derive it from a GAPLESS monotonic sequence that is assigned at COMMIT order, or use a committed-LSN/CSN-based snapshot watermark (fold only events with commit-CSN <= a snapshot taken at batch start). If only ingest_ts is available, the re-scan band must extend back to the oldest possibly-open ingest TX's start AND fold must additionally do a periodic full backfill sweep keyed on a per-span 'last_folded_event_id < max_event_id' dirty check so any never-scanned event is eventually caught regardless of ingest_ts. Add a reconciliation job: SELECT spans where max(event_id) in span_events > fold_version in span_current.
- **[HIGH] payload refcount decrement-before-DROP ordering has a crash window that leaks OR over-decrements depending on idempotency key choice, and the increment/sweep race mitigation is optional. §2.3 mandates 'refcount-1 then DROP partition' in the same logical job but they are in SEPARATE transactions (record decrement into payload_ref_decrement, then DROP). If crash occurs after DROP commits but before the decrement is APPLIED (§3.3 applies asynchronously), the partition is gone so the decrement batch can never be re-derived from span_events — but §3.2 inserts the decrement rows BEFORE drop, so that is covered. However: if the SAME partition is DROPped, then later a remelt rehydrates+re-freezes overlapping data, the batch_id (partition name) may COLLIDE with a stale applied=false row, and ON CONFLICT DO NOTHING will SILENTLY SKIP the new decrement = refcount stuck high = payload never GC'd (leak), or if it re-applies, double-decrement = refcount goes negative and sweep deletes a still-referenced payload = SILENT DATA LOSS of shared payload bytes for OTHER traces.**
  - 缓解: Make batch_id globally unique per decrement event (partition_name + monotonic epoch, never reused), so a re-freeze of overlapping data gets a fresh batch and ON CONFLICT cannot mask it. Add a hard floor: sweep must use 'refcount = 0' exactly AND advisory-lock or grace-window against increment (do not ship the grace window as 'optional' — under concurrent CAS-reuse it is the ONLY thing preventing deletion of a live payload). Add an assertion/alert on refcount < 0 as a corruption tripwire. Reconcile refcount periodically against actual span_events.payload_ref counts for un-dropped partitions.
- **[MEDIUM] Vector index blue-green swap depends on an UNCONFIRMED query-layer contract (dynamic index name from index_lifecycle.index_name). The design itself flags this (§1.5) but it is load-bearing: if the optimizer only binds a fixed index name, the fallback is DROP-then-CREATE-same-name, which opens a brute-force window AND, critically, DiskANN 'cannot build on empty table' means if the DROP succeeds and the CREATE fails/crashes, the tenant is left with NO index and the next build attempt sees a non-empty table (fine) but until then every recall query does a full scan = latency cliff, not data loss. Concurrent recall during the swap may also pin the old index preventing DROP (lock), stalling the lifecycle FSM in BUILDING/STALE.**
  - 缓解: Confirm the query-layer contract first; if dynamic index selection is unsupported, accept the brute-force window only for small tenants and gate DROP-old behind a confirmed successful CREATE-new (never DROP before the new index is ACTIVE and queryable). Make the FSM crash-safe: BUILD_FAILED with the old index name still recorded so recall falls back to old index, not to nothing. Never leave a state where index_name points to a dropped object.
- **[CRITICAL] Freeze judgement §3.1 and remelt re-open create a state-machine livelock / lost-late-event window around the SETTLING<->FREEZING<->FROZEN transitions. The trace state is DERIVED from root+control with no single durable owner. Window: freeze enqueues (step 0) based on 'last_event_at < now()-N days', but a late event can arrive AFTER enqueue and BEFORE step 3 (frozen_registry insert). During steps 0-2 the ingestor checks frozen_registry, finds NO hit (not registered yet), and routes the late event to span_events normally — but step a (DFS pre/post) already ran on the OLD span set, step b wrote cold WITHOUT the late span, and step c registers the trace as frozen. The late event is now orphaned in span_events: not in cold, and ingestor will not re-route it to inbox (it predates registration), and the fold loop excludes frozen traces. = SILENT DATA LOSS of the late event. The §3.4 'register before drop' ordering closes the drop race but NOT the encode/cold-write race.**
  - 缓解: Take the freeze advisory lock and re-validate 'no new events since root_end' INSIDE the same TX that does step a (DFS), and re-check max(event_id) for the trace at step c; if a new event appeared between step a and step c, ABORT the freeze and re-enqueue as a fold-then-remelt. Alternatively, register the trace in frozen_registry FIRST (so ingestor routes new events to inbox immediately) and only THEN encode/write-cold from a consistent snapshot, treating any inbox arrival as a remelt. The current order (encode -> cold -> register) is backwards for race-safety.

## 复审 2 —— 裁决: risky

**最致命一击:** The folding watermark is built on `ingest_ts` (a `now()`-at-INSERT clock value) while the table is partitioned by `ts`, and openGauss now() is transaction-START time under READ COMMITTED — so a long-running ingest transaction commits rows whose ingest_ts is already BELOW the advanced watermark, and they are NEVER folded: permanent, silent loss of span end/attrs/status with no error and no detectable trace. This is not a tunable-δ problem; δ only covers transactions still open at scan time, not ones that started-early-and-committed-late relative to a faster sibling. The same class of bug recurs three more times (fold_version=max(event_id) on a non-monotonic SEQUENCE silently rejecting out-of-order late events; the freeze TOCTOU dropping in-flight-committed late events at the instant of partition DROP; remelt rehydrating fold-base from lossy CStore and erasing original attrs). Every one of these is a SILENT data-loss path, and the design's recovery model — 'resume forward from a co-committed step marker' — assumes DDL-DML transactional atomicity for DROP/EXCHANGE PARTITION that is explicitly unverified on this build; if that assumption is false (plausible: partition DDL often takes internal commits), a crash mid-step-d drops the hot partition while the step marker says 'not done', wedging or double-dropping. The correctness boundary must move from time-watermarks and global-id guards to TRANSACTIONAL dirty-queues populated in the same TX as the ingest INSERT (so 'have I seen every committed event' is an invariant, not a clock guess), and the destructive partition DDL must be proven atomic-or-idempotent-by-existence before any of this is safe to ship. Until those two changes, the system will silently lose late and out-of-order events — precisely the workload it exists to handle."

**缺口:**
- NO RECOVERY for the fold worker's in-flight micro-batch across MULTIPLE (tenant,trace) groups: §2.5 folds each group in its own TX, then advances the shard watermark in a separate TX 'only after committing all groups'. If the worker crashes after folding groups 1..k but before the watermark advance, the next run re-scans the whole window — fine for idempotency, but the design never bounds how many groups a batch contains, so a crash near the end of a huge batch re-does all of it; more importantly the CAS watermark advance `WHERE last_fold_ts = :last_fold_ts` SILENTLY DISCARDS the worker's progress if another worker advanced it, but two workers should never hold the same shard's advisory lock — so the CAS-fails branch is either dead code or a symptom that advisory-lock-per-shard is not actually preventing concurrency (lock released on connection drop mid-batch lets a second worker in). The interaction of advisory-lock-release-on-disconnect with the multi-TX batch is unanalyzed.
- The advisory locks are SESSION-level (pg_try_advisory_lock / pg_advisory_unlock in §0) for fold, but the design ② uses pg_try_advisory_xact_lock (xact-level). Session-level locks survive across the multiple short TXs of the freeze/fold pipeline — GOOD — but they LEAK on crash only until the backend connection is reaped (could be minutes under TCP keepalive), during which a restarted worker is locked out → liveness gap. Conversely xact-level locks release at each COMMIT, so the freeze pipeline's 4 separate TXs are NOT continuously protected by a single xact lock — between step TXs another worker could grab the lock and interleave. The design never reconciles which lock scope each pipeline uses or proves continuous mutual exclusion across the multi-TX pipeline.
- No watermark/cursor persistence for the REMELT inbox drain. §4.5 'capture the set of event_ids processed at start of TX3-1, delete only those in TX3-3' is stated as a recommendation, not built into the schema — remelt_jobs has no column to persist that event_id set. On crash between TX3-1 and TX3-3 the captured set is lost (it was in worker memory), so resume either re-deletes nothing safely or, if it re-captures 'all inbox rows now', deletes late-of-late events that arrived during the crash → silent loss. The persisted-cursor needs to be a column (e.g. processed_up_to_event_id) committed in TX3-1.
- frozen_registry / freeze_jobs / actual-physical-state can diverge with no reconciler. There is no background auditor that detects: (a) frozen_registry says frozen but the cold partition is missing/empty (rebuild crashed); (b) freeze_jobs step=4 (done, deleted) but hot rows still present (DROP silently no-op'd); (c) a trace in span_current with encoding_state=1 but no freeze_jobs row and no frozen_registry entry (orphaned by a crash before enqueue). The whole design is 'forward-only resume from step'; nothing periodically cross-checks the three sources of truth against the physical tables. At months-of-retention scale these drifts accumulate silently.
- Clock-skew / leap-second / NTP-step assumption in every now()-based decision (δ safe_lag, grace windows, expire_at, lease_until watchdog). All workers and the ingest path are assumed to share a monotonic, skew-free wall clock. A backward NTP step on the fold worker makes hi_ts = max_ingest - δ jump backward, re-folding a wide band (safe, idempotent) but also can make a freeze grace-window check pass early → drop a partition before the true grace elapsed → late-event loss. No use of a logical/DB-server clock (e.g. always SELECT now() from the SAME DB session rather than worker host clock) is mandated.
- The design claims it does not need a distributed lock, relying on per-shard/per-trace pg_advisory locks. But advisory locks are SCOPED TO A SINGLE openGauss NODE. The schema/architecture docs contemplate this running on a clustered/信创 deployment; if there is ever a read-replica-promoted standby or a second primary (DR failover), two app nodes pointing at what they think is 'the primary' can each hold the 'same' advisory key on different DB instances → concurrent freeze/remelt of the same trace → the exact double-DROP / double-EXCHANGE corruption the locks were meant to prevent. The 'no distributed lock needed' claim hides an assumption of exactly-one-writable-DB-endpoint that is never stated or enforced.

**风险:**
- **[CRITICAL] SILENT DATA LOSS — the freeze watermark scans/prunes on `ingest_ts`, but `span_events` is partitioned by `ts` (schema line 65), and ingest_ts is `DEFAULT now()` assigned at INSERT under READ COMMITTED. now() = transaction START time in openGauss/PG, NOT commit time. Two ingest transactions T1 (starts t=10, big batch, commits t=40) and T2 (starts t=20, commits t=25): T2's rows become visible BEFORE T1's, yet T1's rows carry the EARLIER ingest_ts=10. The fold worker reads max(ingest_ts)=20 at t=26, sets hi_ts = 20 - delta, advances watermark past 10. When T1 commits at t=40, its rows (ingest_ts=10) are now BELOW the watermark and the (last_fold_ts - delta) safety re-scan band will eventually slide past them too. Those events are NEVER folded into span_current → permanent silent loss of attrs/end/status updates with zero error. The whole δ/safe_lag mechanism is a band-aid over a watermark built on the wrong, non-commit-monotonic column. Worse: it claims partition pruning on ingest_ts in §2.2, but the table is pruned by ts — the scan is a full hot-table scan, not pruned.**
  - 缓解: Do not advance a watermark on a clock-assigned column under MVCC. Either (a) drive folding off a commit-monotonic cursor — a dedicated bigserial/snowshot `fold_seq` assigned at COMMIT via the snowflake event_id IF it is proven commit-ordered (it is not, see below), or (b) use the appendix-I pattern of a per-(tenant,trace) dirty-queue populated transactionally by the ingest path itself (ingest writes (tenant,trace) into a fold_dirty table in the SAME transaction as the event INSERT, so visibility of the dirty-mark and the event are atomic), and fold by draining that queue with FOR UPDATE SKIP LOCKED. This removes the time-watermark entirely and makes 'did I see all committed events' a transactional invariant, not a δ guess. Keep δ only as a secondary latency knob, never as the correctness boundary.
- **[CRITICAL] FOLD CORRECTNESS DEPENDS ON event_id GLOBAL MONOTONICITY THAT THE SCHEMA DOES NOT GUARANTEE. The design switches `fold_version` to `max(event_id)` (snowflake) 'for strict monotonicity'. But appendix-I defines event_id via `CREATE SEQUENCE ... CACHE 1000`, which is explicitly NON-monotonic across concurrent writer sessions (each session burns a cached block; block N+1's session can commit before block N's). The schema doc itself flags this as unresolved ([验] item, snowflake only 'alternative'). If event_id is not strictly monotonic AND not commit-ordered, then `WHEN MATCHED AND f.n_events/max_event_id > c.fold_version` can REJECT a legitimately-newer fold whose max(event_id) happens to be numerically lower than an already-applied one — the late/out-of-order event is silently dropped from the folded state (no error, fold_version guard eats it). This converts the idempotency guard into a silent-loss mechanism precisely for the out-of-order/late events this system exists to handle.**
  - 缓解: fold_version must be derived ONLY from the set of events actually folded, not from a global id whose ordering vs commit is unproven. Safe choice: fold_version = count of distinct (span_id, seq) folded, OR a content hash of the folded event-id set, with the MATCHED guard being `f.fold_version <> c.fold_version` (re-fold whenever the input set changed) rather than strict `>`. Strict `>` is only safe if you can PROVE the version source is monotonic with respect to event arrival into the folded window — which requires either commit-ordered ids or, again, the transactional dirty-queue. Pin down event_id generation (snowflake with a monotonic clock + per-node sequence) as a hard prerequisite and add a midnight-spanning + out-of-order regression test.
- **[CRITICAL] THE ENTIRE CRASH-RECOVERY STORY RESTS ON UNVERIFIED DDL-IN-TRANSACTION ATOMICITY. Freeze step-d (DROP PARTITION + UPDATE freeze_jobs SET step=4 in one TX) and remelt step-2 (CREATE staging + INSERT...SELECT + EXCHANGE PARTITION + DROP TABLE + UPDATE remelt_jobs in one TX) both assume that if the worker crashes mid-statement, the DDL ROLLS BACK atomically and step/state does not advance. openGauss documents that DROP/EXCHANGE PARTITION are *supported*, but there is NO authoritative confirmation that they roll back cleanly when interleaved with DML in the same explicit transaction (web search inconclusive; some openGauss DDL paths take internal commits / are not fully MVCC-transactional like heap DML). If DROP PARTITION commits internally but the subsequent `UPDATE freeze_jobs SET step=4` is lost to the crash, you have DROPPED THE HOT PARTITION (data gone) while step still reads 3 → resume re-runs step-d → second DROP PARTITION on an already-gone partition ERRORS and the job wedges, OR (worse) the partition auto-recreated by a concurrent late INSERT gets dropped again, taking fresh late events with it. The design's 'each step is one atomic TX' claim is the single load-bearing assumption and it is explicitly only 'static-analysis likely'.**
  - 缓解: MANDATORY PoC: in a transaction, do `INSERT into a control table; ALTER TABLE ... DROP PARTITION; <kill -9 the session before COMMIT>` and verify on restart that BOTH the insert AND the drop rolled back. Repeat for EXCHANGE PARTITION + DML. If DDL is NOT atomic with DML, restructure: make the destructive DDL the LAST action and idempotent-by-existence (check `pg_partition` for partition presence before DROP; treat 'already gone' as success not error), and split the step-marker UPDATE into a SEPARATE preceding TX that records 'about-to-drop partition X' so resume can detect a half-done drop. Never co-mingle the progress marker with the irreversible DDL in a TX whose atomicity is unproven.
- **[HIGH] POST-FREEZE LATE-EVENT LOSS WINDOW BETWEEN step 3 (registered) and the ingest path actually honoring frozen_registry. The design says registering in frozen_registry BEFORE dropping the hot partition 'closes the race'. It does not. The ingest path checks frozen_registry per-batch; between the instant step-c commits and the instant a given in-flight ingest transaction (which already passed its frozen_registry check microseconds earlier, under READ COMMITTED) commits its INSERT into the soon-to-be-dropped hot partition, step-d can fire and DROP that partition — taking the just-inserted late event with it. The frozen_registry check and the event INSERT are not atomic with respect to the DROP. This is the classic check-then-act TOCTOU across two processes with no shared lock. The advisory lock is on hash(tenant,trace) in the FREEZE worker, but the INGEST path takes no such lock, so there is nothing serializing ingest against step-d.**
  - 缓解: Either (a) make the ingest path take pg_try_advisory_xact_lock(FREEZE_CLASS, hash(tenant,trace)) before deciding frozen-vs-hot, so ingest and step-d serialize on the same key (cost: per-event lock on frozen traces only — acceptable since frozen traces are rare in the write path); or (b) never DROP a hot partition until a grace window strictly greater than max ingest-commit latency has elapsed since step-c, AND re-scan the partition for any rows of this trace inserted after frozen_at immediately before DROP, re-routing them to late_event_inbox in the same TX as the DROP. The design's own §7 grace-window rule (>max feedback latency) addresses feedback timing but NOT the in-flight-commit TOCTOU at the instant of drop.
- **[HIGH] DROP PARTITION GRANULARITY MISMATCH: freeze is per-(tenant,trace) but DROP PARTITION is per-time-partition (a day of ALL tenants/traces). Freeze step-d's 'Opción A: ALTER TABLE span_events DROP PARTITION p_2026_06' drops an ENTIRE day/month partition the moment ONE trace's job reaches step 3. That partition contains events for thousands of OTHER traces that are NOT yet frozen (still ACTIVE or SETTLING) → mass silent data loss of unrelated live traces. The design hand-waves this in §3.5 ('only when ALL traces in the partition have step=4') but provides NO mechanism to know that — there is no per-partition refcount of un-frozen traces, and the freeze_jobs table is keyed by (tenant,trace) with no partition linkage that would let a worker prove a whole partition is drained. Meanwhile 'Opción B: DELETE selectivo on span_current' deletes from the FOLD table, not from span_events, so the source events are never reclaimed and the partition never shrinks.**
  - 缓解: Decouple the two clocks entirely. Reclaim span_events ONLY via whole-partition DROP driven by a TIME-based retention rule on the partition (drop a ts-partition only when partition_high_bound < now() - max_TTL - max_feedback_latency, i.e. when EVERY trace whose events fall in it is provably past any freeze/remelt window) — never triggered by an individual trace's freeze. Freeze's job is to materialize pre/post + write cold + register; it must NOT drop hot partitions per-trace. Maintain a per-time-partition counter of (traces with events present) decremented as traces freeze, and only let the GC worker DROP when that counter hits zero OR the time-retention rule fires, whichever is safe. Make step-d a no-op for span_events and move all hot reclamation to the time-partition GC worker in design ②.
- **[HIGH] REMELT REHYDRATION CAN PERMANENTLY CORRUPT/LOSE A FROZEN TRACE on crash. §4.2 says if hot rows are gone, 'rehydrate from cold to span_current, then apply inbox'. But (a) cold (CStore) holds ONLY low-cardinality analytic columns (schema §10) — NOT attrs, input_text, tags, encoding details. Rehydrating from cold reconstructs a LOSSY span_current row missing all retrieval/attrs content. The subsequent re-fold deep-merges late attrs onto an EMPTY attrs base → the frozen trace's original attrs/text are silently lost from the rebuilt cold partition. (b) The rebuild reads cold partition + merges, then EXCHANGE-swaps. If the worker crashes after EXCHANGE commits (state still =1 because the marker UPDATE was in the same unproven-atomic TX) the original partition is GONE and the staging is now live but the job thinks it must redo step-2 → it rebuilds from the NEW (already-swapped) partition, double-merging or, if it reads the wrong source, dropping the non-remelted traces. The 'merge = cold-without-this-trace ∪ recalculated-trace' is not idempotent under partial completion because the source of truth (which partition is canonical) flips on swap.**
  - 缓解: (1) Never treat cold as a rehydration source for fold base — cold is lossy by design. The fold base for a remelt MUST be the original events: if the hot partition is already dropped, the original events are GONE, so remelt is impossible without re-reading them. This means late_event_inbox must carry ENOUGH to rebuild, OR the freeze must retain a full-fidelity row-store archive (not just CStore) for any trace that could receive late events — which collides with the whole cold-tiering premise. Resolve this explicitly: keep a row-store retrieval mirror (the schema §10 already mandates one for search) and fold-base remelt from THAT, not from CStore. (2) Make the EXCHANGE idempotent by tagging each cold partition with a monotonic rebuild_version stored in remelt_jobs BEFORE the swap; on resume, compare the live partition's embedded version marker to decide whether step-2 already happened, instead of relying on the co-committed state column.
- **[MEDIUM] STATE-MACHINE WEDGE: overflow (encoding_state=3) and BUILD_FAILED have no automated exit, and a poisoned job blocks its (tenant,trace) forever. §3.2: if DFS finds a cycle/orphan, encoding_state=3 and the freeze job is 'marked with error, removed from normal queue → manual review'. But the freeze_jobs row is NOT deleted (step stays <4), so (a) the trace's hot partition is never reclaimable through the normal path, and (b) any late event for it enqueues a remelt that takes BOTH advisory locks and immediately hits the same un-encodable topology → remelt also wedges. There is no dead-letter, no attempts cap → max_consec_errors actually triggering a terminal state, no alerting wired to these. A single malformed trace (cycle from a buggy SDK assigning parent_span_id = self) silently pins a partition and starves GC.**
  - 缓解: Add a terminal DEAD state to every job state machine with an attempts cap (the design has `attempts`/`last_error` columns but never increments/checks them in the pseudocode). On cap, move the job to a dead-letter table, EMIT an alert, and CRITICALLY decide the partition-reclamation policy for dead traces (fall back to linear-by-seq encoding so the trace is still freezable/droppable rather than pinning the partition). Wire max_consec_errors → backoff → dead-letter for real; right now it's a column with no code path.
- **[MEDIUM] PAYLOAD REFCOUNT LEAK/UNDERFLOW on the DROP-PARTITION + remelt interaction. Design ③ decrements refcount before DROP via the batch_id idempotent queue — correct in isolation. But remelt re-introduces events (and their payload_refs) for a trace whose partition was already GC'd and refcounts already decremented. The late events' payload_refs get +1 on re-insert, but if the SAME payload was decremented to 0 and swept between the freeze-drop and the remelt, the remelt's reference now points to a DELETED payload_store row / freed external blob → dangling pointer, silent content loss when the user opens that span. Conversely, double-decrement on a crash between refcount-apply (committed) and partition-DROP (separate step) is handled, but the cross-feature ordering refcount-dec(freeze) → sweep(0) → remelt-reref is not analyzed anywhere.**
  - 缓解: The payload sweep grace window (payload.sweep_grace_sec, default 300s) is FAR shorter than the remelt/feedback window (days). Make payload sweep respect the SAME frozen-trace late window: never physically delete a payload whose only-recent-referencing traces are within their remelt-eligible window, OR have remelt re-materialize (re-CAS-insert) any payload it references that has refcount<=0, under the per-sha advisory lock, BEFORE rebuilding. Add a regression test: freeze trace → GC drops partition → payload sweeps to 0 → late event arrives referencing that payload → assert content still retrievable.
