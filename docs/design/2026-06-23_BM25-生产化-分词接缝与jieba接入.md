# 检索索引生产化：接缝 + 团队 jieba / graph_index FFI 接入

> 2026-06-23 ｜ 现状索引见 [CURRENT_STATE.md](../CURRENT_STATE.md)。本篇讲两块自有索引（BM25 中文检索 +
> graph_index 向量 ANN）怎么从「验证级 std-only 实现」过渡到「接团队真库」的生产路径。两块同一套模式：
> **trait 接缝 → 引擎 `CoordinatorBuilder` 注入口 → 工作区外独立 FFI crate（Vortex 同款隔离）→ 离线 mock 测试**。

## 1. 背景：现有 BM25 到底缺什么

`yt-engine/src/bm25.rs` 的 `Bm25TextIndex` **不是占位**——真倒排（token→文档→词频）+ idf + 文档长度归一（k1=1.5, b=0.75）+ 确定性排序都在。但离生产级有 5 个缺口：

| 缺口 | 性质 | 需要 jieba？ |
|---|---|---|
| 分词 = bigram 无词典 | 精度损失（跨界假词 似盗/刷风）、无词级语义 | **是** |
| 没有 block-max-WAND | top-k 全量扫 postings，百万文档慢 | 否，纯 Rust 自有 |
| 删除/更新不处理 | span 被取代后倒排留旧词 | 否 |
| 纯内存、无段内倒排 | 重启全 rebuild、内存无界 | 否 |
| 全局单锁 | index/search 抢一把锁 | 否 |

**关键判断：5 个里只有「分词」需要 jieba，其余 4 个是纯 Rust 自有索引的活、不卡 C 库。**

## 2. 路线决策（已定）

- **走「接团队 jieba（FFI）」，不自研纯 Rust 词级分词。** 团队已有实战过的 cppjieba + 词典，重造无意义；
  且产品命题是「自有 IP 当一等公民」，jieba 是团队 IP，接它正合命题。买的是 Vortex（列式*格式*），
  不是整套引擎——查询/索引层仍自有。
- **分工：FFI 复用团队分词，倒排 + BM25 评分 + WAND 仍是引擎自有 `Bm25TextIndex`。**
  这是决策文档 §2.1「FFI 复用评分/分词、重写存储」的落点。

## 3. 已落地（2026-06-23）

1. **分词接缝**：`bm25.rs` 抽出 `pub trait Tokenizer`，`Bm25TextIndex` 改持有 `Box<dyn Tokenizer>`，
   默认 `CjkBigramTokenizer`（行为不变）。换分词器 = 换实现，倒排/评分一行不动。
2. **引擎注入口**：`CoordinatorBuilder`（`with_tokenizer` / `with_bm25` / `with_graph`），覆盖
   `build/open/open_durable`；不传 = 默认 bigram，所以 `WriteCoordinator::new/open/open_durable` 行为不变。
3. **纯 Rust 中文词级分词 `ChineseTokenizer`**（`tokenizer_cn.rs`，std-only 零依赖，**在引擎里、已设为引擎默认**）——
   团队 cppjieba 真库不在本环境，索性自己写一套生产级的，不等 C 库：
   - 算法 = jieba 默认 `cut` 核心两步：**词典 DAG + 最大概率路径 DP**（用词频解切词歧义，不是 bigram 盲切）。
     经典 "研究生命" → 研究/生命，靠 P(研究)·P(生命) > P(研究生)·P(命)，有测试钉死。
   - **全量词典默认装满**：jieba 全量 `dict.txt`（34.9 万词，MIT）**`include_str!` 嵌进二进制**，
     `ChineseTokenizer::full()` 默认就用——**开箱即生产级**，契合单二进制私有化部署（不外挂词典文件）。
     之上叠 `DOMAIN_DICT` 领域词（风控/盗刷/会话/超时… jieba 没有的，`add_absent` 不覆盖原词频）。
   - **自有词典导入**：`with_user_dict(text)` 在全量词典上叠用户词（同 jieba 格式，可覆盖词频/加生词）；
     `with_dict(Dict)` 完全自定义。词典来源/许可见 `data/JIEBA_DICT_NOTICE.md`。
   - **OOV**：未登录词按单字（DP 里单字概率低，自然让位词典词）。HMM/Viterbi 救 OOV 是下一步增强。
   - 接真 `Bm25TextIndex` 端到端测过；8 单测（词级切分、歧义、混排、OOV、全量词典默认、自有词典叠加、自定义词典、接 BM25）。
4. **jieba FFI 独立 crate** `yitrace-tokenizer-jieba`（Vortex 同款隔离、工作区外）——**保留**作另一条路：
   团队真库就位时也能 FFI 复用（同 trait 接缝，二选一）。`JiebaTokenizer` + ABI 契约 + 离线 mock（3 测）。

测试：引擎 90 单测 + jieba mock 3 测全绿；引擎依赖树仍零外部依赖（词典是嵌入数据，不是依赖）。

> **引擎默认 = 纯 Rust `ChineseTokenizer`（jieba 全量词典）**，开箱即词级中文检索。两条分词路同一 trait 接缝：
> ① `ChineseTokenizer`（默认，加载全量词典 + 自有词典叠加）；② FFI `JiebaTokenizer`（接团队 cppjieba，备用）；
> ③ `CjkBigramTokenizer`（无词典 bigram，兜底）。一行 `with_tokenizer` 切换。

## 4. 向量 ANN：同一套模式（2026-06-23 同步落地）

ANN 的接缝（`GraphIndex` trait）与注入口（`CoordinatorBuilder::with_graph`）和 BM25 共用，已就位。
不同点：BM25 只有「分词」走 FFI（倒排/评分自有）；**ANN 整套（建图/搜索/过滤）都是团队 graph_index 的 FFI**，
本 crate 只做编组。难点在护城河——

- **进图过滤跨 FFI**：带过滤近邻搜索要在**图遍历过程中**回调谓词决定收点（不是搜完再筛——选择性谓词下
  post-filter 召回会崩，见 `yt-engine/graph.rs` 的实测对照 1%→post 0.17 / in-graph 1.00）。
  ABI 让 `vexgraph_search` 收 `(filter_ctx, filter_fn)`，C 遍历时回调；本 crate 把引擎的
  `&dyn Fn(u64,u64)->bool` 闭包包成 thin 指针 + trampoline 传进去（`catch_unwind` 兜 panic 不穿越 C 边界）。
- 独立 crate `yitrace-vecindex-graph`（Vortex 同款隔离），`mock` 默认 feature 用 Rust 桩（精确过滤
  top-k）验证**回调真跨边界被调用** + 结果过滤排序正确（4 测，含捕获环境的闭包、维度不符安全）。
- ABI 提案契约见该 crate `ABI.md`；真库 `--features link` + `VEXGRAPH_LIB_DIR` 接。

## 5. 待做（按序）

1. **两库真链接**（构建机）：拿到团队 jieba / graph_index → 核对各自 ABI → `--features link` 跑通
   → 真召回对标（jieba 词级 vs bigram；graph_index 进图过滤 vs 当前 NSW 实测）。
2. **BM25 段内倒排 + block-max-WAND**：当前内存倒排建游标时已枚举全部 postings，WAND 收益要等 postings
   落到持久/段内才兑现，故二者同阶段做。验证：WAND top-k 与全量打分逐位一致 + 跳过计数。
3. **BM25 删除/更新处理**：span 取代/删除后从倒排移除 postings。非正确性 bug（`join_folded` 已按快照兜底），
   高 churn 下提精度。
4. **ANN 量化/SIMD**：团队 graph_index 自带（algorithm/distance/PQ 经 FFI 复用），随真库链接到位。
