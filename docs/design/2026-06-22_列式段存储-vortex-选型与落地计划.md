# 列式段存储：Vortex 选型与落地计划

> 日期：2026-06-22
> 决策：**列式段存储采用 Vortex，team fork 一份自维护。** 查询执行用 DataFusion。中文 BM25 / 图式向量索引继续自研（团队 IP）。
> 背景：要支持几 T 量级数据，行式自研编码不够（长跨度聚合/全库检索会拖大文本列）。需上列式。

---

## 1. 决策

| 项 | 决定 |
|---|---|
| 列式段格式 | **Vortex**（fork 自维护，钉死版本） |
| 查询执行 | DataFusion（原生读 Vortex/Parquet） |
| 中文倒排 / 向量索引 | **继续自研**（团队 IP，作 DataFusion 自定义扫描节点接入） |
| std-only 骨架 | 保留作正确性回归与参照，列式段做成独立 crate / feature 隔离 |

**为什么不自己写列式格式**：几 T + 长寿命金融数据，自写等于重造 Vortex/Parquet——压缩跟不上（= 多花磁盘/IO，恰是 TB 规模最痛）、"几年后还读得了"的格式里易埋细微 bug。团队 IP 该花在差异化（中文 BM25、图式向量、引擎逻辑），不是大路货字节格式。

---

## 2. 候选对比（协议均已核实，2026-06-22）

| 格式 | 协议 | 成熟度 | Rust/DataFusion | 格式稳定 | 随机访问 | 结论 |
|---|---|---|---|---|---|---|
| **Vortex** | **Apache-2.0** | v0.75，LF 孵化，每周发版 | 好（DataFusion 可读） | **0.36+ 冻结、向后兼容** | **快 Parquet ~100×** | **选它** |
| Apache Parquet | Apache-2.0 | 极成熟，PB 级生产 | 一等（DataFusion 原生） | 30 年稳定 | 弱（为扫描设计） | 备选/兜底 |
| Lance | Apache-2.0 | v7，活跃 | 好 | 仍演进 | 中 | 已否（中文 BM25 OPEN、commit 冲突、格式 churn） |
| Nimble（Meta） | Apache-2.0 | 实验性，明说"无稳定性保证" | C++ | 不稳定 | — | 否（现在不能押多年） |

**四者都 Apache-2.0** —— 无 AGPL/BSL/ELv2/SSPL 雷，协议不是区分点，纯按技术选。

---

## 3. 为什么 Vortex 比 Parquet 更贴我们

我们的检索是**索引驱动取行**：BM25/graph 给出命中行 → 去段里取那几行（= **随机访问**）。引擎已实现的"只折叠命中行"正是这个模式。

- **Vortex 随机访问快 ~100×、扫描快 10–20×** —— 正中我们的访问模式（随机点取 + 长跨度聚合）。
- **Parquet 随机点查恰是弱项**（按行组/页解码），会拖累"找相似/检索"延迟——而那是我们的卖点。

当初对 Vortex 的最大顾虑是"格式 churn 不敢押长寿命数据"——**已消除**：文件格式自 0.36 冻结并承诺向后兼容，我们接的是 0.75。

## 4. 风险与缓解（fork 自维护是核心缓解）

| 风险 | 缓解 |
|---|---|
| LF 仍在孵化、younger、无公开 TB 生产案例 | **Apache-2.0 + 格式规范公开** → 最坏情况自己 fork 维护 / 自写 reader，数据永不锁死 |
| 库 API 高频变更（~7–8 release/月）= 维护税 | **fork + 钉死版本**，按自有节奏升；churn 在库 API、不在文件格式，盘上数据安全 |
| 上游若停更 | 已 fork，自维护；格式冻结，存量数据照常读 |
| 破 std-only / 离线编译 | 列式段做**独立 crate / feature**；fork 进内网镜像 + vendoring 支持气隙离线编译；std-only 骨架保留作回归 |

**fork 策略**：fork `spiraldb/vortex` → 内网镜像，钉版本（如 0.75），需要的修复自己 backport，离线 vendoring 供气隙构建。等于"把外部项目纳入自有代码库"，合"自主可控"定位。

---

## 5. 落地计划（分阶段，`SegmentStore` trait 是接缝）

引擎早已把段存储抽象成 `SegmentStore` trait（`flush_to_segment` / `scan_fold_inputs` / `scan_records` / `unlink_segment`），就是为这一刻留的换入点。

**Phase 0 — fork & vendor**
fork Vortex 到内网、钉版本、配 vendoring（离线/气隙可编译）。最小化引入 DataFusion 所需件。

**Phase 1 — `VortexSegmentStore`（列式段落盘）✅ 已落地（2026-06-22）**
独立 crate `yitrace-segstore-vortex/`（工作区外）实现了 `SegmentStore`：`flush_to_segment` 把一批 `WalRecord` 建成列式 `StructArray`（20 列，每字段一列）写成 `.vortex` 文件；`scan_records` 读回（`into_arrow_preferred` 转 Arrow 逐列取值）逐字段重建。`columnar_segment_round_trips_all_fields` 测试通过（含可空列、中文、logs、fold-input 行号映射、unlink）。**引擎工作区离线 84 测试照常、依赖树无 vortex —— 骨架零污染。**
- 关键实现点（踩过的坑）：① 写到内存 `ByteBufferMut` 再 std::fs 落盘（`tokio::fs::File` 不被 `VortexWrite` 接受）；② session 必须 `.with_tokio()` 配运行时句柄、且在 `rt.enter()` 上下文里配；③ 开 vortex 的 `tokio` feature。
- 待清理：`into_arrow_preferred` 已 deprecated（换 `execute_arrow`）；logs 用 NUL 连接压成单列（v1 简化，后续换真 list<utf8>）。

**Phase 2 — 投影 + 谓词下推（列式的收益兑现点）🔄 谓词下推已落地（2026-06-22）**
- ✅ **谓词下推**：`VortexSegmentStore::scan_in_time(seg, from, to)` 用 Vortex 表达式 `and(gt_eq(col("ts"),lit(from)), lt_eq(col("ts"),lit(to)))` + `scan().with_filter(...)`，把过滤**推进文件扫描**、只解码命中行。测试 `predicate_pushdown_filters_by_time_range`：5 行段查 [200,400] 只回 3 行、窗外回空、单点回 1 行。私有 `read_filtered(seg, Option<Expression>)` 统一带不带过滤两条路。
- ⏳ **投影下推**（只读需要的列）：API 已知（`select(fields, root())` + scan 投影），但**最大收益要等引擎有"列感知读路径"**——现在 `scan_records` 必须重建完整 WalRecord（要全字段）；投影对"聚合只读 token/agent 列、不碰大文本列"才发力，需引擎侧改聚合路径返回窄结果。先记，待引擎配合。
- ✅ **接进引擎（2026-06-22）**：`SegmentStore` trait 加默认方法 `scan_fold_inputs_in_time(seg,from,to)->Option<Vec<FoldInput>>`（默认 None=不下推，std-only 存储不变、零回归）；`fold_query` 在「段无删除（deletion_seq==0）+ 有真实时间窗」时调它走下推，否则回退全扫（**下推丢物理行号，删除按物理行号，二者不能共存**）。VortexSegmentStore 覆盖该方法接 `scan_in_time`。注意：段扫描路径本来**没有**行级时间过滤（只段级 zone-map），所以下推给列式存储**新增**了与 memtable 一致的行级时间过滤。测试：引擎 mock `segment_time_pushdown_used_and_row_filters`（验证有窗才下推 + 行级过滤）；Vortex 端到端 `engine_uses_vortex_pushdown_end_to_end`（WriteCoordinator+VortexSegmentStore，时间窗读穿过引擎走真下推）。引擎 85 测试绿、依赖树无 vortex；segstore 3 测试绿。

**Phase 3 — DataFusion 查询执行**
把手写的 `read_spans_query`/折叠换成 DataFusion `ExecutionPlan`，BM25/graph 作自定义扫描节点给出行选择。这是当初锁定的目标架构（Rust + DataFusion + Vortex + 自有索引）。

**Phase 4 — 索引驱动的随机取行**
把"只折叠命中行"接到 Vortex 随机访问（按索引给的行指针直接取行）—— Vortex 100× 随机访问在这里兑现价值。

## 6. 列 schema 映射（SpanFields → Vortex 列）

一个段 = 一个 Vortex 文件，列对应折叠字段（身份列 + 可折叠列）：

| 列 | 类型 | 用途 |
|---|---|---|
| trace_id, span_id | u64 | 分组键、zone-map |
| ext_span_id | utf8 | 去重身份（event_id 重算用） |
| seq, event_type | u64/u8 | 折叠定序、类型 |
| ts | i64 | **时间 zone-map 主键**（剪枝核心） |
| status, duration_ns | u8/u64 opt | 摘要/过滤 |
| parent_span_id | u64 opt | 树/图 |
| input_tokens, output_tokens | u64 opt | 成本聚合 |
| session_id, agent_name, tool_name, model | u64/utf8 opt | 会话/成本/过滤维度 |
| **input_text, output_text** | utf8 opt | **eval 评测对象；大列，投影下推主要受益对象**（多数查询不读） |
| eval_score, eval_label | u32/utf8 opt | 评测结果（upgrade 补写） |
| logs | list<utf8> | 日志并集 |

**关键**：input_text/output_text 是大列，列式让"数 token / 列表 / 聚合"等查询**完全不碰它们**——这是上列式最大的单点收益。

> 删除位图 / upgrade 补写块 / manifest 仍按现有自研 + persist.rs 落盘（它们小、且是引擎自有逻辑，不进 Vortex）；Vortex 只接**不可变段的列式数据**。

---

## 7. 影响与边界

- **破 std-only / 离线**：用独立 crate/feature 隔离；fork+vendoring 支持气隙离线编译；零依赖骨架保留作正确性回归。
- **这是多周工程，不是一次提交**：当前 std-only 沙箱无法联网拉 crate，本计划是供团队执行的蓝图；可先做的离线 prep = 锐化 `SegmentStore` trait（投影/谓词感知）、固化列 schema。
- **不变的**：确定性 event_id、四源折叠、快照隔离、审计级不重不漏、中文 BM25、图式向量——这些自有 IP 与正确性骨架不动，Vortex 只换"段的字节怎么摆"。
