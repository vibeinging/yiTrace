# 附录 Q:Level-1 红队复审(两路)

## 复审 1 —— risky

**最致命一击:** A 方案最核心的卖点——\"宽数据进自管段、堆表只留占位行+ctid、复用标准 IndexScan/amgettuple 契约、对外 SQL 不变\"——与本版 openGauss 的执行器实现直接冲突,我已亲验:IndexNext(nodeIndexscan.cpp:100-114)从 AM 返回的 TID 处取的是堆元组(scan_handler_idx_getnext 返回 HeapTuple,ExecStoreTuple 进 slot),上层投影列与 MVCC 可见性全部基于该堆元组(index_fetch_heap/IndexFetchTuple,indexam.cpp:554-577)。amgettuple 只能产 TID,产不了列。DiskANN/BM25 这两个被引为先例的 AM 恰恰证伪了 A 的用法:它们索引的是\"真实堆表上的真行\"(diskann_build.cpp:219 IndexBuildHeapScan(heap)),宽数据只是额外副本,堆里有源真行——而 A 把这关系倒置。结论:堆里要么仍有每事件真行(膨胀短板①根本没搬走,还多复制一份到段)、要么只有占位行(query 返回占位数据)。要让 AM 不回堆直供合成的折叠当前态列,必须做 CustomScan 或带自管可见性的 index-only 合成 IndexTuple——而设计把这一整块推给\"二期评估\"(§B.8 自承\"openGauss 对列式晚物化支持度未确认\")。叠加第二刀:新增 Index AM 无 CREATE ACCESS METHOD 路径(gram.y 无此语法),必须编进 bootstrap pg_am.h + 内核二进制,即\"动内核\",直接触发团队自己 2026-06-17 决策摘要列为承重风险的\"内核级重测评/丢批次窗口\",却被本设计包装成\"轻 fork、对信创回炉最友好\"。所以裁决:A 的可行性命门(AM 不回堆直供折叠列 + 自管可见性)和认证代价(动内核)两件最关键的事都被\"二期/待验/轻 fork\"话术回避了。必须把\"自定义 AM 能否在本版 openGauss 上不回堆、从段直供多列并正确判可见性\"提为先于一切的 Phase 0 第0号 go/no-go PoC——这一项不绿,A 整条路线连\"对外 SQL 不变\"都做不到,应回落比较 C(CStore Level 0.5)是否才是现实终态。

**仍在回避(punts):**
- 最大 punt:把"AM 直接产列值、不回堆表(真列式投影/晚物化)"整段推给"二期评估 CustomScan / 升格 B 的 orientation"(§B.8),但这正是 A 能否成立的命门——没有它,堆里要么留每事件真行(膨胀短板①未解、只是又复制一份到段),要么留占位行(query 返回的是占位数据)。用"二期"把 A 的根本可行性问题藏起来。
- 第二大 punt:把 fold 性能"不再线性变慢"的兜底再次推给"若 PoC 实测仍是瓶颈→升格 B 的向量化折叠算子(§B.10 已预埋钩子)"。这与团队 v1 被批评的"把折叠吞吐推给以后 benchmark"是同一个回避动作,只是换成了"以后升格 B"。折叠是否够快从头到尾没有任何实测或量级估算。
- 把"压缩"短板②的真压缩(FST/FSST/ALP/zero-copy)推给二期,一期只有弱编码,等于宣称根治了"压缩 XOR 可检索"但 MVP 实际只交付了"可检索 + 弱压缩","对标 Vortex 同字节根治"在一期是话术。
- 把跨段树一致性、段内按 pre 重排成本 vs 收益、folding 快照隔离,全部打包进"【PoC待验】",用待验标签代替设计——这些不是参数细节,是会决定数据正确性的核心规则,属于设计应当先回答的,不是 PoC 顺带验的。
- 把"信创回炉"成本用"轻 fork、对回炉最友好、内核固定表改动接近零"三连乐观措辞淡化,实质是把决策摘要明确标红的"动内核=触发内核级重测评=可能丢半年批次窗口"这一承重风险偷偷降级。这是对组织已有诚实结论的回避。

**风险:**
- **[CRITICAL] KILL-SHOT 架构错配:把宽数据放进自管段、堆表只留"占位行+ctid锚"——但标准 IndexScan 执行路径从 AM 返回的 TID 处取的是堆元组并从堆读投影列。亲验 nodeIndexscan.cpp:100-114 IndexNext 调 scan_handler_idx_getnext 返回 HeapTuple,ExecStoreTuple(tuple, slot, indexScan->xs_cbuf) 后上层从该堆元组读列;indexam.cpp:537-577 index_fetch_heap/IndexFetchTuple→tableam_scan_index_fetch_tuple 取"可见堆元组"并在堆上判 MVCC。即 amgettuple 只能产 TID,列值与可见性都来自堆。DiskANN/BM25 正是这么用的:它们 IndexBuildHeapScan 真实堆表(diskann_build.cpp:219)、每行存 heapTid 回指真行,宽数据(向量)是额外副本,堆里仍有源真行。设计把这关系倒置(堆=占位),结果:① 若每 span/event 仍各有一行真堆元组,则 ASTORE 膨胀短板①根本没搬走,只是又复制一份到段;② 若堆里只有占位行,query 返回的就是占位数据而非宽列,折叠/检索取不到真实列。§B.8 自己写"起步先 amgettuple+回堆表(宽列放 AM 段,堆表只留占位行+ctid)"是自相矛盾——回堆表拿到的就是占位行。**
  - 缓解: 要让 AM 直接供列,只有两条真路:(a) 完整 index-only-scan——amcanreturn=t + xs_want_itup,amgettuple 产出携带"全部投影列"的合成 IndexTuple,且要让可见性绕开堆(LSM 无 heap VM,index-only 的 all-visible 前提不成立,需自管可见性);(b) CustomScan 自定义扫描节点(§B.8 自标 openGauss 列式晚物化支持度未确认)。两条都不是"复用标准 IndexScan/amgettuple 契约"的轻活,本质是 B 方案的执行器工程被提前到 A。必须把这个判定作为 Phase 0 第0号 PoC(先于 P0-1/2/3):在本版 openGauss 上证明一个自定义 AM 能不回堆、从段直接供多列给执行器并正确判可见性,否则 A 的"轻 fork+对外 SQL 不变"整体不成立。
- **[CRITICAL] "轻 fork / 对信创回炉最友好"是失实定性。亲验 gram.y 无 CREATE ACCESS METHOD 语法、pg_am 是 PG9.2 老目录无 amhandler 列(pg_am.h:35-72);新增 Index AM 必须写进 bootstrap pg_am.h 硬编码 OID + 内置 C 函数,随内核编进 gaussdb 二进制(diskann/bm25 都是这么进去的,见附录 K)。即 traceseg = 改内核二进制 = 送测客体变更。团队自己的 v2 决策摘要(2026-06-17)把"动内核→触发内核级重测评、CESI 一年约2窗、错过丢半年"列为承重风险,并把"v1 不动内核"作为战略前提(决策摘要§0/§5/附录 K-L 全员结论:v1 零新内核 C 改动)。本设计直接逆转该前提却包装成"对信创回炉最友好",是把最该正面承认的认证代价偷换成乐观措辞。**
  - 缓解: 诚实重定级:traceseg 是"中等内核 fork"(新增独立编译单元 + bootstrap 目录 + 可能新 RM + 执行器接缝),认证上等同"重过内核级测评",不是"轻"。必须与决策摘要§3 的认证时间线(测评段串在产品成熟之后 + 排批次窗口)对齐重算首单时间,并在 Phase 0 商务子项里向 CESI 书面确认"内核新增 Index AM"是否触发底座栈重测(而非仅应用层增量)。在这条坐实前,A 不应被描述为比 Level 0 更省认证。

  - 缓解: Phase 0 必须新增一项:证明自定义 AM 能在"无 order-by-op、纯标量谓词"下被优化器选中并经 amgettuple 高效产 TID(indxpath.cpp:1047 需 found_clause——即谓词算子要匹配 AM 注册的 opclass strategy)。同时"多谓词 bitmap-AND 在单 AM 内合并"要明确:因无 amgetbitmap,traceseg 无法与其他索引经 BitmapAnd 组合,所有谓词必须在 AM 内自己消化,跨索引组合能力丧失,需评估是否可接受。

  - 缓解: 这与第1条同根:合成行无法用"堆 TID + 堆可见性"表达。需要 AM 自带可见性与合成行直供(CustomScan/index-only 合成 IndexTuple)。在 PoC 第0项里一并验证:折叠合成行如何携带 xs_recheck/可见性、并发快照下的可重复读如何保证(段不可变有利,但 memtable/L0 活段 + deletion/upgrade 的快照一致性需明确设计,目前文档只说"活段走热缓冲",未给快照隔离规则)。

  - 缓解: 把"AM 直供列 + 自管可见性"从二期提到 Phase 0/Phase 1 关键路径并据此重估工时(很可能 MVP >20 人月)。若 Phase 0 第0项证明本版 openGauss 的 CustomScan/列式晚物化不支持,则 A 退化为"段里存宽数据但仍要回堆取真行"=膨胀未解,需重新比较 C(Level 0.5 CStore)是否其实是更现实的终态。

**缺口:**
- 全程静态读码,未编译未 initdb 未跑任何 PoC——这一点设计自己承认,但"地基坐实"的措辞过强。最致命的"AM 能否不回堆直供列并自管可见性"在本版 openGauss(PG9.2.4,无 PG12 pluggable AM、可能无成熟 CustomScan 列式晚物化)上完全未验证,却被默认可行。
- 并发/快照隔离规则缺失:活段(memtable/L0)+ deletion vector + upgrade map 在 MVCC 快照下如何保证可重复读、如何与 DiskANN 增量 insert 对账,文档只字未给。金融政企对"审计可复现"零容忍(决策摘要§4),这是准入项。
- 折叠正确性只有"规格+一致性测试集"的承诺,无任何 EXPLAIN/benchmark/丢数据回归证据。决策摘要§5 与附录 N 已把"晚到+乱序不丢事件"列为 risky 必验项,本设计把晚到说成"=普通写一个新段、零特殊路径",但没证明 upgrade map 在乱序+崩溃下不丢/不串。
- 成本与时间未对齐团队官方账本:决策摘要把首单定为 28-40 月、烧 5000万-1.1亿,且明确"动内核"会加重认证长杆;本设计的 13-15 人月 MVP 只算了工程,未算认证回炉与招聘期,且与"不动内核"战略直接冲突,却未在文中正面处理这个冲突。
- FST/ALP/FSST/zero-copy compaction 均标"二期/自研、本仓无现成",但压缩比与随机读的核心卖点(对标 Vortex)恰恰依赖这些;一期 plain 兜底意味着 MVP 的"压缩"短板②只解了"内嵌倒排可检索"一半,"压缩"那一半在一期是 plain/delta/RLE/dict 的弱压缩,未达 Vortex 形态。

## 复审 2 —— sound-with-fixes

**最致命一击:** 读码证明:你引为'轻 fork、内核改动接近零'的两根支柱在本仓自相矛盾。(1) disk_container 模板自身把 DiskANN WAL 写死在容器里——diskvector.hpp:244/268/291/321 直接调 `_logmgr.diskann_xlog_add_elem` → `XLogInsert(RM_DISKANN_ID, XLOG_DISKANN_SET_ELEM)`,而 redo(redo_diskannxlog.cpp:193)是按 DiskANN 专有 op-code 分派、绝非 relation-agnostic;你自己引的先例 BM25 同样用 disk_container,却照样新建了 RM_BM25_ID + 完整 bm25xlog.cpp(8 种记录)+ redo_bm25xlog.cpp + redo_xlogutils.cpp 两处注册——TraceSeg 的段元数据/删除位翻转/upgrade-map 记录根本映射不到 DiskANN op-code,所以'复用 RM_DISKANN redo、固定表改动接近零'被先例直接证伪:你必须像 BM25 一样新增 RM(横跨 redo_xlogutils + 3 个 recovery dispatcher 共 15 处),这就是 Phase 1 的内核二进制变更,信创回炉在 Phase 1 就触发,而非你声称的 Phase 3。(2) amgettuple 只回 ctid 让执行器回堆表取元组(diskann_scan.cpp:134),index-only-scan 只能返回'被索引的键列',无法返回任意宽 payload——所以'堆表只留占位行+宽数据全在 AM 段+SQL 不变'物理上不成立:要么堆表存折叠后的宽行(ASTORE 死元组膨胀回归,短板①前提崩塌),要么必须上 CustomScan/列供给(那正是你推给 Phase 3 的方案 B 的主要成本)。A 相对 B 的'轻'与'信创友好'优势,建立在两条经不起读码的假设上;诚实结论是 A 与 B 的成本差被显著高估,而 A 的两处生死项(新 RM 必建、执行器列供给)必须在 Phase 0 拍板,否则 MVP 工期与信创时间窗会双双踏空。

**仍在回避(punts):**
- 短板② '压缩 XOR 可检索 同字节根治' is partially punted to Phase 2. Phase 1 uses plain encoding for the high-value string columns (name/dotted_order via FSST = Phase 2) and floats (cost/latency via ALP = Phase 2), and DiskHashTable instead of FST. So the actual 'compress + searchable on the same bytes' for the columns that matter most is deferred, while the root-cause claim is written as already-achieved. The design says 'FST/ALP 二期补, 不阻塞' — that IS the punt: the differentiating compression is pushed to 'later'.
- 短板③ fold performance is punted to '若 PoC 实测仍是瓶颈, 升格 B 的向量化折叠算子'. The escape hatch for the named weakness is 'do方案 B later'. Phase 0 gates fold correctness but NOT fold latency under many-active-traces — the precise condition the short-coming describes. The hard case is moved to a future phase guarded by 'if it turns out slow'.
- '内核改动接近零 / 信创回炉最友好' is a punt by misdirection. By steering to 'reuse RM_DISKANN', the design pushes the new-RM kernel-fixed-table cost (which BM25 proves is unavoidable for a non-vector engine) out of view. It is not that kernel change is small — it is that the design declined to count it. The信创 re-evaluation trigger is presented as Phase-3-B-only when it actually fires in Phase 1.
- The bloat root-cause depends on the heap-bypass executor model that doesn't exist yet (CustomScan/column-supply, self-flagged 未确认). If the fallback 'heap holds placeholder + 回堆表' is taken and the heap ends up holding the folded wide row, ASTORE dead-tuple bloat returns. The design keeps both the bloat-is-root-caused claim AND the回堆表 fallback without reconciling that the fallback can reintroduce the bloat. The reconciliation is silently deferred to PoC.
- Snapshot/MVCC consistency of merge-on-read reads is simply not mentioned — a punt by omission of a correctness requirement, not just an optimization.
- Multi-tenant resource isolation in the shared, global, fixed-size vector buffer pool (keyed on g_instance.diskann_cxt singleton, documented 'read-only buffer') is punted. A read-write LSM with memtable flush + compaction does not fit a read-only global buffer; the design says 'reuse vector_buffers' without addressing that the pool is read-only and singleton-global, so traceseg + DiskANN + multiple tenants contend on one fixed pool.

**缺口:**
- No quantitative read-path budget. The design asserts 'sub-second cold search' and 'O(n log k) not linear' but never bounds k (segments per query), never models deletion-vector scan cost, never states active-trace segment fan-out. Phase 0 P0-4 only tests fold CORRECTNESS, not fold PERFORMANCE — the performance of the thing being root-caused (短板③) is never gated by a PoC.
- No concurrency/locking story for the LSM. memtable writes + background compaction + foreground reads on shared segments: what is the segment lifecycle lock model? disk_container is built for build-once-read-many (DiskANN/BM25); concurrent flush+compact+read is unproven here and unaddressed.
- amgetbitmap is waved off ('BM25/DiskANN 都没实现, 不强求') but trace queries are explicitly multi-predicate (tenant AND time AND status AND text). Without bitmap-AND across predicates, the AM must internally evaluate all predicates per segment — fine, but then the cost model for multi-predicate selectivity and the interaction with the planner (which will see a single index with unknown selectivity) is unspecified. costestimate for a fold+merge AM is non-trivial and gets 0 PoC attention.
- Snapshot/MVCC visibility across the merge. Level 0 heap gives MVCC for free. The AM's folded current-state must respect the querying transaction's snapshot (don't show events from uncommitted/future writers, show consistent state during long scans). The design never says how segment+memtable+upgrade-map reads are made snapshot-consistent. This is a correctness gap, not just performance.
- FST term dictionary, FSST, ALP, zero-copy compaction are all '二期/自研, 本仓无现成'. The compression ratios that justify '压缩 XOR 可检索 根治' (citing SmithDB's 88.8MiB→3.8KiB) come from FST which is Phase 2. Phase 1 ships DiskHashTable + plain encoding for strings/floats — so the headline compression win is deferred while the claim is stated in present tense.
