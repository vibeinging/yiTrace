# 附录 L：内核边界红队复审（sound-with-fixes —— 确认不需改内核, 但抓到 USTORE×GIN 建表bug）

**裁决:** sound-with-fixes

**最致命一击:** The core verdict ('v1 needs no kernel C change') is CORRECT and survives adversarial review — the Table AM closed-enum proof is solid (NUM_TABLE_AM=2 in tupdesc.h:31; hardcoded ternary in tableam.h; no CREATE ACCESS METHOD grammar), and every SQL primitive v1 relies on (MERGE WHEN MATCHED WHERE, @~@ OID 5048/5049, bm25_score OID 4527, INTERVAL partition, partitioned bm25/diskann indexes) is verified present in the kernel. BUT all three reports declared the schema buildable without ever running its DDL, and they missed a flat self-contradiction that breaks the headline retrieval path on day one: span_current is declared STORAGE_TYPE = USTORE (schema line 105) while carrying USING gin (jsonb_path_ops) and USING bm25 indexes — and indexcmds.cpp:783-789 hard-rejects EVERY non-ubtree index AM on a USTORE table ('%s index is not supported for ustore'), with the literal comment '/* Forbidden to create gin index on ustore table. */'. So JSONB-GIN and Chinese-BM25 on the fold table fail to CREATE. The fix is trivial and confirms the no-kernel thesis (change USTORE→ASTORE; ASTORE accepts gin+bm25 per parse_utilcmd.cpp:4300-4315 and still does in-place MERGE UPDATE), but it exposes the real weakness of this deliverable: the 'sound' rests on static source-reading, and a one-token storage-engine choice in the schema would have produced a broken build that none of the three verification passes caught. Verdict: no kernel change needed — but the schema as written does not run, and the merge-on-read performance claim is asserted, never measured. Ship only after (1) the USTORE→ASTORE fix and (2) an actual EXPLAIN/benchmark of the fold at target scale."

**缺口:**
- No EXPLAIN/row-count/benchmark anywhere proves the merge-on-read fold is fast enough at 10^8 span/day single-node. The entire 'no kernel operator needed' rests on an unmeasured assertion. This is the one load-bearing gap that could, if it fails on PoC, retroactively flip the verdict toward needing kernel work (and into the closed-Table-AM trap). It must be measured, not asserted.
- The reports never ran the DDL. The USTORE×GIN/BM25 hard reject (indexcmds.cpp:783-789) proves static source-reading missed a real build-time failure. The poc/verify_opengauss_yitrace.sql exists but there is no evidence it was executed against a live yiTrace build; until it is, every [验] item is 'probably fine,' not 'confirmed.'
- BM25 + non-@~@ scalar predicate interaction (post-filter vs predicate-pushdown, under-fill behavior, per-tenant index strategy) is unverified on the target. It's inside the AM's supported range but the recall/latency characteristics at high scalar selectivity are unknown — the same hole J left open for DiskANN, never closed.
- Whether MERGE's WHEN MATCHED ... WHERE (the fold_version monotonicity guard, schema [验] #1) works in the default dbcompatibility mode or requires B/Dolphin mode is unconfirmed; the grammar node exists (gram.y:23092 opt_merge_where_condition) but mode-gating is untested.
- No verification that ASTORE + RANGE+INTERVAL partition + LOCAL partial indexes (the partial WHERE on span_events ix_evt_parent and the partial indexes on span_current) all coexist; GLOBAL forbids partial/expression indexes and partition-table partial-LOCAL index syntax is version-sensitive.

**风险:**
- **[HIGH] SELF-CONTRADICTION IN v1 SCHEMA (all 3 reports + appendix-J/H missed it): span_current is declared `WITH (ORIENTATION = ROW, STORAGE_TYPE = USTORE)` (schema line 105), yet the SAME table carries `ix_cur_attrs ... USING gin (attrs jsonb_path_ops)` (line 116) and `ix_cur_input_bm25 ... USING bm25 (input_text)` (line 266). The kernel HARD-rejects this at execution time: /Users/Four/JobProjects/yitrace/openGauss-vector-main/src/gausskernel/optimizer/commands/indexcmds.cpp:783-789 — `if (rel->rd_tam_ops == TableAmUstore) { ... if (strcmp(accessMethod,"ubtree")!=0) elog(ERROR, "%s index is not supported for ustore"); }`. The comment at line 782 is literally `/* Forbidden to create gin index on ustore table. */`. So the two HEADLINE retrieval indexes (JSONB-GIN and Chinese BM25) on the fold table FAIL TO BUILD as written. (The 9 plain btree indexes survive — parse_utilcmd.cpp:4229 auto-rewrites a USING-less CREATE INDEX on a USTORE table to ubtree — but GIN and BM25 specify USING explicitly and hit the reject.) This is a day-one DDL failure, not a runtime edge case.**
  - 缓解: This does NOT require kernel change — it is a one-token DDL fix: declare span_current as ASTORE (drop `STORAGE_TYPE = USTORE`, or set it to the default heap). Verified at parse_utilcmd.cpp:4300-4315 that a non-ColStore, non-USTORE row store (ASTORE heap) accepts gin AND bm25 (both in the row-store whitelist). ASTORE supports in-place UPDATE via simple_heap_update, so the MERGE fold UPDATE branch still works; the schema's stated reason for USTORE ('被折叠原地更新→UStore适用') is a bloat-avoidance preference, not a correctness requirement. Net: the 'no kernel change' verdict survives, but the reports' source-verification was incomplete and would have shipped a broken schema. Add this to the [验] list as a CONFIRMED-BROKEN-must-fix, not an open question.
- **[MEDIUM] Merge-on-read performance gap (the user's Q1 + Q4): the fold runs as an application-driven `MERGE INTO ... USING (GROUP BY span_id ...)` micro-batch with `SET query_dop=1` (serialized, no parallel aggregate, schema line 128) plus a non-inlinable custom ordered aggregate `tv_jsonb_deep_merge_agg` and array_agg(ORDER BY) in the target list. At the stated <1e8 span/day single-node scale the hot partition is ~10^8 rows/day; serialized fold + plpgsql deep-merge of large JSONB + cross-midnight partition sort can make fold latency and live-query cost materially worse than a kernel merge-on-read operator. appendix-J already flagged this (no EXPLAIN/benchmark exists) and the schema itself defers large traces to 'application-layer DFS'. If PoC shows fold can't keep up, the temptation is a kernel operator — which lands back on the closed Table AM wall.**
  - 缓解: Stays app-layer-fixable for v1 scale: (a) keep the per-(tenant,trace) predicate so fold touches a working set, not the whole partition; (b) cap attrs_patch size and route large JSONB to CAS so deep-merge stays cheap; (c) widen micro-batch interval and lean on the span_live() set-returning function for freshness; (d) push only large-trace folds to the application O(n) path. None of these touch the kernel. Kernel operator only becomes justified if PoC proves a kernel-level bottleneck AND all app-layer knobs are exhausted (matches appendix-H's 'defer self-research until PoC fails' gating). Mandatory: actually run the EXPLAIN/benchmark — it is the single missing piece of evidence behind the whole 'no kernel needed' claim.
- **[MEDIUM] BM25/DiskANN filtered usage-range under-fill (Q3): the §9 BM25 query mixes a `@~@` predicate with NON-bm25 scalar predicates (`tenant_id=:t AND start_time>=now()-interval '30 days' AND input_text @~@ ...`). bm25_score()+ORDER BY forces the BM25 index-scan path (verified: bm25_parse.h:254-257 and inverted_index.md:76 — score is only valid under the @~@ index scan), so the tenant/time predicates are applied as a POST-filter on the BM25 top-K stream — a small tenant buried in a large corpus can under-fill (same failure class as the filtered-ANN under-fill J already flagged for DiskANN). Also inverted_index.md:60 restricts multi-column matching to AND-combined @~@/@-@ conditions only (OR must be embedded in the query string or done via UNION).**
  - 缓解: Usage-range, not kernel: this falls inside what the existing BM25/DiskANN index AMs already support (partitioned bm25 and diskann index code paths both verified present: bm25.cpp:1102, diskann_scan.cpp:236), so no new kernel code. Mitigate at query layer: per-tenant LOCAL bm25 indexes or a tenant-partitioned retrieval table so the scalar filter is satisfied by partition pruning before scoring; and the same brute-force-below-selectivity-threshold fallback the schema already adopts for DiskANN. Add a recall@k vs selectivity benchmark to the [验] list.
- **[LOW] Verification was static-only and the reports over-claimed completeness. The schema self-lists 9 [验] items still unconfirmed on a running instance (MERGE WHEN MATCHED WHERE under which dbcompatibility mode, exact bm25 WITH params, DiskANN inplace-filter column order, LOCAL+partial-index coexistence on the partitioned event table, ASTORE+INTERVAL combo). The reports asserted these 'don't change the kernel verdict' — true — but the USTORE×GIN/BM25 break above shows that 'static source check' missed a hard execution-time reject. Other [验] items could hide similar build-time rejects (e.g., partial-index LOCAL on span_events line 71-72 — GLOBAL forbids partial/expression indexes per J risk).**
  - 缓解: Run the poc/verify_opengauss_yitrace.sql on the target build and explicitly attempt each DDL (especially: the USTORE→ASTORE fix, every LOCAL partial index on span_events, bm25/gin/diskann CREATE INDEX, and a MERGE with WHEN MATCHED ... WHERE). None of the likely failures imply kernel changes — they imply DDL adjustments — but they must be empirically cleared before 'no kernel change' is declared final rather than 'static-analysis likely'.
