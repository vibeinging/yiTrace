# 附录 F：第二轮红队复审

> 注：原计划 2 路复审，critique:2 因 socket 断开失败，此为幸存的 critique:1（裁决 sound-with-fixes）。

## 总裁决: sound-with-fixes

**最可能失败的一击:** 最可能致命的一击不是技术，是市场顺序与准入前置的错配：这个产品最强的需求信号来自『客户主动要求部署』的金融政企类客户，但计划把这类客户的三个硬性准入条件——HA主备(Phase 3)、信创全栈认证(Phase 2才起跑、周期12-24月)、数据脱敏——全部排在产品生命周期的中后段。web search 证实 82% 金融政企以信创认证为采购准入前置。叠加结果：你能在 ~10-11 个月做出『可售完整平台』，但它对最想买、最该买、又在主动要的金融政企客户是『不能投标的标』(缺HA)、『进不了采购名单的供应商』(缺信创)。于是真正的首个付费客户被迫只能是不要求HA/信创的互联网或中小企业——但那批客户恰恰是 Phoenix(单容器+零gate+全开源)和 Langfuse(免费MIT)免费打得最凶、最不愿为私有化和中文付溢价的人群。两头落空：高价值客户够不着(准入没就绪)，够得着的客户不付费(免费对手卡死)。化解唯一路径=在 Phase 0 锁定首客画像，若押金融政企，则 HA+信创必须从『后置/并行』升级为决定总时间线的关键路径前置，并诚实把金融政企首单时间重报为 17-29 个月而非 10-11 个月；若押互联网中小，则必须重新论证在 Phoenix/Langfuse 免费夹击下凭中文+语义召回+飞轮能收到多少溢价——这个溢价假设目前全靠未验证的⑥四级验证撑着，是整个商业模式尚未被任何数据支撑的单点。"

**能力缺口:**
- 【缺口：首个付费客户的画像/采购前置条件未在 Phase 0 锁定】整个时间线、HA 排期、信创排期、SKU 定价都依赖『首客是谁』，但计划从头到尾没有明确首个付费客户是金融政企还是互联网/中小企业。这是最大的规划缺口——金融政企首客会把 HA+信创+脱敏+昇腾全部变成 v1 准入前置，时间线和预算结构完全不同。建议 Phase 0 第一件事就是锁定首客画像并据此分叉两套路线图。
- 【缺口：信创认证作为关键路径未进总时间线甘特图】计划把信创『单列预算、Phase 2 并行启动』，但没把它画进决定首单时间的关键路径。12-24 月认证周期 + 82% 金融政企以认证为准入 = 信创完成时间很可能是金融政企首单的瓶颈，比平台就绪更晚。总时间线缺一条『信创关键路径线』。
- 【缺口：团队组建期/招聘周期完全未计入时间线】计划承认要从 3-4 人扩到 13-16 人（缺口 10-12 人），但路线图默认 Phase 1 起就有 8-10 人高效并行。从 0 招齐资深 React×3-4 + Rust 服务×2-3 + 算法人，在中国市场至少 3-6 个月招聘+磨合期，且新团队前 2 个月产出打折。时间线缺一个显式的组建期，导致总工期被低估 2-3 个月。
- 【缺口：备份/恢复的可信度未被验证为产品级】计划给了 RPO≤5min/RTO 30-90min 和 yitrace-restore 单命令，但私有化最容易出事的是『备份了但恢复不回来』。缺一个『定期自动恢复演练/备份校验』机制，以及向量索引+倒排索引在恢复后的重建时间是否被算进 RTO（计划 RTO 写的是『拉快照+回放WAL+重建/加载索引』，但 XL 档几十TB 的索引重建可能远超 90 分钟）。RTO 估算可能只算了数据没算索引重建。
- 【缺口：openGauss 作为内嵌元数据库的运维/版本/打包负担被当成零成本】计划反复说『元数据库用 openGauss = 信创零成本』，但 openGauss 作为内嵌元数据库要打进单二进制/Compose、做 schema 迁移框架、备份恢复、与平台生命周期绑定，这部分工程量不是零；且 openGauss 本身的资源占用（相比 SQLite/轻量PG）在『单机轻量开箱』理念下值得权衡。计划没算 openGauss 内嵌化的工程成本。
- 【缺口：单机故障的业务连续性（非HA客户）兜底缺失】对买『形态1单机+冷备』的中小客户，主机硬件故障=服务中断 30-90 分钟+。计划没说这期间摄入的 trace 是否丢失（客户 agent 还在跑、还在发 trace，摄入端宕机期间这些 trace 去哪了）。缺一个『摄入端断点续传/客户侧 SDK 缓冲重试』的端到端可靠性设计，否则单机宕机=丢数据，对可观测产品是硬伤。

**风险清单:**
- **[CRITICAL] 【①HA/采购失分 — 最严重的现实落地风险】计划把『主备热备』整体后置到 Phase 3（启动后 13-16 个月才有），v1 只给『单机+增量备份，RTO 30-90 分钟』。但金融政企的招投标技术评分表里，HA/RPO/RTO 通常是『一票否决/废标项』而非加分项——不是『失分』而是『没有资格投标』。计划自己也承认『金融政企会在合同里写 RPO/RTO』，却把满足它的能力放到产品生命周期的最后阶段。结果：你最想卖、客单价最高、又是『客户主动在要』的那一类客户（金融政企），恰恰在 Phase 2 首客交付时拿不到能中标的 SKU。更糟的是 web search 证实『82% 金融政企客户明确要求供应商具备信创认证』——HA 缺失和信创缺失会在同一批客户身上叠加废标。这不是工时问题，是『首个付费客户可能根本不能是金融政企』的市场顺序问题，会直接打乱『dogfood→首客→商业化』的现金流假设。**
  - 缓解: (a) 把『主备只读热备』从 Phase 3 上提到 Phase 2 末尾甚至作为首客交付的硬前置——计划自己论证过『不可变 Lance fragment + WAL 流让复制退化成文件级增量同步，比 PG 流复制还简单』，那么它就不该是最贵最晚的功能，应是中等成本早交付。(b) 在 Phase 0 做一次客户分层：明确首个付费客户是不是金融政企；若是，HA+信创+脱敏必须进首客 MVP，时间线相应顺延 3-6 个月并在预算里明示；若首客是互联网/中小企业则可按原计划。(c) 把 keepalived/VIP+fencing 的脑裂处理列为独立工程项并单独标注工时（计划只说『复用成熟件』，但自动故障切换+fencing 在私有化无人值守环境的测试成本被严重低估，至少 4-6 人周而非顺带）。
- **[HIGH] 【④竞争正面战场 — 单机简单性护城河被高估】计划的核心差异化之一是『单机单二进制 vs 对手重栈自托管』，并反复用 Langfuse v3 需要 ClickHouse+PG+Redis+S3 四件套来论证。但 web search 证实 Arize Phoenix 自托管就是『单 docker 容器 + 可选 Postgres，SQLite 起步』，部署简单性与你们相当甚至更轻；Phoenix 还零 feature-gate（评估器内建、全开源、英文 RAG 指标已有）。也就是说『单机开箱即用 + 企业治理免费』这条线，Phoenix 已经占住了，你们对它没有部署简单性优势。真正只对 Langfuse 成立的差异化（重栈）被计划当成了对整个对手阵营成立。剩下能站住的差异化收窄到：中文分词/中文RAG指标/中文judge、语义trace召回、信创/openGauss、国内IM、飞轮——这些是真的，但『单机简单』不是。计划的差异化表把『单机开箱』列为对所有对手的结构性优势，是站不住的，会误导定价和销售话术。**
  - 缓解: (a) 把竞争叙事从『单机 vs 重栈』收敛到 Phoenix 结构上做不到/不愿做的四点：原生中文全链路、语义trace召回+飞轮、信创零成本（openGauss自有内核）、国内生态（钉钉/企微/飞书/中文judge模板）。这四点 Phoenix 确实没有，且短期没动力做。(b) 对 Phoenix 单独做一次功能/性能 bake-off，找出它在中文场景、规模化（Phoenix 大数据量 UI/查询是否撑得住几十TB）、企业治理深度（Phoenix 的 RBAC/审计/SSO 成熟度）上的真实短板，把差异化建立在实测而非『它是英文SaaS』的笼统判断上。(c) 定价别押在『我们部署更简单』，押在『中文+信创+飞轮』的不可替代性上。
- **[HIGH] 【②③平台层工时仍可能被低估，现实化系数 ×1.5-1.8 偏乐观】计划已经做了一次诚实修正（名义90人周→现实化×1.6-1.8），这是进步。但仍有三处系统性低估：(1) 评估框架的『中文 RAG 指标 4 人周』只算了 prompt 重写，没算指标质量验证闭环——RAGAS 类指标要在中文语料上调到可信，需要构造中文 ground-truth 评测集、与人工标注对齐、反复调 judge prompt，这是算法迭代而非编码，真实成本是数倍。(2) 多模态/大payload 在 UI 侧的边界处理（超长上下文 trace 渲染、几十KB-1MB 单span payload 的虚拟化+懒加载+diff）被计划塞进『span详情懒加载2人周』，实际是 trace 浏览器最容易崩、最吃打磨工时的地方。(3) 『有效 8-10 人并行』假设了团队从 Phase 1 就满编且无招聘/磨合损耗——13-16人团队从0组建（计划承认第一轮只有3-4人内核），招齐3-4个资深React+2-3个Rust服务+算法人，在中国市场至少3-6个月招聘期，且新团队前2个月产出打5折。计划的时间线把『团队已满编高效并行』当默认起点。**
  - 缓解: (a) 中文 RAG 指标拆成『v1 给可用的中文judge模板（先上线）』+『v1.5 给经人工标注对齐验证过的可信指标』两步，别承诺 v1 就是高质量；单列一个『中文评测集构造+标注对齐』算法工时（建议 6-8 人周）。(b) 把『大payload/超长trace 的 UI 健壮性』提为 trace 浏览器的独立风险项，单列 4-6 人周打磨+边界测试，并在 POC 用客户真实最大 payload 压测。(c) 在路线图最前面加一个显式的『团队组建期』（3-6 个月，与 Phase 1 部分重叠），并把 Phase 1 前两个月的有效并行人数按 0.5-0.6 折算；总时间线据此再加 2-3 个月缓冲。诚实的总时间线应是『启动→可售完整平台首客交付 ~13-15 个月（含组建期）』而非 ~10-11 个月。
- **[HIGH] 【③框架无关摄入持续追随成本被当作可控常态，实为长期人力黑洞 + 交付完整度风险】计划承认 OTel GenAI 仍 Development、v1.37 已 breaking、需『专人季度跟踪』。但把它定为『专人季度跟踪』本身就是低估：OpenInference + OpenLLMetry + LangSmith-compat + OTel semconv 四套方言，每套都随上游框架（LangChain/LangGraph/AutoGen/Dify 各自版本）漂移，且计划只认证 4 个框架『钉版+回归用例』，其余 best-effort。私有化场景下，客户用的框架版本是客户决定的、且常滞后或超前于你认证的版本——『客户升级 LangChain 后 trace 字段丢失』会变成现场 P1 故障和交付验收阻塞，而不是『best-effort 兼容』可以打发的。Dify 更是计划自己标注的『重点缺口、无一线 instrumentor、节点语义需自研』，却仍排进 v1 四框架认证。摄入完整度是『指 endpoint 迁过来』这个核心卖点的根，一旦首客的某个框架版本掉字段，整个迁移卖点的信任崩塌。**
  - 缓解: (a) 把摄入团队从 2 人提到 2.5-3 人，并把『版本追随』从季度兼职改为常态化的持续集成防线：对认证的每个框架建『多版本矩阵 CI』（不只钉单版，而是覆盖客户可能用的版本区间），上游新版自动跑回归并告警。(b) 对 Dify 单独立项评估——若 v1 节点语义自研成本过高，先降级为 v2 或『best-effort+raw_attrs兜底』，别让它拖累 v1 四框架的交付承诺。(c) 在合同/SOW 层面把『迁移兼容』的承诺边界写死：只承诺已认证框架的已认证版本区间 100% 保真，其余 raw_attrs 无损保留但字段映射 best-effort——把技术现实变成合同条款，避免现场背锅。(d) raw_attrs 无损兜底是好设计，但要确保 UI span 详情面板对『未归一的 raw_attrs』也能可读展示，否则掉字段时客户看到的是空白而非降级展示。
- **[HIGH] 【⑥信创成本/周期被『隔离到平台外』但仍是金融政企的准入前置，且与时间线脱钩】计划把信创认证『单列预算 12-24 月七位数，非开发工时』并在 Phase 2 并行启动——这比第一轮有进步。但有两个未闭合：(1) web search 证实 82% 金融政企明确要求供应商先具备信创认证才能进采购，意味着信创不是『拿到客户后并行做』，而是『没有它根本进不了这批客户的采购名单』——12-24 月的认证周期若在 Phase 2（启动后 ~5 个月）才起跑，则金融政企首单最早要等到启动后 17-29 个月，远晚于计划宣称的『~10-11 个月可售』。(2) 计划用『元数据库用 openGauss = 信创零成本』来缓解，但信创认证是对整个交付物（CPU/麒麟OS/数据库/中间件/推理栈昇腾全栈适配+等保三级+国密）的认证，openGauss 只覆盖其中数据库一项；Lance/DataFusion/Rust 运行时/ONNX推理/Qwen3模型在国产CPU+麒麟上的适配与等保，计划完全没算。昇腾/CANN 推理路径还被放到 v2。**
  - 缓解: (a) 把信创认证的起跑点从 Phase 2 提到 Phase 0/Phase 1（与 dogfood 并行），因为它的周期(12-24月)是总时间线的关键路径——如果金融政企是目标客户，认证完成时间决定首单时间，必须最早启动。(b) 做一次全栈信创适配清单：不止 openGauss，要逐项确认 Lance/DataFusion/Rust toolchain/ONNX Runtime/Qwen3-int8 在鲲鹏/飞腾+麒麟/UOS 上能否编译运行、性能是否可接受、是否需要昇腾适配；其中『ONNX/embedding 在国产CPU上的吞吐』计划已标为待实测，但它同时是信创适配项，要合并评估。(c) 把昇腾/CANN 推理路径从 v2 上提——若目标客户配国产卡且无 GPU，纯 CPU Qwen3-int8 在国产CPU上的吞吐可能撑不住 1亿span/天的 embedding 负载（计划自己承认是重负载），昇腾就不是 v2 锦上添花而是 v1 准入。(d) 在商业计划里明确：金融政企线的真实可售时间 = max(平台就绪~13个月, 信创认证完成~认证起跑+18月)，别用互联网客户的时间线对金融政企报价承诺。
- **[MEDIUM] 【⑤语义召回付费意愿 + 气隙 embedding 落地的双重不确定，且二者耦合放大风险】计划的⑥验证章节很诚实（埋点→A/B→加价档→agent回路ROI 四级），这是亮点。但有两个耦合风险未点破：(1) 语义召回的差异化价值高度依赖气隙 embedding 在客户国产硬件上的真实吞吐，而计划自己承认『0.6B-int8 在国产CPU上的 sentences/sec 无可信公开基准，必须实测』——如果实测吞吐撑不住，要么降采样（召回质量下降，差异化减弱），要么上昇腾/GPU（客户成本上升+信创适配，溢价被抵消）。即『付费意愿验证』和『embedding 能否跑起来』不是两个独立问题：embedding 跑不动→召回质量打折→付费意愿验证用的是降级版功能→得出的市场信号失真。(2) 『模型升级即全量重 embed』在 XL 档（9亿向量）是数天级的离线重建作业，私有化现场无人值守环境下这是高风险运维动作，计划只写了一句『写进运维手册』。**
  - 缓解: (a) 把『目标国产硬件上 Qwen3-0.6B-int8 吞吐实测』提为 Phase 0/Phase 1 的硬阻塞项（不是 POC 才做），因为它的结果会反向决定语义召回差异化的可信度和 embedding 部署形态（CPU/昇腾/GPU），进而决定整个差异化叙事能不能成立。先拿一台目标国产CPU机器实测，再决定语义召回投入力度。(b) 验证顺序上，第1级『埋点测使用率』应该用『不依赖气隙 embedding 的版本』——即先用 root span 的关键词+已有 EMBEDDING span 截获向量做最低成本召回，验证行为信号；只有行为信号为正才投入气隙 embedding 调优。把『验证市场』和『跑通 embedding』解耦，避免为验证花大成本。(c) 给『全量重 embed』设计成可断点续跑+影子索引切换（新模型在影子索引上重建完成后原子切换，不停服），并纳入 HA/运维工程工时，别留给运维手册。
- **[MEDIUM] 【⑦dogfood 自家产品与外卖第三方客户的需求分叉，是路线图的隐性串行依赖】计划的路线图把 Phase 1 完全押在 dogfood 自家 AgenticData（轨A富语义+深集成），Phase 2 才做轨B（LangSmith-compat+第三方框架）。但风险在于：自家 agent 的 trace 形态、字段、规模、UI 需求会深刻塑造 Phase 1 建出来的引擎接口契约、内部规范模型、trace 浏览器交互——而这些是为『我们控制的 vex.* 富语义』优化的。等到 Phase 2 接第三方框架（字段稀疏、无 vex.*、方言杂、版本漂移）时，Phase 1 为自家场景做的设计假设（如『trace 树由摄入时 dotted_order 直接建』『EMBEDDING span 向量可截获』『thought/decision 喂语义召回』）很多在轨B不成立，可能触发返工。即 dogfood 优先反而可能让产品过度拟合自家 agent，外卖客户交付时发现规范模型/UI/召回供给侧都要为『贫信息第三方 trace』再适配一轮。计划假设两轨共用一套底座无缝，但富语义轨和贫语义轨对同一套召回/UI/评估的要求差异可能比预想大。**
