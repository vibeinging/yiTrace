# yiTrace 自研引擎 Phase 0 实测清单（加固版）

## 这份清单要干嘛

在投 40-60 人月自研 trace 引擎之前，先用最小最便宜的实验，把最可能让整件事翻车的假设挨个证伪。**先花小钱、再决定花不花大钱。** 每个实验只回答一个问题：这个假设是真是假；绿了说明可以继续投，红了说明要么止损、要么得把额外代价补进账本再重算。

**总成本（加固后，比原草稿上修）**：

- 真正的放行门禁（A2/A1/B/C/D 五项）：约 **27-43 人周**（原草稿 19-31，红队上修后）。
- 完整 Phase 0 含 E：约 **47-78 人周**，且 E 方差极大、上界不封顶（见下文"无底洞标注"）。
- **重要前提**：以上是"满编、人已到位"的口径。决策摘要自己承认这 10-12 人多数还不存在、还在招人。若工程人未到位，B/C/A1/D/E 的人周不变，但**自然周期 +3-6 个月，且新人前 2 个月产出按 5 折计**。给决策层的时间线必须给两个版本（见末尾）。

**哪几个绿了才放行**：必须 **A2、A1、B、C 四个"能一票否决"的门全绿**，且 **D 绿（或红灯已有明确低成本修复并把工时回填账本）**，才放行整个 40-60 人月自研。**E 不是放行门**（它在放行之后做），但 E 的"崩溃原子性"子项是上线前金融政企准入的硬门——不绿不能交付。

**与原草稿相比，本版做的主要修正**（直接回应红队）：
1. A2 拆成纯需求侧的 A2a（零代码、最便宜的一票否决）和可选的 A2b 糙 demo（挂到 A1 之后），并把判据从"客户嘴上说要"换成"客户的钱/合同结构暴露真实偏好"。
2. A1 拆成 A1-基建（第 0 周即起跑、不依赖 B/C）和 A1-自研侧接入（依赖 B/C）。带过滤召回的测试点从被精挑的 1%-10% 甜区，下沉到真正会塌的 0.1%-1% 高选择性区间，并锁死 ef/过采样预算。
3. B 的范围从"桥 cppjieba"扩到"复现整条分词流水线（cppjieba + snowball 词干 + 用户词典/停用词/同义词的实际内容）"，人周上修到 6-9；增加纯 C++/jieba-rs 零 FFI 前置证伪；判据按词频加权，不准用全语料平均稀释系统性漂移。
4. C 把"进图过滤能不能把召回拉回来"这个真问题拉进 Phase 0 本体（最小原型），不再推到"后续 spike"；空过滤召回的金标准从 DuckDB 出厂版（本身是半成品）换成暴力精确 KNN；人周上修到 8-12。
5. E 明确标为"放行之后的实现期第一阶段"，不算进放行门禁的总账；放行前只保留最小的崩溃原子性子项作信号；崩溃测试从"随机 kill 数百次"改成"对每条已知窄临界点确定性插桩 kill"；并发协议先用模型检查纸面证伪再写代码。
6. 把 D 的依赖从 A1 改成 C；E 的依赖补上 D。把 A2a + B 分词一致性设成第 3 周的"早期止损门"，不等到第 9 周大评审。
7. 全程标注哪些工时不确定、哪些是无底洞、哪些红灯"嘴上能砍实际砍不掉"。

## 总表

下面每行一个实验：要排什么风险 / 怎么最省做 / 绿灯判据（已堵掉假绿口子）/ 红灯怎么办 / 工时 / 依赖 / 能否一票否决。

### A2a — 付费意愿（承重墙，纯需求侧，零代码）

- **要排的风险**：整条路线的承重墙。如果客户其实只为"trace 能查、中文召回好、延迟低、数据不出境"付费——这些任何成熟后端（自托管 ClickHouse-Langfuse 加一层中文召回）都能给——那么"自建 BM25/graph_index 当旗舰 IP"这个产品命题当场塌掉。这是 #1 风险，也是最便宜的一票否决。市场不在工程。
- **怎么最省做**（采纳红队"更省替代"）：
  1. **先做桌面挖掘（1 周内、零客户档期）**：直接翻已成交/在谈金融政企客户的真实标书和技术评分表，看"索引自研/自主可控"到底是写成硬性资格条款、技术评分加分项，还是只写"检索效果/性能/数据不出境"。**关键**：条款必须具体到"倒排/向量索引算法自研"这种颗粒度——泛泛的"自主可控/国产化"一律不算证据，因为开源自托管（源码在手、境内部署、数据不出境）同样能勾这个框。
  2. **再做 3-6 家深访的"报价剥离对话"**：对同一客户口头给两个功能等价方案——方案 X（开源自托管后端 + 中文召回，明说召回/延迟/源码境内/数据不出境全等价，便宜 30-40%）vs 方案 Y（自研索引旗舰，更贵）。只问："这俩功能一样、X 还便宜，你为什么仍选 Y？能为这个差异多付多少？加价理由是什么？"
  3. **要一份反向证据**：找 1 家已经在用买侧方案的同类客户问，而不只问自己的存量客户（存量客户对你有人情和路径依赖，最容易给"你们自研我支持"的礼貌性伪绿）。
- **绿灯判据（堵掉"客户嘴上说要"的假绿）**：在"剥离共同项"的对话里，多数目标客户（金融政企为主）**在被明确告知"换开源后端、召回延迟一模一样、源码同样给你、同样境内部署、数据同样不出境"之后，仍坚持要自研索引，并能把加价理由落到"索引算法是你自研"本身**（而不是召回/延迟/合规/主体境内——后四者方案 X 已具备）；且能从**至少 1 份真实标书里找到颗粒度到"索引算法自研"的硬条款或评分项**。一句话：判据看的是"客户在功能不可区分时是否仍为自研身份多掏钱"，不是"客户说不说想要自主可控"。
- **防自欺硬约束（红队第四面）**：
  - 访谈结论必须由**不拥有自研路线的人**（外部顾问或商务，非引擎负责人）记录和判定。
  - "红灯触发什么动作"必须**在跑 A2a 之前由决策层书面签字承诺**，否则红灯到来时会被重新解读成"客户其实支持继续做"。memory 里已写明真实目标函数可能是"自己做是目的本身"——这条约束就是防止结论被事后改写。
- **红灯怎么办（诚实标注：这是嘴上能砍、实际最难砍的一处）**：红灯 = 直接否掉"自研引擎当旗舰"的前提。立刻停掉 B/C/A1/D/E 全部工程投入，转买侧，把省下的人月投平台层（占交付 60-70%、且真正买不到）。**但这个"立刻全停"只有在两件事都成立时才真能发生**：(a) A2a 真的先于一切工程跑完；(b) 公司层面没有把"自研引擎"写死进对外叙事/投资人承诺/招聘理由。所以才有上面的书面签字承诺约束。
- **工时**：2-4 人周（桌面挖掘 1 周 + 深访 1-3 人周）。**自然周 2-3 周**（桌面挖掘不受客户档期约束，深访受档期约束但已不在关键路径上卡死后续）。
- **依赖**：无。
- **一票否决**：是。最高优先级止损。

### A2b —（可选）糙 demo，挂到 A1 之后

- 如果客户一定要看 demo 才肯表态，等 A1 自研骨架出来再顺手做，**不要让 demo 卡在承重墙前面**。原草稿把"1 个技术做糙 demo"塞进 A2，等于偷偷给承重墙加了 BM25/jieba 的工程前置——本版剥离。A2b 不是门禁，不计入放行判定。

### A1-基建 — 评测脚手架（第 0 周即起跑，不依赖 B/C）

- **要排的风险**：A1 整体要证明"自研栈在客户真正看重的检索质量上确实赢得过强买侧"。但其中买侧基线、标注查询集、评测脚手架这些**完全不依赖自研侧**，原草稿把它们压在 B/C 之后白等 4-6 周。
- **怎么最省做**（采纳红队拆分）：第 0 周与 A2a/B/C 并行起跑。搭买侧（自托管 ClickHouse-Langfuse + jieba 预分词 FTS + usearch HNSW + 一层中文召回，这是成熟件，1.5-2 人周可起）+ 造**带相关性标注的中文查询集** + NDCG@10/MRR 评测脚手架 + 带过滤召回的"选择性谱 + 热数据态"测试夹具。**这份夹具同时复用为 B 的 top-k 一致性、C 的召回一致性的现成验收夹具，省一份重复工。**
- **早信号（采纳红队"更省替代"）**：买侧基线一搭好，先测"买侧 token 过滤式 FTS 的 NDCG@10 到底有多低"。如果买侧加 jieba 预分词后排序质量其实并不差，那"自研 BM25 要显著高出 ≥0.1"这个门会非常难达，A1 大概率偏红——这个信号在自研侧一行代码都没接之前就能拿到。
- **绿灯判据**：基建本身不是门禁，它的产物（标注集、夹具、买侧基线）齐备且可复算即可。**标注集硬要求**：来自客户真实日志采样（去敏后）、标注 **≥300 条**、覆盖口语化/长尾查询、**预注册查询集与标注规则后再跑分，禁止跑完挑着报**。
- **红灯怎么办**：买侧基线异常高 → 提前预警 A1 大概率红，提示决策层是否还要投 B 的倒排重写。
- **工时**：3-5 人周（买侧搭建调优 1.5-2 + 标注集构造 1.5-2.5 + 脚手架 0.5-1）。**标注集构造是算法迭代不是编码，容易被低估，单列。**
- **依赖**：无。
- **一票否决**：本身不是，是 A1 的前置。

### A1-自研侧接入 — 技术对标（依赖 B/C）

- **要排的风险**：排掉"造得出但赢不过买侧"。如果拿钢人化的买侧（不是裸 chDB 稻草人）来比，自研栈在中文检索相关性、带过滤召回率上并不显著更好，那么即使 A2a 说客户为自研付费，付费理由也不成立——"造得出"不等于"造得比买的好"。
- **怎么最省做**：把 B 的 BM25 评分、C 的带过滤搜索接进 A1-基建那套同一夹具。自研侧用 DataFusion + Vortex 段 + graph_index（FFI 只读搜索）+ BM25（FFI 评分）拼一个能跑查询的最小骨架，**不做 WAL/compaction/崩溃恢复（那是 E）**。比四件：①中文相关性质量；②带 service/time/status 过滤的向量召回；③折叠正确性 + 读放大；④点查/取一条 trace 的延迟。顺带实测 chDB/ClickStack 嵌入态实际暴露的 auth/RBAC/审计面（把"零安全面"从文档断言升到实测）。
  - **诚实标注（工时不确定项）**：自研侧不是"把 B/C 成品拼起来"。决策文档 §7 明说"无任何单一先例同时做 filter 下推 + fold 算子 + bm25_score ScalarUDF + RRF 融合 + 跨四源快照在一个计划里"，DataFusion TopK 下推还是 OPEN issue（#16358）没有标准 API，要自写 physical 节点 + 优化规则。这件事决策文档自己列为"最大不确定项"之一，工时方差大。
- **绿灯判据（堵掉三个假绿口子）**：
  1. **相关性**：在预注册的客户真实查询集（≥300 条标注）上，自研栈 BM25 排序质量 NDCG@10 **绝对值高出买侧 ≥0.1 且统计显著**。配套硬约束：**样本量/标注量达标**（≥300 条，否则"统计显著"在小集上无意义）；**对称调优或盲测**——自研侧和买侧用同等力度调参，禁止"给自研调满、买侧不调"人为拉开差距（否则证明的是"我方更会调自己的参"，不是"架构本质更优"）。
  2. **带过滤召回（核心，堵掉"挑甜区"假绿）**：必须画**整条选择性谱（0.1% / 1% / 5% / 10% / 50%）**，而不是只测被精挑的 1%-10%。真门禁是**0.1%-1% 高选择性区间召回不塌**——这才是 trace 单查询的常态（查某一条 trace 的某类 span）。同时：
     - **ef / 过采样预算锁死到一个上线可承受的固定值**，在该固定预算下同时测召回 AND QPS 退化曲线，**两条都守住才算绿**。不准靠偷偷把 ef 调大把召回拉回来——上线真实工况不可能为每个查询无限加 ef。
     - 必须在**热数据态**测一遍：边写边查、新段未建图走 flat 兜底、与已建图段混合，复现附录 Q 点名的"活段无全局图召回路径"风险。静态已建图语料测出来的召回对生产无效。
     - 过滤维度必须包含**高基数 trace_id/status 的 AND 组合**，不能只测 service/time 这种低基数列。
  3. **折叠正确性两侧都得对，读放大自研不显著更差。**
- **红灯怎么办（堵掉"无限再接一版"的止损泥潭）**：
  - 红灯先查"是不是骨架没接对（BM25 评分没真正下推、过滤没真进 search_layer）"。但**"接对"的客观标准必须在跑 A1 之前就写死**：例如火焰图里证明 bm25_score 评分发生在扫描层、过滤进了 search_layer 的 ef 导航而非结果裁剪。**"接对"的认定最多允许一次返工**；超过一次仍输，**强制进入与 A2 合并重审，不允许第三次"再接接看"**。这是防止止损被沉没成本反向绑架。
  - 接对了仍赢不过买侧 → 与 A2 合并重审路线，大概率翻回买侧，自研前提动摇。
  - 只是"带过滤召回塌" → 坐实 C 里的进图过滤（ACORN 式）改造是必做项（见 C，两处账本合并回填一次，不重复计）。
- **防止把工程结论冒充市场结论（红队点名的最危险伪绿）**：A1 跑出"自研检索质量更好"**不能当成 A2 的代理证据**。"技术上确实更好所以客户会买"是把工程结论冒充市场结论。付不付费由 A2a 单独判，A1 只判"技术上赢没赢"。
- **工时**：8-13 人周（上修自原 5-8。自研侧 DataFusion 自定义节点是最大不确定项；A1-基建已单列，不重复计）。
- **依赖**：B、C（FFI 桥接打通）；A1-基建可前置并行。
- **一票否决**：是。

### B — BM25 能否搬出内核

- **要排的风险**：排掉"BM25 这块 IP 搬不出 openGauss 内核、或搬出来后行为漂移"。真正差异化的 IP（block-max-WAND skip-list 倒排、min_should_match、稀疏 + BM25 融合）活在 buffer-coupled 的倒排代码里。两个具体雷：①分词桥进 Rust 后分出来的词和现有产品不一致——token 不同直接导致召回/打分漂移；②已知的"并行建索引 skip-pointer 指针 bug"（inverted_list.cpp:341-344，原团队自己没解决）。
- **范围修正（红队第四面，重要）**：现有中文分词流水线**不是 cppjieba 单独一个库**，而是 dict_jieba.cpp + tokenizer.cpp + dsnowball.cpp + token_pool.cpp 四件，且**词典焊死在 openGauss TS-template（OID 3884 内置模板）+ pg_ts_dict/pg_ts_content 目录里**，活在数据库内核里、不在某个可直接拷的文件里。客户 trace 语料大量是中英混合（service 名、属性 key、报错串），snowball 英文词干那一段不复现，逐 token 一致率根本到不了 99.5%。所以 B 的范围是"**复现整条 tokenizer 流水线**（cppjieba cut + dsnowball 词干 + token_pool + 当前生产词典/停用词/同义词的实际内容）"，不是只桥 cppjieba 主干。
- **怎么最省做**（采纳红队"更省替代"，零 FFI 先证伪）：
  1. **先做零 FFI 的分词对齐**：拿现有产品对一批真实客户语料 dump 出分词结果当 ground-truth；再用独立编译的 cppjieba + dsnowball（**纯 C++ 命令行，不进 Rust**，同版本同词典）和 jieba-rs（纯 Rust，无 FFI）各跑一遍做 A/B。如果连同语言的 C++ 复现都对不齐（因为词典没从内核目录里导全），FFI 桥进 Rust 是白费力气——这一步 2-3 天就能暴露"词典不可移植"这个真问题。如果 jieba-rs 都能逐 token 对齐到阈值，分词可移植性风险当场大幅下降，FFI 是否必要可重估。
  2. **前置小任务单列**：把"词典内容如何从 openGauss 目录里完整导出"单列成一个独立任务——这是"活在内核数据库里"的状态，header-only 桥接 cover 不了。
  3. 分词对齐通过后，再用 cxx/C ABI 把流水线桥进 Rust 跑通 cut_mix/cut_query/highlight + 用户词典加载。
  4. **倒排**：用 Rust 写最小倒排段（skip-pointer + block-max），跑通 bm25_score + min_should_match + AND/OR 布尔查询，在同一份语料同一组查询上和现内核比 top-k。
  5. **当场拍板那个并行建索引 bug**：修（净新增并发正确性研发）还是退回串行建索引（建索引吞吐退化），把代价写进预算。
- **绿灯判据（堵掉三个假绿口子）**：
  1. **分词一致性（不准用全语料平均稀释）**：除了整体逐 token 一致率 ≥99.5%，必须**单独看三类子集**——高频业务词、用户词典词、客户真实查询里出现过的词；**这三类 + 金融实体词/产品名必须 100% 一致**，任一子集系统性漂移即判红。测试语料**必须含中英混合 token 和已加载用户词典命中的 case**，不准用纯中文短句。对"归因到词典版本差异"的不一致 case，必须验证"统一版本后是否真能消除"——消不掉的一律判红。
  2. **倒排一致性（堵掉"静态最小段没碰到 bug 路径"假绿）**：同一份语料同一组查询（含 min_should_match 和布尔组合），Rust 重写倒排 top-k 文档集合与排序与现内核完全一致（允许浮点末位误差，文档顺序不能换）。**关键**：必须用**并行建索引路径**（不是串行最小段）构造能触发 :341-344 指针错误的数据，验证并行建索引下 top-k 仍一致；并且必须包含**增量更新后（新增/删除文档触发段维护）**的查询，把 skip-pointer 维护正确性验到——否则那个被点名的 bug 根本没被测到。若选择"退回串行"，则判据改成"证明串行吞吐退化幅度可接受并写进 SLA"，而不是回避这条路径。
  3. 并行建索引 bug 有明确的二选一结论且代价已计入预算。
- **红灯怎么办**：①分词系统性漂移且无法消除（jieba-rs / cppjieba 都救不了）→ BM25 IP 可移植性存疑，重审"自有中文倒排当旗舰"是否成立（止损级）。②倒排 top-k 对不上 → 定位是评分公式还是 skip-pointer/block-max 实现，修复并把工时计入；若是并行 bug 在作祟且修复成本高，退回串行并把吞吐退化写进 SLA（非止损，是减配）。无论哪种，修正后的真实人月回填总账。
- **工时**：6-9 人周（上修自原 4-6：加 dsnowball 复现 + 词典内容导出对齐 + 中英混合语料专项 2-3 人周）。
- **依赖**：无（与 C 并行，FFI 桥接相互独立）。
- **一票否决**：是（分词系统性漂移无法消除这一种）。

### C — graph_index 能否搬出内核 + 进图过滤能否把召回拉回来

- **要排的风险**：①算法搬出内核后在新段式存储上建图/搜索召回对不上；②（更致命）团队的"带过滤向量搜索"是半成品——出厂只用空过滤器（DummyFilter），且是 post-filter-during-search（不通过的点仍占 ef 预算），trace 要的 service/time/status 过滤一接进去，中等/高选择性下召回结构性塌方，不是调参能救。
- **范围修正（红队第四面，重要）**：原草稿 C 只验"会不会塌"，但决策文档已预先写明"大概率塌"——实测出塌方**不产生新信息**，只是确认已知结论。真正决定 40-60 人月成败的未知问题是"**进图过滤（ACORN 式）能不能把召回拉回来、那 3-5 人月够不够**"。原草稿把这个最关键问题推到"后续 spike"，等于把最大那笔水分推到 Phase 0 之外。**本版把它拉进 C 本体。**
- **怎么最省做**（采纳红队"更省替代"，先用出厂二进制证伪一半）：
  1. **先用现有出厂版（DuckDB 版）零搬迁证伪塌方**：直接喂一份带 service/time/status 标签的向量，手工把 DummyFilter 换成真实选择性的 filter_func（原内核改一行，不重写存储），在 0.1% / 1% / 5% 选择性下测召回。post-filter 塌方是算法属性，不是"搬进 Rust 段存储"才出现的——2-3 天就能确认塌方程度，把"确定要做 ACORN 改造"这个结论在搬迁工开始前就下掉。
  2. **算法搬迁**：把 algorithm.h / 距离内核 / PQ / RaBitQ 通过 extern C ABI FFI 封装（加异常边界）复用进 Rust，在最小 Rust 段式 block store 上跑通构图 + 搜索（可借鉴 DuckDB 那 174 行 shim 的解耦思路）。
  3. **进图过滤最小原型（本版新增、关键）**：在 graph_index 算法骨架上接一个粗糙的"过滤点也参与导航"的改法（哪怕性能差），看召回是否**原理上可恢复**。不必做完整 ACORN 工程，但这是真正决定那 3-5 人月该不该信、会不会变成 8-10 人月的信息。
- **绿灯判据（堵掉三个假绿口子）**：
  1. **空过滤召回的金标准换成暴力精确 KNN（堵掉"和半成品对齐"假绿）**：不要用 DuckDB 出厂版当金标准——出厂版自己就是 DummyFilter/post-filter 半成品，对齐它只证明"没引入新错"、证明不了"够好"。召回质量门禁 = Rust 段式存储召回@10 对齐**暴力全扫精确 top-k**（与 DuckDB 出厂版的对比保留作"移植无回归"的辅助检查，不当质量门禁）。**语料必须用真实 trace 采样向量**（含附录 Q 点名的"只对 root/LLM/error span 建 embedding 导致大量 span 无召回路径"这种实际分布），不准用算法 demo 数据集（分布良好、向量区分度高、两边都接近 1.0 的那种）蒙混。
  2. **带过滤召回测试点下沉到 0.1%-1%（堵掉"判据和声明的风险自相矛盾"）**：C 在 risk 里写明"低选择性下结构性塌方"，那判据就必须覆盖低选择性。画整条选择性谱，**0.1%-1% 高选择性区间塌即判红 / 确定触发 ACORN 改造**，无论 1%-10% 多好。**ef/过采样预算锁死到固定上线值**，同时记录召回守住时的 QPS/ef 膨胀代价——post-filter 即使靠加大 ef 把召回拉回 0.9，QPS 也可能塌到不可用，判据不能只看召回不看延迟代价。必须测 service/time/status 多谓词 AND 组合 + 高基数列。
  3. **进图过滤可恢复性（本版新增门禁）**：进图过滤最小原型在 0.1%-1% 高选择性下能把召回**原理上拉回 ≥0.9**（哪怕 QPS 差）。拉得回 → 那 3-5 人月有底；拉不回 → 这是更严重的红灯，进图过滤本身路线存疑，需重审。
- **红灯怎么办**：
  - 连空过滤召回（对暴力 KNN）都对不上 → 算法移植本身出问题，搬迁可行性红灯，重审（止损级）。
  - 带过滤召回在 0.1%-1% 塌（大概率）→ 不止损，但**确定触发进图过滤改造，回填账本**。与 A1 的"坐实 ACORN"合并成**一条结论回填一次**，不两处各填一遍。诚实标注：这 3-5 人月本身方差大，进图过滤原型若显示"勉强能恢复但 QPS 代价大"，真实工可能往 8-10 人月漂。
- **工时**：8-12 人周（上修自原 5-8：加进图过滤最小原型 + 召回恢复曲线 3-4 人周）。
- **依赖**：无（与 B 并行）。
- **一票否决**：是（空过滤召回对不上暴力 KNN 这一种）。

### D — 活 trace 读延迟 + 折叠唯一性

- **要排的风险**：①活 trace（还在持续上报）读延迟随段数线性退化、打不开详情页；②同一 span 重传被当多条，折叠失效静默重复/丢数据。附录 Q 点名：活 trace 事件落在同一 time bucket 同一 trace_id，zone-map 几乎裁不掉，必须给出 L0/L1 段数上界。折叠去重键若用引擎侧雪花 id 而非确定性 event_id，同 span 重传就会被当两条。金融政企对静默丢/重 trace 零容忍。
- **怎么最省做**：构造一批"持续上报中的活 trace"：span-start 先到、end 后到、属性补写（upgrade）、乱序到达。测打开详情页扇出到多少段、画 P99 随段数增长曲线。同一条 span 用确定性 event_id = hash(ext_span_id, seq, event_type) 重传 N 次（乱序、跨 flush/commit 边界）验折叠后恰一条。建在 C 的段式 block store + 一个简单折叠算子之上。
- **绿灯判据（堵掉"玩具负载/无 compaction 裸曲线"假绿 + 补崩溃 replay）**：
  1. **扇出上界（必须区分"有 compaction 下有界"和"无 compaction 裸曲线"）**：负载构造必须**复现附录 Q 的具体膨胀机制**——乱序/晚到事件路由到多个 L1 append-only 段、L1 在等 end/feedback 期间不被 compact、活 trace 事件挤在同一 time bucket 同 trace_id 让 zone-map 失效。**不准用稀疏/已结束 trace 蒙混**。然后二选一：要么在 D 里接入最小但真实的 L0→L1 compaction 触发逻辑（哪怕策略很糙）证明段数能被压回上界 N、P99 在该上界下守住 SLO（X 写死）；要么明确把判据改成"测出无 compaction 下段数增长斜率 + 推算需要的 compaction 频率"，并把"**必须有 compaction 才有界**"这个结论显式写进账本。不能用一个不 compact 的简单骨架既不上 compaction 又宣称"段数有上界"。
  2. **折叠唯一性（补崩溃 replay，不全甩给 E）**：同 span 重传 N 次（含乱序、跨 flush 边界）后折叠恰一条逻辑事件、内容为最后写入语义；property-based 覆盖随机交错。**本版新增**：必须加一个"**崩溃 replay 后重传**"的最小场景（kill 在 flush 提交前后，replay 重建 memtable 后再验同 span 折叠恰一条），把 hardened 设计 M.7 的 replay 漂移风险（seq/event_id 漂移导致 token/cost 翻倍）**至少做一次正面验证**，不要整段甩给排最后、最贵、要全绿才做的 E。任何一次出现两条或丢一条即判红。
  3. **局限声明**：D 的扇出结论显式声明"本结论在简化 L0 分段下成立"，D 绿不等于 E 阶段加了真实 compaction 后仍绿——把"活 trace 扇出在真实 compaction 下复测"明确登记为 E 的一个子项，避免 D 早绿给总账虚假安全感。
- **红灯怎么办**：①扇出/P99 线性退化 → 补活 trace 内存索引 + 区间树（段级 min/max ts + trace_id 路由）设计并计入工时；若加了仍线性退化 → 是 L0 分段策略问题，需重审 compaction 触发条件。**诚实标注**："重审 compaction 触发条件"背后是一笔没估的、会牵动 E 里 compaction 和段生命周期的**跨 spike 连锁改动**，红灯①时必须当场给这笔潜在改动一个**粗估区间（哪怕 1-3 人月的占位）**，不能用"把工时计入"一句话带过。②折叠重复/丢 → 去重键设计有漏，全栈锁死确定性 event_id，身份字段冻结不被 upgrade 覆盖。这条是金融准入零容忍项，不修不能上。
- **工时**：3-5 人周（红队认同大致合理）。
- **依赖**：**C**（C 产出的段式 block store + 折叠算子），**不依赖 A1 整体、不依赖 B**。这样 D 可在 C 的 block store 一通、A1 还在搭买侧/标注集时就并行起跑，提前 2-4 周。
- **一票否决**：否（但折叠唯一性是金融准入零容忍项）。

### E — 并发 + 崩溃正确性（放行之后的实现期，不算进放行门禁总账）

- **定位修正（红队第四面，重要）**：E 验的是"怎么建对"不是"要不要建"，且它本身就是大钱（12-20 人周已是 3-5 人月）。把它塞进"最小最便宜验证"阶段会让 Phase 0 总成本和周期失真。**本版把 E 从放行门禁总账里剥出，明确标成"放行之后的实现期第一阶段"。** 放行前只对 E 取一个最小信号（见下）。
- **要排的风险**：崩溃后丢已确认数据/算两遍/留半截；单写者 + 多读者并发下的回收 use-after-free、长读者把版本/内存钉死、快照串版本。这是 40-60 人月里被低估最严重的一块（团队现有持久化 100% 靠 PG xlog，搬出内核后这套全没了，且团队首次写 LSM）。它吞掉门禁 E 和 8 个 OPEN 点。
- **怎么最省做（顺序：先纸面、再最小信号、最后实现期混沌）**：
  1. **放行前：先纸面对抗（采纳红队"更省替代"）**。8 个 OPEN 里有几个本质是并发协议正确性 / 设计参数问题（M.4 的 EBR announce-before-deref 协议、OPEN-8 的 live_lsn/retained_watermark 两步非原子、OPEN-3 的 compaction 提交重读合并），**用 TLA+/模型检查穷举交错只需几人天**，比写完 Rust 再撒数百次随机 kill 又便宜又能穷尽窄窗口。**纸面打不穿的才值得写代码去混沌验证。** 决策文档 §5 已要求先各出一份 ≤2 页设计草案（段生命周期锁状态机 / 跨四源单一快照）——把"形式化/纸面证伪"作为 E 的前置门。
  2. **放行前：只做最小崩溃原子性信号**（4-6 人周可控、信息量最高）：最小 WAL（每批 CRC + commit-marker）+ 单写者 + 单一事务性 metastore 原子 commit，在 commit/flush 随机 + 确定性插桩 kill，验 current ∈ {完整新, 完整旧}。
  3. **实现期：8 个 OPEN 的完整混沌编排**（明确划到放行之后，不假装能在小钱阶段闭合）。
- **绿灯判据（堵掉"随机 kill 打不到窄窗口""画曲线冒充证明有界""最小查询树冒充成品"三个假绿）**：
  1. **崩溃原子性——确定性故障注入，不是随机撒点**：真正会出半截状态的窗口往往只有几条指令宽（M.3 文件已 fsync / metastore 事务尚未提交之间、M.4 announce slot 之后 / current.load 之前的 TENTATIVE 残窗、M.7 replay 漂移窗、M.8/OPEN-8 live_lsn 与 retained_watermark 两步非原子之间、WAL 单批 marker fsync 前后）。随机撒数百个 kill 点命中几指令宽窗口的概率极低，可能 500 次全落安全区就误判"绿"。**必须对每条已知窄临界点预埋故障注入钩子，逐点确定性触发 kill 并重启验证**；随机 kill 作为补充而非主力。验证：current 永远 ∈ {完整旧, 完整新}，无半截、无孤儿段、无悬空 tombstone、无丢已 ack 事件、无把一条 span 算两遍；用确定性 event_id 在崩溃前后对同一 (ext_span_id, seq) 折叠逐字节一致、token/cost 不翻倍（且这一校验在**确定性命中 replay 漂移窗口的前提下**验，不是随机 kill 顺带验）。
  2. **OPEN-1/2/7 内存无界——证明有上界，不是画曲线**：判据从"给出内存增长曲线"升级为"**证明存在强制失效/回收机制使内存有显式上界 M**"。必须构造"长读者永不释放 + 高频 flush/热段反复 touch"，验证"到达内存上界时强制快照失效真的触发、读者被踢、内存被回收"。一条在测试时长内还没 OOM 的曲线只证明"这几小时没炸"，不证明"有界"。
  3. **OPEN-6 DataFusion 快照——在接近成品的计划形状上验**：判据必须在"至少包含多 partition + spill + 一个融合算子（RRF）+ cancel"的计划上验 release 不早于最后一个 partition，并**明确标注"本结论绑定当前 DataFusion 版本与算子组合，真实接入新算子后需回归重验"**。最小查询树的绿灯不能替成品保证。
  4. **其余 OPEN 逐条有可复算结论**：OPEN-3 提交后无丢删无旧 upgrade 残留；OPEN-5 异常路径无"元数据指着段、段文件已删"（修 RAII 绑定）；OPEN-8 构造 flush 与 pin 交错证明 (retained_watermark, live_lsn] 区间所需行恒在。**每个 OPEN 必须"正面构造交错打穿或证明闭合"，不准用"应该没问题/一般来说"糊。**
- **红灯怎么办**：哪个 OPEN 被打穿就欠一笔对应加固工，回填账本并重测（OPEN-3 丢删→提交事务内重读合并；OPEN-1/2/7→快照租约/超时 + 内存压力强制失效 + 块版本上界/增量 diff；OPEN-5→修 RAII；OPEN-6→改 root 算子持有 snapshot；OPEN-8→改成一条原子读出）。崩溃原子性红灯是金融准入零容忍项，不绿不能上。
- **工时与无底洞标注（诚实）**：原估 12-20 人周对应的是"每个 OPEN 一次过、骨架不返工"的顺风假设，这和"首次写 LSM + 金融审计零容忍"矛盾。**真实区间判 20-35 人周，方差极大、上界开放。** 处置：
  - E 的估算不用纯人周，而是"**人周 + 时间盒 + 退出判据**"：例如跑满 N 人周混沌测试若仍在发现新 use-after-free，则结论是"WAL/并发这块需要外部专家或更长周期"，而不是继续往里填人月——给无底洞装一个底。
  - 放行前只投最小崩溃原子性信号那 4-6 人周；8 个 OPEN 的混沌编排明确划到实现期。
- **依赖**：**A2、A1、B、C、D**（补上 D——E 的崩溃原子性判据反复用到"确定性 event_id 折叠唯一性"，这正是 D 的核心交付物。D 先在无崩溃正常路径上把 event_id/折叠键打穿，E 再在已验证的折叠键之上叠崩溃注入，这样 E 里任何"折叠翻倍/丢"都能干净归因到崩溃路径而非折叠键本身，省掉大量"到底是 event_id 设计错还是 replay 漂移"的定位时间）。
- **一票否决**：否（前面 A2/A1/B/C 已确认要建）。但崩溃原子性子项是上线前金融准入硬门。

## 推荐执行顺序（最便宜最致命的先做）

总原则：最便宜又最致命的先做；评测基建不依赖自研侧就第 0 周起跑别空等；最贵的并发 + 崩溃正确性（E）只在"确定要建"之后才做，且大部分划到放行之后。

**第一波（第 0 周即起跑，并行）——挤掉"整个前提"和"评测基建空等"的水：**
- **A2a（付费意愿，纯需求侧，零代码）** 立即启动。先 1 周桌面挖标书，再 3-6 家报价剥离深访。最便宜的一票否决，绝不能等、绝不能绑工程。
- **A1-基建（买侧基线 + 标注集 + 评测脚手架 + 选择性谱/热数据态夹具）** 第 0 周与 A2a/B/C 并行起跑，**根本不用等 B/C**。这样到 B/C 一打通，A1-自研侧当周就能出分。
- **B（BM25 搬迁）** 起跑：先 2-3 天零 FFI 的分词对齐（命令行 cppjieba + jieba-rs 对真实客户语料），再决定 FFI 投不投。
- **C（graph_index 搬迁）** 起跑：先 2-3 天用出厂二进制证伪带过滤召回塌方，再投 Rust 段存储 + 进图过滤原型。
- 注意：B/C/A1-基建里的工程 PoC 需内核/向量人已到位；若还在招人，这几项起步即顺延，**A2a 不受影响照常先跑**（这恰好印证"最便宜最致命的先做"）。

**第 3 周：早期止损门（本版新增，不等到第 9 周）。** 只看两件最便宜的一票否决：
- **A2a 的付费意愿信号**：红 → 止损规则 #1 立即触发，停掉尚在进行的 B/C/A1，省最大一笔（避免等 6 周访谈和最贵实验一起揭晓）。
- **B 的分词一致率**（含三类子集加权）：系统性漂移且 jieba-rs/cppjieba 都救不了 → 升级 B 优先级先解决分词，不要等倒排重写做完才发现根上 token 就对不齐。

**第二波（B/C 的 FFI 打通后）——挤掉"造得出但赢不过买侧"的水：**
- **A1-自研侧接入**：把 B 的 BM25、C 的带过滤搜索接进 A1-基建那套夹具出分。A1 的②带过滤召回**复用 C 已测的那条选择性谱曲线**，不再重测一遍——同一份语料、同一选择性谱、同一 ef 预算、同一召回口径；C 的"进图过滤可行性"结论与 A1 的"坐实 ACORN"**合并成一条回填账本一次**。
- **D（活 trace 读延迟）依赖 C（不依赖 A1 整体、不依赖 B）**：C 的 block store 一通就能并行起跑，提前 2-4 周。D 是金融准入零容忍项，越早暴露"活 trace 读模型有结构问题"越好，不被对标赛进度绑架。

第一波 + 第二波 = 真正的放行门禁：A2/A1/B/C/D 全绿（尤其 A2/A1/B/C 四个能一票否决的）才证明值得投 40-60 人月。

**第 ~9 周（满编口径）：放行门评审 —— go/no-go 决策点。**

**第三波（仅过门后）——最贵的崩溃 + 并发正确性：**
- **E** 依赖 A2/A1/B/C/D 全绿。先纸面对抗（TLA+/模型检查几人天）+ 最小崩溃原子性信号（4-6 人周）。8 个 OPEN 的完整混沌编排划到放行之后的实现期，装时间盒。

**时间线示意（两个版本，诚实标注）：**

| 阶段 | 满编版（人已到位） | 现实版（招聘中） |
|---|---|---|
| A2a 起跑（商务，零代码） | 第 0 周 | 第 0 周（不受招聘影响） |
| B/C/A1-基建 起跑 | 第 0 周 | 招聘 + 磨合 3-6 个月后 |
| 早期止损门（A2a + B 分词） | 第 3 周 | A2a 第 3 周；B 分词随工程组到位 |
| B/C FFI 打通 | 第 4-6 周 | 工程组到位 + 4-6 周 |
| A1-自研侧 ‖ D | 第 5-10 周 | 相应顺延 |
| 放行门评审（go/no-go） | **第 ~9-11 周** | **大概率 5-8 个自然月** |
| E（仅过门）：纸面 + 最小信号 + 实现期混沌 | 门后 6-10 周起，含时间盒 | 相应顺延，方差更大 |

新人前 2 个月产出按 5 折计；人周不变，自然周期 +3-6 个月。

## 放行 / 止损判定逻辑

**放行规则（哪几个绿了才放行 40-60 人月）：**
必须 **A2、A1、B、C 四个能一票否决的全绿**，且 **D 绿（或红灯已有明确低成本修复并回填工时）**，才放行整个 40-60 人月自研。**E 不是放行门**（在放行之后做），但 E 的崩溃原子性子项是上线前金融政企准入的硬门——不绿不能交付。

**立刻止损规则（哪个红了立刻止损）——按"最便宜先打"排序：**
1. **A2a 红**（客户在功能不可区分时仍不为"索引自研身份"多付钱 / 标书里找不到颗粒度到"索引算法自研"的硬条款）= 最高优先级止损。立刻停掉 B/C/A1/D/E 全部工程，转买侧，人月投平台层。**前置条件**：红灯动作必须事先书面签字、结论由非引擎负责人判定，否则现实中极难真砍。
2. **A1 红**（接对骨架后，自研栈在 ≥300 条客户真实标注查询、对称调优、0.1%-1% 高选择性带过滤召回、热数据态这套真工况下，赢不过钢人化买侧）= 与 A2 合并重审，大概率翻回买侧。**"接对"最多允许一次返工，第二次仍输强制重审，不允许第三次"再接接看"。**
3. **B 红**（分词系统性漂移且 cppjieba/jieba-rs 都无法消除 / 倒排在并行建索引 + 增量路径下 top-k 对不上现内核）= BM25 IP 搬迁可行性存疑，重审"自有中文倒排当旗舰"。若只是并行建索引 bug，退串行并计入工时（非止损，是减配）。
4. **C 红**（空过滤召回对不上**暴力精确 KNN** = 算法移植本身出问题，止损级 / 进图过滤最小原型显示召回**原理上拉不回** = 进图过滤路线存疑，需重审）。

**不构成止损、只构成"加工时回填账本"的红灯（让真实人月从 40-60 往上漂，但不否掉"要建"）：**
- C 的带过滤召回在 0.1%-1% 塌（大概率）→ 确定触发进图过滤改造（名义 3-5 人月，但原型若显示"勉强恢复 + QPS 代价大"可能漂到 8-10）。与 A1 合并回填一次。
- D 的扇出/折叠问题 → 补区间树 / 锁死确定性 event_id；若指向"重审 compaction 触发条件"，必须当场给跨 spike 连锁改动一个 1-3 人月占位估，不准一句"把工时计入"带过。
- E 的任一 OPEN 被打穿 → 对应加固工（见 E 的 fail_action）。

**诚实标注哪些是无底洞 / 工时不确定：**
- **E 是无底洞**：真实区间 20-35 人周、上界开放。必须装"时间盒 + 退出判据"，跑满 N 人周仍在出新 use-after-free 就结论"需外部专家或更长周期"，不无限填人月。
- **A1 自研侧的 DataFusion 自定义节点**（filter 下推 + fold + bm25_score UDF + RRF + 跨四源快照在一个计划里）是决策文档自列的"最大不确定项"，无单一先例，工时方差大。
- **C 的进图过滤 3-5 人月**本身是开放研究问题，原型只验"原理上能不能恢复"，工程化真实代价方差大。
- **招聘是隐藏的最大周期风险**：决策摘要承认 10-12 人多数还不存在，从 0 招齐资深 Rust/内核/向量人至少 3-6 个月、新人前 2 个月产出打 5 折。这条已进时间线主轴（两个版本），不再只躺在"前提风险"段落里。

**一句话**：A2a 是最便宜的承重墙，先单独打、第 3 周设早杀门；A2/A1/B/C 四个止损级门全绿 = 值得投；判据已全部从"跑通了/客户说了"改成"在真工况下可复算、且区分得开为结果付费还是为自研身份付费"；E 最贵、是无底洞、只在确定要建后做、装时间盒、其崩溃原子性是上线前金融准入的最后一道硬门。
