# 第三轮：存储引擎重新决策（信创自主可控为首要 + 金融政企首客）

## 推荐结论

**推荐 (A) 的强化版：以 openGauss 内核为底座做"列存 trace 引擎扩展 / 衍生发行版"，但热区写入路径用自研行式 LSM 旁路（不走 openGauss cstore），列式不可变 segment + 倒排 + 向量召回复用 openGauss 体系。**

一句话：**A 作为认证主体与自主可控叙事载体 + 自研 LSM/segment 作为 trace 数据范式承载层**。完全摒弃 (C) 的 Lance/DataFusion 外国底座，但**不**走 (B) 的"连列存格式都从零自研"的最重路线——把自研深度精准投在 trace 负载真正需要、且 openGauss 补不上的地方。

下面给硬理由。每条都标了查证结论与不确定项。

## 一、改变前两轮结论的几个查证事实（必须先摆出来）

1. **"信创目录"这个前提本身需要修正（重要）。** 多方资料显示：2022 年信创目录停止更新、2023 年起官方转向**"安全可靠测评（安可测评）"**体系，由中国信息安全测评中心按批次发布结果，覆盖 CPU / OS / 数据库等核心产品，有效期相关、面向"在中国境内注册、拥有送测产品完整设计资料/研发文档/代码数据/研发环境、拥有知识产权或授权"的厂商。**含义：你们要进的不是"上一个静态目录"，而是"过安可测评"。** 这把(A)的优势从"openGauss 已在目录"修正为"openGauss 系生态已成体系化通过安可测评、且测评对'代码数据/研发环境/知识产权可控'有硬性要求"——这恰恰是(C)外国底座的死穴。
   - 不确定项：信创目录是否在所有金融政企招投标中"完全失效"存在地区/行业差异，部分招标仍写"信创证书/适配认证"。建议按"安可测评 + 信创适配认证（与麒麟/统信/鲲鹏互认）"双轨准备，**这一点务必让销售/合规向目标客户的具体招标文件核实**。

2. **openGauss 已自带 DataVec 向量引擎（颠覆性利好 A）。** openGauss 6.0.0 起内置 DataVec 向量数据库插件：原生 `VECTOR(n)` 类型（float 最高 16000 维）、L2/cosine/内积、**HNSW + IVFFlat + IVF-PQ + IVF-RabitQ + HNSW-PQ + HNSW-RabitQ** 全套索引，还有 halfvec/bit/sparse 向量。**这意味着"带过滤的语义 trace 召回"这个招牌钩子可以直接长在 openGauss 体系内**，不必为它去依赖外部向量库或外国格式。这是上一轮没有充分计入的，单独把天平大幅推向 A。
   - 不确定项：DataVec 的 HNSW 在"单机本地 NVMe + 与标量/JSONB 过滤下推联动 + 增量建索引"上的实测性能未查到第三方数据；你们团队本就自研 IVF/HNSW/DiskANN，可在 openGauss 框架内替换/增强其向量 AM，这反而是团队最深资产的复用点。

3. **openGauss cstore 不适合 trace 的高频碎片化写（A 的真实短板，已查实）。** cstore 以 CU 为 I/O 单元、**只支持追加写**；单条/小批量 insert/update 靠 delta 表（行存 astore）缓冲，默认 `enable_delta_store=off`，设计明确面向 OLAP。已查实的硬限制：**列存表不支持数组、建议列数 ≤1000、表级约束只支持 PARTIAL CLUSTER KEY（无主外键）、字段约束只支持 NULL/NOT NULL/DEFAULT 常量**。trace 是"高频碎片化写 + 半结构化 attributes + 大字段"——直接拿 cstore 当热区写入层会撞上 delta 表碎片化和约束限制。**这正是必须自研 LSM 旁路、而不是硬套 cstore 的原因。**

4. **"用 Apache-2.0 外国开源做信创"是软肋而非硬伤，但对金融政企+自主可控叙事不可接受（C 的判决）。** 查实：开源软件可用于信创项目，许可证层面 Apache-2.0 允许闭源/商用/二次开发无障碍；安可测评的判定看的是**"自身研发设计、全面掌握核心技术、从硬件到软件全程可控、拥有知识产权/研发环境/代码数据"**。同时供应链安全测评里有**"自研代码占比检测 + 软件成分分析(SCA)"**，部分地区（如上海某些口径）甚至限制开发主体外资比例 ≤25%。
   - 判决：Lance（美 LanceDB 主导的列存格式）+ DataFusion（美主导 Apache 项目）作为**数据面核心**，在"国产首个、完全自主可控的 Agent 专用数据库"叙事下是**自我打脸**——竞标时一旦做 SCA / 自研占比检测，核心存储与查询执行是外国主导项目，叙事和评分双输。许可证合规 ≠ 自主可控叙事。**(C) 在本轮前提下出局。**

## 三、为什么是"A 强化版"而不是纯 A、也不是 B

**不选纯 A：** openGauss cstore 的追加写 + delta 表 + 列存约束限制，与 SmithDB 式 trace 范式（LSM 行式热区 → 列式不可变 segment → 分层 compaction、merge-on-read、run+deletion/upgrade vector、大字段晚物化、自研倒排、trace 树双编码）**不匹配**。硬把 trace 塞进 cstore 表会在活 trace 写入、半结构化 attributes、deletion vector 上付出巨大代价。

**不选纯 B（全自研格式）：** 在"不缺时间（17-29 月）"下 B 看似可行，但有两个杀手级反对理由：
- **认证收益递减。** 安可测评要的是"知识产权/代码/研发环境可控"，openGauss 衍生发行版（基于 Mulan PSL v2、可获完整源码、可自主修改维护、华为主导的国产项目）在测评口径下**已能讲清自主可控**——比"美国 Lance"强一个数量级。从 A 强化版再往 B 走，自主可控叙事的边际提升远小于多造一套列存格式/事务/恢复/优化器的成本。
- **风险叠加。** B 等于放弃 openGauss 的恢复/HA/向量化执行/DataVec/优化器/页面框架，全部重写。金融政企首客要 **HA 主备**——openGauss 的主备流复制、DCF/Paxos 高可用是现成且已被金融客户验证的资产，自研要把这块也补上，纯属把"长杆认证时间"浪费在重复造轮子上。

**A 强化版精准取舍——自研只投在三处，其余全复用 openGauss：**
1. **热区写入层：自研行式 LSM**（行式热区 → 冻结为列式不可变 segment）。绕开 cstore 的 delta 表碎片化和约束限制。这部分用你们的 Rust 能力 + openGauss 存储 AM 扩展接口实现（openGauss 支持自定义表访问方法，类似 PG 的 Table AM），把 segment 落成 openGauss 可识别的列存或自定义 AM。
2. **trace 专用结构：自研倒排 + trace 树双编码（邻接表 + [pre,post] 区间）+ merge-on-read 的 deletion/upgrade vector + 大字段晚物化**。这是 SmithDB 范式的核心差异化，openGauss 没有现成的，必须自研——但它是"在 openGauss 体系内的扩展"，不是另起炉灶。
3. **语义召回钩子：复用并增强 DataVec 的向量 AM**，把你们自研的 IVF/HNSW/DiskANN + 三区增量（活区在线 / 温区近线 / 冷区批量）接进去，与标量/JSONB 过滤下推联动。气隙 embedding 用 Qwen3-Embedding-0.6B（Apache-2.0 但**作为可替换的本地模型权重、非数据面核心组件**，对叙事影响小；若客户极端敏感可声明权重本地化、可替换为国产 embedding）。

**全部复用 openGauss：** SQL 解析/优化器、向量化执行引擎、JSONB（带索引，承载 attributes）、HA 主备（流复制/DCF）、WAL/恢复、页面整理框架、磁盘索引框架、安可测评/信创适配/国密集成路径（openGauss 已有国密 SSL/透明加密生态）。中文全文检索需自研或集成中文分词（zhparser 类，未查实 openGauss 原生中文分词成熟度——**标注为不确定项，需 PoC 验证**）。

## 四、中小规模性能是否够

够。理由：
- 目标负载 **<1 亿 span/天、单机 NVMe、~250GB~25TB**，属单机中小规模，不触碰 openGauss 分布式/大集群短板。
- 写入瓶颈由**自研 LSM 热区**承接（行式追加、高吞吐），不走 cstore 单条写慢路径——短板已被旁路设计规避。
- 列式聚合/扫描走冻结后的列式 segment + openGauss **向量化执行引擎**（3.0.0 起，面向 OLAP 顺序扫描，正好是 trace 聚合查询形态）。
- 语义召回走 DataVec/自研向量 AM 的 HNSW，单机本地 NVMe 上 ANN 这是你们最深资产，本就比 SmithDB 的对象存储+无状态架构有结构性优势（招牌钩子成立）。
- **不确定项需 PoC 实测**：(a) 自研 LSM segment 与 openGauss 列存/向量化执行的拼接开销；(b) DataVec HNSW 增量建索引 + 过滤下推的真实延迟；(c) 中文全文检索在 openGauss 上的方案与性能。建议在认证长杆期内并行做这三个 PoC。

## 一、核心判断：长杆不是"测评"，是"名录批次窗口 + 工委会准入"

这是本轮最重要的修正，直接决定 Phase 0 该启动什么：

- **单项"适配互认证"很快**：在已有信创测试环境下，软件与某一 CPU+OS+数据库栈的兼容性互认证 **1–2 周/栈**；从零搭环境 **≥1 个月**。（来源：极创/麒麟生态适配中心口径）
- **"安全可靠测评"（中国信息安全测评中心 CITTC 主办）测试阶段本身也短**：约 **20 个工作日（~1 个月）**，含环境准备、产品测试、专家评审、发证。
- **真正的长杆是"进名录"的节奏**：安全可靠测评结果**以公告批次发布，一年仅约 2 次**（核实：2023 第1号；2024 第1、2号；2025 第1、2号；2026 已发第1、2号）。错过一个批次窗口=等半年。**且"安可测评"主战场是 CPU/AI芯片/OS/数据库**——yiTrace 作为"Agent 可观测平台/应用软件"，更可能走的是 **CESI 信创产品认证（CNAS/CMA 备案）**，其前置条件正是"符合安全可靠测评结果要求 + 第三方适配报告 + 知识产权报告"。
- **金融政企 82% 要求"先在名录"才进采购名单**的 12–24 个月，= 排队 + 批次窗口 + 适配多栈 + 客户内部立项/招标周期的叠加，**不是单次测评工时**。

**对路线图的含义**：你不能靠"产品做完再测"压缩它——必须在 Phase 0 就**并行启动适配+知识产权固化+建立机构合作关系**，把自己塞进最近的批次窗口，否则每错过一窗就丢半年。这与"我们不缺时间"是一致的：时间不缺，但**批次窗口不等人**。

## 三、① 信创：适配清单与自主可控障碍

### 适配/互认证矩阵（务必三栈全覆盖，金融政企混装常见）
- **CPU**：鲲鹏（ARMv8，永久授权）、飞腾（ARMv8，永久授权）、海光（x86-64 授权）。可选龙芯（LoongArch 自主）作为"完全自主"叙事加分。
- **OS**：银河麒麟、统信 UOS（均支持上述全部国产 CPU）。
- **数据库底座**：openGauss（Mulan PSL v2，已在信创目录）——这是你最大的合规资产。

### 外国开源组件在国产 CPU+OS 上的"能跑 / 自主可控"评估（本轮天平倒向更深自研的依据）

| 组件 | 来源/风险 | 国产 CPU+OS 可编译运行？ | 自主可控/认证风险 | 本轮建议 |
|---|---|---|---|---|
| **Rust toolchain** | 多国社区 | **ARM64/x86-64 = Tier 1/2；LoongArch = Tier 2（1.71+）**，鲲鹏/飞腾/海光均原生支持 | 低（工具链非卡点） | **保留**。Rust 自研栈在国产 CPU 上无障碍，是叙事正资产 |
| **Apache DataFusion / Arrow** | 美国主导 ASF 项目 | aarch64 可编译（DataFusion-Java 已打包 aarch64 native），但需调编译参数 | **叙事软肋 + 审计风险**：美国主导查询引擎与"完全自主可控"冲突 | **本轮替换/自研**：查询执行下沉到 openGauss 内核 + 自研算子，去 DataFusion 依赖 |
| **Lance（列存格式）** | 美国 LanceDB | aarch64 有过构建摩擦（Android/aarch64 Rust build issue） | **叙事+认证双软肋**：外国列存格式做底座，与"自主可控数据库"叙事直接矛盾 | **本轮去 Lance**：用前两轮已定的自研 LSM（行式热区→列式不可变 segment）+ openGauss cstore 列存能力承接列存，把"格式"做成自研 |
| **ONNX Runtime** | 微软 | aarch64 可交叉编译；**CANN EP 存在但为华为/社区维护**，非微软核心 | 中：作为推理运行时是外国件，但 CANN EP 可走昇腾 | **保留但隔离**：embedding 推理运行时可换昇腾 MindIE，降低对 ONNX 的硬依赖 |
| **Qwen3-Embedding-0.6B** | 阿里，Apache 2.0，中文好 | 是（昇腾 0-day 支持 Qwen3 系列） | 低（国产模型，正资产） | **保留**，作为气隙 embedding 招牌 |

> 结论与本轮战略一致：**砍掉 Lance + DataFusion 两个美国底座**，把列存/查询自研化下沉到 openGauss 内核与自研 LSM；Rust/Qwen3 保留并作为正资产；ONNX 用昇腾 MindIE 兜底以备气隙/无 GPU 场景。这同时消除"叙事软肋"和"安可/密评的供应链审计风险"。

## 四、② 等保 2.0 三级 —— 必须做进产品的硬交付清单

三级须**定级专家评审 + 每年复测 + 测评得分 ≥75 才算基本符合**。产品侧（非纯运维侧）必须内建以下能力，否则现场测评一票卡死：

1. **身份鉴别**：双因素（口令+另一因素）；登录失败处理/锁定；会话超时；远程管理通道加密。
2. **访问控制**：基于角色的最小权限（RBAC）；trace/span 数据按租户/项目隔离；敏感操作权限分离（三权分立：系统管理员/安全管理员/审计管理员）。
3. **安全审计**：覆盖用户行为与安全事件的审计日志；审计记录防删改、可备份、留存（建议 ≥6 个月）；审计独立于被审计对象。
4. **数据加密**：传输加密（国密 SM2 协商 + SM4 通道）；存储加密（重要数据 SM4，对 trace 中高敏字段/PII 落盘加密）。
5. **备份恢复**：本地+异地备份；HA 主备（已锁定）；可演练的恢复流程。
6. **入侵防范 / 恶意代码 / 安全管理中心**：最小化安装、关闭多余端口、组件可被堡垒机纳管。
7. **个人信息保护**：与下方脱敏清单合并实现。

---

## 五、③ 国密 SM2/SM3/SM4 —— 三链路集成要求

- **传输**：SM2 做双向证书认证与密钥协商，SM4 做通道加密（国密 TLS/TLCP）。摄入端（OTel/OpenInference/LangSmith-compat + 自家 agent 深集成）与查询端均需支持。
- **存储**：高敏字段/PII 用 **SM4** 落盘加密；可借力 openGauss 自 2.0 起的 SM4 加解密函数（gs_encrypt/gs_decrypt 走 OpenSSL EVP_sm4_cbc），但**透明加密(TDE)能力需确认你的内核分支是否具备**（标注不确定，须验证）。
- **认证**：用户口令用 **SM3** 摘要（openGauss `password_encryption_type=3`，仅 gsql/JDBC/ODBC 支持，须确认你的接入方式覆盖）。
- **密码模块合规**：所用密码须为**经国家密码管理机构认可的商用密码产品**（自研算法实现需过检测，或集成已认证的密码模块/密码机）——这是密评必查项，**别用未认证的自实现 SM 算法**。

---

## 六、④ 金融行业（JR/T 等）额外要求

- **定级**：存重要数据系统须**等保三级**；存核心数据须**四级或关基(KII)保护**。yiTrace 存的是 trace/span（多为重要数据）→ 锁定三级；若客户把它接核心业务链路则可能拔高。
- **强制密评（2025 新规，硬约束）**：《关键信息基础设施商用密码使用管理规定》（2025-07-01 施行）+《中国人民银行业务领域数据安全管理办法》（2025-06-30 施行）要求金融 KII **每年开展密评**，并于**每年 1 月 31 日前**报告商用密码使用与评估情况。→ 密评从"加分"变"年度强制"。
- **HA/RPO/RTO**：银监 BCM 指引对重要业务 **RTO ≤ 4 小时、RPO ≤ 0.5 小时**；行业灾备常要求**两地三中心**（同城 >50km、异地 >300km）。单机中小定位下，至少须交付 **HA 主备 + 同城备份 + 可演练恢复**，并在 SKU 文档中明确 RPO/RTO 承诺。参考标准：JR/T 0072（金融等保测评指南）、JR/T 0205（分布式数据库金融应用灾难恢复要求）。

## 七、⑤ 昇腾/CANN —— 是否前置？

**结论：不是 Phase 0 前置，是"按订单条件触发"的 Phase 1–2 适配项**，但要提前做架构隔离以便快速触发。

- 昇腾对 **Qwen3 系列 0-day 支持**，推理走 **MindIE**（或 ONNX Runtime CANN EP）；Atlas 300I Duo / 910B 上部署 embedding/rerank 有成熟实践。
- 你的气隙 embedding 是 **Qwen3-Embedding-0.6B（小模型）**，CPU（鲲鹏/海光）即可跑活区/温区在线推理，**多数中小单机客户无需 AI 卡**。
- **仅当客户机房配国产 AI 卡且禁用 GPU/要求卡上推理时**，才需昇腾适配（1–3 月 + 测试卡成本）。
- 工程动作（现在就做，成本低）：把 embedding 推理后端抽象成可插拔（CPU / ONNX-CPU / 昇腾 MindIE / CANN EP），避免临单时大改。**自主可控叙事建议优先 MindIE 路线**（华为自研栈），弱化对微软 ONNX 的依赖。

## 八、HA / 脱敏 / 等保的"产品化硬交付清单"（开发待办，必须进产品而非靠运维补）

**HA（金融硬指标）**
- 主备复制 + 自动故障切换（failover），切换后满足 RTO ≤ 4h、RPO ≤ 0.5h；
- 同城备份 + 异地备份导出；备份可校验、可演练恢复；
- 健康探活、脑裂防护、切换审计留痕。

**数据脱敏 / 个人信息保护（金融准入前置）**
- 摄入侧 PII 检测与脱敏（trace/span 里的手机号、身份证、卡号、prompt 中的敏感内容）；
- 静态脱敏（落盘前）+ 动态脱敏（查询返回按角色掩码）；
- 脱敏规则可配置、可审计；与 SM4 存储加密叠加。

**等保三级内建能力**
- 三权分立账户体系（系统/安全/审计管理员）+ RBAC + 多租户/项目隔离；
- 双因素登录、失败锁定、会话超时；
- 防删改审计日志（独立存储、≥6 个月留存、可导出）；
- 国密传输（SM2+SM4 TLCP）+ 国密存储（SM4）+ 国密口令（SM3）；
- 最小化部署、端口收敛、可被堡垒机/审计系统纳管。

## 九、给决策者的三条收口建议

1. **Phase 0 立刻并行启动且不可压缩的四件事**：知识产权固化(①)、三类 CPU×两类 OS×openGauss 适配互认证(②)、CESI 信创产品认证报名抢最近批次(③)、与麒麟/鲲鹏/统信建合作关系。**批次窗口一年 2 次，错过即丢半年**，这是 17–29 个月时间线里唯一不能靠加人解决的部分。
2. **本轮去掉 Lance + DataFusion 两个美国底座**：换为自研 LSM 列存 + openGauss cstore + 自研算子。既消除"完全自主可控"叙事软肋，也消除安可/密评供应链审计风险；Rust/Qwen3 保留为正资产，ONNX 用昇腾 MindIE 兜底。
3. **密评已是金融年度强制（2025 新规）**：国密三链路集成(⑦)必须在 Phase 0–1 进产品，且**密码实现须用经国家认可的商用密码产品**，禁止上线未认证的自实现 SM 算法。

## ① 一句话定位 + 杀手级 demo

### 一句话定位

> **yiTrace：国产首个、完全自主可控的 Agent 专用数据库——在一台国产服务器上，用一句中文问出你的 Agent 到底干了什么。**

拆成对金融政企的三层语义（按采购决策者的关心顺序排列）：

1. **准入层（CIO/采购/安全合规听）**：纯国产技术栈、信创目录在册、等保三级、国密加密——**能进采购名单**。
2. **差异层（业务/AI 负责人听）**：原生中文、SQL 可查、带过滤的语义 trace 召回——**别家做不到**。
3. **私有化层（基础设施负责人听）**：单机一柜、NVMe 本地盘、气隙可运行、无需公网无需对象存储——**数据不出域**。

一句话提炼成可贴在标书首页的话术：**"对手能做可观测，但做不了'信创自主可控'；我们两样都做，且自主可控是结构性的、不是包一层皮。"**

### 杀手级 demo 脚本（现场 8 分钟，目标=让在场金融客户"哇"+"这别家给不了")

设计原则：**每一幕都同时击中"自主可控"和"语义召回"两个钩子**，让评审无法把"国产"和"能力差"划等号。

**第 0 幕｜亮国产底座（30 秒，建立准入信任）**
- 大屏左下角常驻一个"自主可控成分仪表盘"（见 ③），现场展示运行在**鲲鹏/飞腾 CPU + 麒麟/统信 OS + 昇腾推理卡**的国产整机上。
- 一句话:"这台机器里没有一颗 x86，没有一行依赖境外云。"
- 现场断网（拔网线/进气隙模式），强调:"接下来全程离线。"

**第 1 幕｜中文问"出事了吗"（语义召回 + 中文，1.5 分钟）**
- 输入一句中文自然语言:**"昨天有哪些理财顾问 Agent 在回答'保本'相关问题时绕过了合规检查？"**
- 系统把这句中文 embedding（Qwen3-Embedding-0.6B，昇腾上推理），在语义 trace 召回三区里做**带过滤的语义召回**：过滤条件=时间窗+租户+工具调用链含"合规节点缺失"，语义条件="保本/刚兑"近义召回。
- 1 秒内返回 23 条可疑 trace。**钩子点**:"这是中文语义理解，不是关键词 grep——'保本''刚兑''零风险'都召回了，这是 SmithDB/Langfuse 给不了的。"

**第 2 幕｜下钻一条 trace 树（双编码读路径，1.5 分钟）**
- 点开一条 trace，瞬间展开完整调用树（读侧 [pre,post] 区间编码，子树查询 O(1) 范围扫描，无递归）。
- 高亮:LLM 节点输入输出（大字段晚物化，点开才拉全文）、本该出现的"合规校验工具"节点缺失。
- **钩子点**:"百万级 span 的树，下钻不卡，因为读路径是区间编码不是递归 join。"

**第 3 幕｜SQL 直查 + 审计可解释（SQL 可查 + 等保/合规,1.5 分钟）**
- 切到 SQL 面板，写一条标准 SQL:`SELECT agent_id, count(*) FROM spans WHERE semantic_match('保本话术') AND NOT EXISTS (compliance_node) GROUP BY agent_id`（语义召回作为 SQL 函数/算子暴露）。
- **钩子点**:"金融审计要的是可复现、可留痕、可写进报告的查询语言。这是一条能贴进监管报送的 SQL，不是一个黑盒 AI 接口。" 同时展示数据列在磁盘上是国密 SM4 加密、查询全程审计日志（等保三级要点）。

**第 4 幕｜双轨摄入（带动叙事,1 分钟）**
- 左边:本行自研 Agent 通过 AgenticData 深集成上报（带因果/成本/中间态的"富 trace"）。
- 右边:一个跑在 LangChain 的存量应用通过 OTel/OpenInference 上报（框架无关）。
- **钩子点**:"你已有的、未来要换的 Agent 框架我们都收;你用我们自家 Agent，能看到别人看不到的深度。"

**第 5 幕｜收尾暴击（30 秒,把准入护城河钉死）**
- 一张对比卡:把 SmithDB/Langfuse/Phoenix 摆上去，逐项打勾/打叉:信创目录、等保三级、国密、数据不出境、原生中文语义召回。
- 最后一句:**"它们不是'暂时没做'信创——是架构上做不了。下一节我讲为什么。"** → 自然过渡到护城河论证。

> Demo 的"哇"不是单点炫技，而是**"国产居然能这么强 + 强的地方对手结构上补不上"**的复合冲击。气隙断网那一下是金融政企最吃的一口。

## ② 结构性护城河：对手「做不了信创自主可控」（采购准入级，最硬）

这是本轮最硬的论点：**比中文/语义召回更硬，因为它是采购准入级——做不到=连标书都进不了，能力再强都是零。**

### 2.1 准入是"长杆"，不是"加分项"（已核实）

- 信创认证由工信部体系（信创工委会/中国电子工业标准化技术协会）主导，金融政企采购普遍要求供应商**先在信创目录在册**才进采购名单。等保 2.0 三级 + 国密（GB/T 39786《信息系统密码应用基本要求》）对三级系统是**强制项**:存储/传输需用国密 SM2/SM3/SM4，禁用 RSA-1024/SHA-1 等。
- 含义:**采购是一道"与门"**——信创 AND 等保 AND 国密 AND 能力，任一为 0 则整体为 0。对手在前三项结构性为 0。

### 2.2 为什么对手是"结构上做不了"，不是"还没做"（已核实其架构）

| 对手 | 许可证/架构（核实结论） | 为什么结构上做不了信创 |
|---|---|---|
| **Langfuse** | MIT，但 v3 起核心数据层=**ClickHouse**（高写入观测数据）+ Postgres + Redis + **S3/对象存储**（大 payload）。架构演进的方向就是"上 ClickHouse、上对象存储"。 | ① 依赖 ClickHouse（俄裔创始团队、美国注册公司主导）+ S3 生态;② 架构哲学是云原生分布式多组件，**与"单机气隙+国产整机"反向**;③ 公司是外国主体，不会去走中国信创认证。即便客户自己改，**整条依赖链没有任何一环在信创目录**。 |
| **Phoenix（Arize）** | **Elastic License 2.0（ELv2，非 OSI 开源）**;OTel 采集。 | ① ELv2 限制商业再分发，国产厂商无法合法二次发行做信创产品;② 美国公司主导，无信创认证路径。 |
| **SmithDB/LangSmith** | 闭源/对象存储+无状态架构（前两轮已定其 ANN 哲学与单机语义召回冲突）。 | 闭源、美国主体、SaaS 优先，**根本不提供可私有化的可信创版本**;数据出境是金融政企红线。 |
| **DataFusion / Lance / ONNX Runtime** | DataFusion=Apache 基金会（美国主导）;Lance=美国 LanceDB 公司（YC 系）;ONNX Runtime=微软。均为美国主体/主导。 | 若 yiTrace 把它们放在**关键路径**，自己就成了"披国产皮、芯是美国件"——既是叙事软肋又是认证风险（见 ⑤）。这正是本轮要把它们降到最低的原因。 |

**护城河论证的钉子（标书/答辩用语）**：

> "可观测能力是可追赶的工程问题;**信创自主可控是主体资格问题**。Langfuse 依赖 ClickHouse 和对象存储、Phoenix 用的是非开源的 Elastic License、SmithDB 是美国闭源 SaaS——它们的**公司主体、许可证、架构哲学**三样里没有一样允许它们走完中国信创认证。这不是时间问题，是**它们不会、也无法为中国监管重写自己**。而我们从内核（openGauss，Mulan PSL，已在信创目录）到 embedding（Qwen3，Apache 2.0，昇腾推理）到整机（鲲鹏/飞腾+麒麟/统信）是端到端国产。"

### 2.3 护城河的"时间锁"——长杆反而是壁垒

认证 12–24 月是**双刃壁垒**:它推迟了我们的首单（17–29 月），但**同样把任何想转身做信创的对手挡在 1–2 年外**。我们用这段必然要等的时间，把自研深度做到对手即便想抄也来不及（见 ③ 的自研栈）。**"我们已经在认证流程里、它们连主体资格都没有"——这是会随时间变宽的护城河。**

### 2.4 护城河分层（从硬到软，按采购权重）

1. **准入级（最硬，结构性）**:信创目录 + 等保三级 + 国密 + 数据不出境。对手=0。
2. **能力级（次硬，工程性）**:原生中文 + 带过滤语义 trace 召回 + 单机百万 span 不卡。对手追赶需重写存储与 ANN 哲学。
3. **生态级（增益）**:openGauss 内核家族（拉动 yiTrace）+ AgenticData 深集成回路（见 ④）。

## ③ 架构如何最小化外国组件（关键路径国产化 + 自主可控成分清单）

原则：**关键数据路径（写入→存储→索引→查询→加密→审计）必须 100% 自主可控或可信创封装;外国开源只允许出现在"可替换的非关键工具位"，且全部 vendoring（源码内置、不在线拉取、可审计）。**

### 3.1 三件外国组件的处置（本轮的核心动作）

| 组件 | 原计划位置 | 本轮处置 | 理由 |
|---|---|---|---|
| **Lance（美国列存格式）** | 列式不可变 segment 底座 | **自研列存 segment 格式**，复用前两轮已定的 LSM/merge-on-read/晚物化/自研倒排设计;**或**直接落到 **openGauss 列存（CStore，已核实支持列存+向量化批量访问）** 做元数据与部分列存。**Lance 退出关键路径。** | Lance 是叙事最大软肋（关键路径上的美国件）。团队有 PG/openGauss 内核 + Rust + 向量索引能力，自研列存格式是"不缺时间"前提下最该做深的一块，且直接成为"自主可控"卖点。 |
| **DataFusion（美国主导查询引擎）** | SQL/向量化执行 | **不在关键路径引入。** SQL 入口走 **openGauss 执行器 + 自研查询优化器**（团队已有查询优化能力）;Rust 侧自写算子（语义召回算子、区间树扫描算子）。 | 避免把"查询大脑"交给 Apache（美国主导）治理的项目;团队本就有查询优化器，自研可控且是差异点。 |
| **ONNX Runtime（微软）** | embedding 推理运行时 | **替换为昇腾原生栈:CANN + MindIE（或 vLLM-Ascend）**，Qwen3-Embedding-0.6B 直接在昇腾 NPU 上推理（已核实昇腾对 Qwen3 全系 day-0 适配，可导出 OM 格式）。**无昇腾的客户**:回退到鲲鹏/飞腾 CPU 上的国产推理（或自研轻量推理 kernel），**绝不默认依赖 ONNX Runtime**。 | 把推理运行时也国产化，叙事闭环;ONNX Runtime 仅作"导出中间格式"的离线工具，不进运行时关键路径。 |

> 标注不确定项（需进一步核实，勿对外承诺为既成事实）：
> - Qwen3-Embedding-0.6B 在**昇腾 NPU 上的 embedding（非生成）专门适配**与 OM 导出精度/性能，搜索仅确认 Qwen3 系列 day-0 适配与可导出 ONNX/OM，**embedding 任务在昇腾上的端到端验证需自测**。
> - openGauss CStore 是为 OLAP 设计的列存，**能否直接承载"不可变 time-partitioned segment + 自研倒排 + merge-on-read"**需做 PoC;更可能的形态是"自研 Rust 列存 segment + openGauss 存元数据/目录"。
> - 信创目录在册的具体是 openGauss 本身还是其商业发行版（如华为 GaussDB、云和恩墨 MogDB 等），**yiTrace 作为"基于 openGauss 内核的新产品"需独立走信创适配认证**，不能直接套用 openGauss 的在册资格。

### 3.2 自主可控成分清单（端到端，可贴进标书/做成 demo 仪表盘）

| 层 | 组件 | 来源/许可证 | 自主可控等级 | 说明 |
|---|---|---|---|---|
| **整机 CPU** | 鲲鹏 / 飞腾 / 海光 / 兆芯 | 国产 | ★★★ 完全国产 | 已核实生态互认证成熟（飞腾+统信+达梦等组合已规模落地） |
| **操作系统** | 银河麒麟 / 统信 UOS | 国产 | ★★★ | 信创主流 OS，已核实适配成熟 |
| **AI 加速卡** | 昇腾 Ascend（可选） | 国产 | ★★★ | embedding 推理;无卡可回退 CPU |
| **数据库内核** | 自研 LSM/列存 + openGauss 内核家族 | Mulan PSL v2 | ★★★ 自研为主 | 行式热区/列式 segment/compaction/倒排/双编码 trace 树**全自研**;openGauss 承元数据/SQL 入口 |
| **查询引擎** | 自研查询优化器 + openGauss 执行器 | Mulan PSL v2 | ★★★ | **不用 DataFusion** |
| **向量索引** | 自研 IVF/HNSW/DiskANN（yiTrace 同源） | 自研 | ★★★ | 已有自研能力 |
| **Embedding 模型** | Qwen3-Embedding-0.6B | Apache 2.0（阿里，已核实） | ★★★ 国产模型 | 中文强、可商用、气隙可运行 |
| **推理运行时** | CANN + MindIE / vLLM-Ascend；CPU 回退 | 国产/开源 | ★★☆ | **不默认用 ONNX Runtime** |
| **加密** | 国密 SM2/SM3/SM4（存储 SM4、传输国密 TLS、完整性 SM3） | 国标 | ★★★ | 等保三级+密评强制项 |
| **Web 前端** | Vue Flow + ECharts（AgenticData 同源） | 开源（国际） | ★★☆ 非关键路径 | 前端组件不触数据安全核心，可接受;如评审苛刻可备国产替代说明 |
| **可观测协议** | OTel / OpenInference / LangSmith-compat | 开源协议（国际标准） | ★★☆ 仅协议 | **协议非组件**——只解析其数据格式，不运行其代码，是兼容性而非依赖 |

**清单的对外话术**：
> "关键数据路径（写入、存储、索引、查询、加密、审计）100% 自主可控;外国开源仅出现在**协议兼容（OTel，只解析不运行）**和**非安全前端组件**两个位置，全部源码内置、可审计、可替换。"

### 3.3 "可替换/可信创封装"工程纪律（守住真实性的技术保证）

- **依赖白名单 + 国别标注**:CI 里跑一个 `provenance-check`，对每个第三方依赖标注（主体国别/许可证/是否关键路径），任何"美国主体 + 关键路径"依赖**红线告警，禁止合入**。
- **全 vendoring + 气隙构建**:所有依赖源码内置，构建在国产 OS + 国产 CPU 的离线环境完成，产出可复现哈希——这同时是等保/密评的供应链证据。
- **抽象边界**:embedding 推理、列存格式、SQL 执行均封装在 trait/接口后，外国件即便临时存在也"一个接口可换掉"，便于在认证答辩时演示"可替换性"。

- **打包 SKU**:对金融政企出"国产 Agent 全栈"套餐 = AgenticData（建 Agent）+ yiTrace（观测）+ yiTrace（知识库/RAG），一次招标、一套信创认证、一支团队。

## ⑤ 风险：自主可控叙事被发现"核心用外国开源"会反噬——如何守住真实性

**风险本质**:在金融政企/信创评审/竞品攻击面前，"自主可控"是一句**可被逐依赖核查的承诺**。一旦被发现关键路径上有美国开源（Lance/DataFusion/ONNX Runtime/ClickHouse 之流），轻则叙事崩塌、重则**被认定虚假宣传、信创资格作废**——这是当场打脸、且不可逆的。

### 5.1 守住真实性的七条纪律（从"别吹破"到"可证伪"）

1. **关键路径零美国主体依赖（硬红线）**:本轮已把 Lance/DataFusion/ONNX Runtime 全部移出关键路径（见 ③）。CI `provenance-check` 红线拦截，任何 PR 引入"美国主体+关键路径"依赖一律拒绝合入。**这是叙事真实性的工程地基。**

2. **话术分级，绝不过度承诺**:
   - 对**关键路径**用"100% 自主可控/自研";
   - 对**协议兼容（OTel）**说"兼容国际标准协议，仅解析数据格式、不运行其代码"——**协议≠组件**,这是经得起核查的精确表述;
   - 对**前端组件（Vue Flow/ECharts）**诚实标注"非安全关键路径的国际开源组件"，不混进"自主可控成分"里冒充。
   - **宁可少吹一项，不可被抓一项。**

3. **自主可控成分清单可公开、可审计、可第三方复核**:把 ③.2 的清单做成产品内置的"自主可控仪表盘"（demo 第 0 幕那个），每一项标来源/许可证/国别/是否关键路径。**主动暴露薄弱项（前端、推理回退）**比被对手揭发可信得多——"我们自己都标出来了"本身就是真实性背书。

4. **许可证合规独立审查（防 Elastic License 式坑）**:已核实 Phoenix 是 ELv2（非开源、限再分发）——**绝不引入任何非 OSI 开源或限制再分发的境外组件**到可分发产品中。openGauss=Mulan PSL v2、Qwen3=Apache 2.0,均可商用可再分发，已核实安全。建立"许可证准入清单"，法务+工程双签。

5. **气隙可复现构建作为铁证**:全栈在国产 CPU+国产 OS 的离线环境构建，产出哈希可复现。评审/客户可要求"在你们机房用你们给的源码重建一遍"——**能复现=没有偷偷拉境外二进制**,这是最硬的真实性证明。

6. **"可替换性"作为退路演示**:即便某非关键位暂时用了国际开源，把它封在接口后，能现场演示"换成国产实现"。答辩话术:"它在不在不影响自主可控判定，因为它可替换、不在关键路径、且我们已有国产替代。"

7. **认证文档与宣传口径强一致**:对外 PR 的"国产首个/完全自主可控"措辞，必须与提交给信创/等保/密评的技术文档**逐字对齐**。宣传不得超出认证材料能支撑的范围——否则一旦认证材料与市场话术对不上，就是给对手和评审递刀子。

### 5.2 反噬场景预案（被对手当场打脸时怎么接）
