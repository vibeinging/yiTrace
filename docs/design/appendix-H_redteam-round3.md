# 附录 H：第三轮红队复审（裁决 risky —— 指出 v3 引擎决策的两个未验证承重假设）

**最可能失败的一击:** 最可能失败的一击:首单押在一个'用错误前提论证出来的、技术地基未验证的'方案上,导致同时踩两颗雷而无退路。具体链路——(1)花18+月重写引擎去掉 DataFusion/Lance,事后发现'外国开源伤认证'根本不是真门槛(查实:开源被官方认定可自主可控,openGauss 自身就 25% 是 PostgreSQL 代码且照样进信创目录),自研投入买了个不存在的认证门槛;(2)与此同时'A强化版'依赖的 openGauss 自定义 Table AM 接口实际不存在(内核停在 PG 9.2.4,早于 PG12 的 pluggable storage),自研 segment 无法复用 openGauss 的优化器/向量化/HA/DataVec,'高复用低工程量'的核心卖点落空,被迫 fork 内核做重度改造,工程从'扩展'膨胀为'准自研数据库',严重延期;(3)引擎延期 → 能过认证的成熟产品延期 → 错过安可批次窗口(一年仅2次)→ 首单从29月滑到40月+;(4)期间因放弃了互联网/中小快钱客户,零回款、零PMF反馈、零现金流,母体若不能无限输血则资金链在兑现首单前断裂。一句话:在没有拿到任何一份真实客户采购需求书、没有验证 Table AM 接口、没有向测评机构确认认证门槛的情况下,就用'不缺时间'麻醉自己去做最重的自研——把可证伪的关键假设全部推迟到工程做完之后才会暴露,这是顺序错误,致命。正确顺序:先用1-2周做三个零成本的事实核实(客户需求书、Table AM PoC、认证机构书面确认开源/自研占比口径),任何一个翻盘都会改写引擎决策——这三件事的成本几乎为零,却能避免18个月的方向性返工。"

**能力缺口:**
- 认证门槛的事实核实缺口(最关键):v3 把整个引擎重决策(去DataFusion/Lance、A强化版而非C)建立在'外国开源伤自主可控认证'之上,但这一前提与查实到的官方/行业口径('开源可自主可控、可用于信创、openGauss自身就含PG代码')直接冲突。整份设计没有一处引用了'应用软件类信创认证明确对外国开源关键路径判负'或'有硬性自研占比阈值'的权威依据——这是支撑全部工程取舍的承重墙,却是空的。
- openGauss 自定义存储引擎/Table AM 接口的存在性:'A强化版'的核心机制(自研 segment 作为 openGauss 可识别的AM,从而复用其优化器/向量化/HA/DataVec)从未被验证。openGauss 内核基线为 PG 9.2.4,早于 PG 12 的 pluggable Table AM。这个接口若不存在,A强化版的'高复用'卖点和工程量估算全部失效。v3 自己把它列为不确定项④,但仍据此下了'B边际收益不抵成本'的结论——结论先于验证。
- 现金流与runway的量化缺口:17-29月无回款期的总烧钱(测试环境+逐栈适配+信创/等保/密评年费+团队)没有数字;母体 AgenticData/yiTrace 是否盈利、能提供多长 runway 没有交代。'政府订单养着'是定性愿望,无现金流模型支撑。
- 认证失败/拖期的Plan B完全缺失:v3 通篇假设认证会过、会在17-29月内过。没有任何'若安可批次连续错过、若等保整改反复、若密评不通过、若拖到3年+'的应对。把唯一首单押在一条12-24月且依赖外部批次窗口(一年2次)的长杆上,却无降级路径(如先做不要信创认证的商业/出海/互联网客户兜底收入)。
- B(全自研)与C(Lance/DataFusion)被过快淘汰:C 出局的理由(自主可控+SCA)已被证据削弱,应重新进入候选;B 的'边际叙事收益不抵成本'结论建立在'A真能高复用openGauss'之上,而这一点未验证。三选一的评分表在两个关键假设(认证门槛、Table AM)被质疑后需要重算。
- 目标客户采购需求书始终未拿到:三份产出反复把'招标文件到底要安可结果还是信创证书''等保/密评精确报价''数据是否核心数据'列为待核实——这些不是边角料,是决定认证路径、成本、定级、时间线的根参数。在拿到至少一个真实意向客户的采购需求书前,整个路线图的精度是估算叠估算。

**风险清单:**
- **[CRITICAL] 中心论证的事实基础被推翻:v3 把 DataFusion/Lance/ONNX 移出关键路径的首要理由是'外国 Apache 开源 = 自主可控叙事软肋 + 安可/SCA 认证减分'。但联网查实显示官方与行业主流口径恰恰相反——开源软件(含国外 Apache/MIT)被明确认定为'可自主可控',因为源码开放、可自行编译、不受出口管制;真正被排斥的是'不开源的黑盒厂商'。所谓'SCA/自研占比检测'存在,但其定位是'提升自研比例'的改进手段,不是带固定阈值的一票否决门槛,且公开资料未给出适用于应用软件的硬性比例线。更反讽的是 openGauss 本身 25% 代码同源 PostgreSQL(源自 PG 9.2.4,保留约25万行PG接口/公共函数代码),它能进信创目录恰恰证明'用开源底座照样过认证'。这意味着 v3 为'去外国开源'付出的巨大自研成本(自研列存格式/自研查询算子/替换ONNX)在认证层面可能是'为一个不存在的门槛买单'。**
  - 缓解: 在动工前,让合规/销售拿目标金融客户的真实采购需求书 + 向 CESI/测评机构正式书面确认:(a)应用软件类信创认证是否对'外国主导开源组件在关键路径'判负;(b)是否有自研占比硬阈值。若答案是'开源可接受'(大概率),则把'去DataFusion/Lance'从'认证必须'降级为'纯叙事选择',用叙事收益 vs 工程成本重新做ROI——很可能保留 DataFusion(用 Arrow 列式+成熟SQL执行)反而是更优工程解,只在对外话术上把它描述为'可替换的开源组件,源码内置可审计'即可,无需真的重写。
- **[CRITICAL] 'A强化版'引擎架构建立在一个未经验证、且很可能不成立的工程假设上:把'自研行式LSM + 自研列式不可变segment'作为'自定义 openGauss Table AM'插进 openGauss 体系内。查实:openGauss 的存储引擎是固定集合(astore追加更新 / ustore原地更新+undo / MOT内存 / cstore列存),没有公开证据表明它暴露了 PostgreSQL 12+ 的 pluggable Table Access Method 接口供第三方注册自定义存储引擎(openGauss 内核停在 PG 9.2.4 基线,而 Table AM 是 PG 12 才引入的)。若 Table AM 接口不存在,'自研 segment 长在 openGauss 体系内'就不成立,你要么(a)fork 并改 openGauss 内核加存储引擎(等于重度内核工程,不再是'扩展'),要么(b)自研 LSM 作为独立进程/旁路,与 openGauss 仅松耦合(那 openGauss 的优化器/向量化执行/HA 还能不能盖到你的 segment 上就是大问号——很可能盖不上,DataVec/向量化只对它自己的表类型生效)。两条路都和 v3 宣称的'高复用、低工程量'矛盾。这是整个引擎决策最可能崩的技术地基。**
  - 缓解: Phase 0 第一优先级 PoC:验证 openGauss 当前内核分支是否有可用的自定义存储引擎/Table AM 注册接口。若无,诚实承认'A强化版'实为'fork openGauss 内核做存储引擎扩展',据此重估工程量(内核级改动,不是插件级);并明确划清自研 segment 能否被 openGauss 优化器/向量化执行/DataVec 向量索引/HA 主备覆盖——这四项复用是 A 相对 B 的全部优势所在,若覆盖不到,A 强化版相对纯自研(B)的优势会大幅缩水。
- **[HIGH] 双重自研负担被低估:v3 既要'自研 LSM/列存/倒排/trace树双编码/merge-on-read'(SmithDB范式核心),又要把它们塞进/适配 openGauss,还要在认证长杆期并行做3个PoC + 国密三链路集成 + HA + 脱敏 + 等保三级内建能力 + 昇腾适配。这是一个内核团队规模下的多线重活。'不缺时间(17-29月)'是个危险的麻醉剂——认证排队时间不等于工程产能,17-29月里认证占掉的是日历时间(排队/批次窗口),不是团队工时,团队在这段时间要同时交付'能过认证的成熟产品+自研引擎+全套合规能力',工程关键路径很可能比认证长杆更紧。一旦自研引擎延期,认证材料(需要可测的成熟产品)也跟着延,首单时间线从29月滑向40月+。**
  - 缓解: 做减法:在 Phase 0-1 用'最省力能过认证的引擎形态'(可以是 openGauss cstore + 行存热表的朴素组合,甚至先接受性能不极致)抢下认证和首单,把'自研深度引擎'作为 v2 产品在认证后迭代上去。认证要的是'功能/安全/兼容合格',不是'性能业界第一'。把自研深度与认证解耦,避免自研延期连累认证长杆。
- **[HIGH] 全押金融政企 + 放弃互联网/中小快钱 = 现金流单点脆弱。17-29月零首单回款,期间还要烧:三类CPU测试环境(数十万一次性)、认证费(信创5-15万+等保9万/年+密评数万-数十万+逐栈适配)、内核团队薪资。政府/金融订单的二阶问题查实方向明确:回款账期长、定制化重、单客户深度绑定。'自家先用+政府订单养着'是乐观假设——AgenticData/yiTrace 自身是否已盈利能反哺?若母体不盈利,这是用一个未盈利业务养另一个18+月不回款的业务。同时把'低替换阻力的增量刚需(Agent可观测)'的早期采用者(互联网/AI公司,愿意快速付费、能给产品打磨反馈)全部放弃,等于放弃了唯一能在认证期内产生收入+验证PMF+打磨产品的客户群。**
  - 缓解: 不要二选一,做时间错位的双轨:认证长杆期(必然要等)正好用'互联网/AI公司/出海客户'的SaaS或私有化轻量版去跑收入、验证PMF、打磨语义召回这个真差异化钩子——这批客户不要信创认证、付费快、反馈快,且能反哺金融政企版的成熟度。金融政企作为'认证后兑现的大单'在后台并行推进认证。把互联网客户从'放弃'改为'认证期的现金流与PMF引擎'。明确母体(AgenticData/yiTrace)的造血能力作为 runway 假设的前置核实项。
- **[HIGH] 叙事真实性是双刃剑且 v3 自己埋了雷:v3 高调主张'国产首个、完全自主可控',并设计了'自主可控成分仪表盘'公开每个组件国别。但 openGauss 25% 代码同源 PostgreSQL、Rust toolchain 是多国社区项目(LLVM后端、非中国主导)、Qwen3 虽国产但仍是 Apache 开源模型、前端 Vue Flow/ECharts、协议 OTel——一旦你把'完全自主可控'喊到最满,竞争对手或苛刻评审用同一把'成分检测'尺子量你自己,'完全'二字立刻被证伪(你的栈里到处是开源)。v3 一边说'对手用ClickHouse/开源所以不自主可控',一边自己也建在开源上,这个双标会被当场打脸。**
  - 缓解: 放弃'完全自主可控'这种绝对化措辞,改用可证伪的精确表述:'关键数据路径国产化/可审计/源码内置/无境外SaaS依赖/数据不出境'。把护城河叙事的重心从'我们纯国产 vs 对手用开源'(站不住,双标)转移到'我们可私有化+气隙+信创认证在册+数据不出境 vs 对手是境外公司主体/SaaS优先/不提供可信创版本'——后者(公司主体+许可证再分发限制+SaaS架构)才是对手真正结构性补不上的,且不会反噬到你自己。Phoenix 的 Elastic License 限再分发、Langfuse/SmithDB 是境外公司主体,这些是真壁垒;'用没用开源'不是。
- **[MEDIUM] openGauss DataVec 作为'招牌语义召回钩子'载体的可行性未实测,且与自研段冲突。查实 DataVec 6.0 的 HNSW/IVF QPS 数据是针对它自己的向量表类型;v3 又要自研 segment 承载 trace 数据、又要把自研 IVF/HNSW/DiskANN'接进 DataVec 的向量AM'。但若 trace 数据存在自研 segment(非 DataVec 的表类型),DataVec 的向量索引能否覆盖到这些数据上是未知的——很可能要么数据存DataVec表(放弃自研segment范式),要么自研全套向量索引(放弃DataVec复用)。'既复用DataVec又自研segment'可能是不可兼得的。带过滤的语义召回(向量+标量/JSONB过滤下推联动)在 DataVec 上的实测延迟 v3 自己也标了未查到。**
  - 缓解: PoC明确二选一边界:语义召回数据是走DataVec表(享受其向量索引+向量化,但受其表约束)还是走自研segment(自研全套向量检索)。不要假设两者能无缝拼接。把'带过滤的语义trace召回'端到端延迟做成早期PoC硬指标,因为这是招牌钩子,跑不出来整个差异化叙事塌方。
- **[MEDIUM] cstore 用作列存层的硬约束可能波及范式落地:查实 cstore 只支持追加写、不支持数组、建议列数≤1000、表级约束仅 PARTIAL CLUSTER KEY、字段约束仅 NULL/NOT NULL/DEFAULT。trace 的半结构化 attributes(高基数、动态key)若想列式化,撞列数上限和无数组支持;v3 用JSONB承载attributes可绕开,但JSONB在列存表里的压缩/扫描效率与'列式聚合'的性能承诺存在张力(JSONB本质是行式半结构化,享受不到列式聚合优势)。'列式聚合扫描走向量化执行引擎'的性能承诺对JSONB字段不成立。**
  - 缓解: 对trace做schema-on-write的列拆分(把高频查询的attributes提升为真列,低频的留JSONB),并PoC实测向量化执行引擎对'真列+JSONB混合'的聚合性能。明确对外性能承诺只针对已物化的真列,不对JSONB字段做列式聚合性能承诺。
